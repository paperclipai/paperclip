# Chapter Operating Model

기준일: 2026-03-16

## 목적

이 문서는 현재 운영 중인 `2개월 Chapter + 가벼운 OKR + Linear 실행 관리` 방식이  
지금 팀 규모와 운영 방식에 맞는지 점검하고, 더 가볍고 선명한 형태로 정리하기 위한 문서다.

핵심 질문은 이것이다.

`지금 우리 팀에게 맞는 챕터 운영 방식은 무엇인가?`

## 먼저 결론

지금 방식의 방향 자체는 좋다.

- 분기를 2개월 단위로 쪼갠 것
- 장기 비전과 현재 실행을 연결하려는 것
- Linear와 연결해 실행 단위를 관리하려는 것

이 세 가지는 맞다.

문제는 `방식이 맞는데 포맷이 과하다`는 점이다.

지금 팀 규모와 상태에서는
`Initiative > Chapter > Linear Projects / Cycles / Issues`
정도면 충분하다.

지금처럼 Initiative, O, KR, Project를 모두 따로 두면
관리 체계가 풍성해지는 게 아니라  
오히려 `무엇이 진짜 목표인지 흐려질 가능성`이 높다.

핵심은 이것이다.

`Chapter는 계층 하나를 더 늘리는 관리 시스템이 아니라, 6~8주짜리 운영 브리프여야 한다.`

## 현재 Chapter 2 문서에서 보이는 좋은 점

2026년 3월 3일에 잡은 `2026 Chapter 2` 페이지에는 좋은 의도가 분명히 보인다.

- 챕터 기간을 명시했다
- 한 줄 테마가 있다
- 리소스 배분을 적으려 했다
- 외주, Atlas, Flotter, 게임개발을 한 번에 보려고 했다
- Linear와 연결할 Project 계층을 염두에 뒀다

즉, 실행 체계를 만들려는 의도는 맞다.

## 현재 Chapter 2 문서의 문제

### 1. 계층이 하나 많다

현재 문서는 아래가 섞여 있다.

- Chapter
- Initiative
- O/KR
- Project

이건 큰 조직에서는 가능하지만, 지금 팀에서는 과하다.

결과적으로 아래가 흐려진다.

- 지금 챕터에서 진짜 끝내야 하는 것이 무엇인지
- Linear Project가 무엇을 대표하는지
- 문서에서 보는 목표와 실제 작업이 어떻게 연결되는지

### 2. 2개월치 목표 범위가 넓다

2026년 3월 3일 기준 Chapter 2에는 아래 축이 같이 들어가 있다.

- 외주
- Atlas 구축
- SaaS 플로터 출시
- 스토리 중심 게임개발

작은 Builder 팀 기준으로는 한 챕터에 너무 넓다.

특히 리소스 배분에 `외주 50% / 아틀라스 50% / 게임 0%`라고 적어놓고  
플로터와 게임 항목이 같이 들어가 있어 실제 우선순위가 흐린다.

### 3. Outcome과 Deliverable이 섞여 있다

예를 들면 아래는 대체로 `산출물`이다.

- 컴포넌트 100개
- Block 50개
- Template 50개
- 에이전트 시스템 구현

하지만 챕터 문서에서 먼저 보여야 하는 것은  
`그래서 2개월 뒤 회사 상태가 어떻게 달라졌는가`다.

즉 지금 문서는 `일 목록`이 앞에 나오고  
`상태 변화`가 뒤로 밀려 있다.

### 4. 성공 조건이 크고 추상적이다

예를 들어 아래 항목들은 2개월 안에 끝나는 챕터 목표치고 범위가 크다.

- 기능 전체 구현을 전달하면 전체 구현 및 시스템 누적
- 디자인 에이전트 시스템 구현
- QA 에이전트 시스템 구현

이런 문장은 방향성으로는 좋지만  
챕터 성공 조건으로는 너무 크다.

챕터 문서에는 `2개월 안에 끝날 수 있는 성공 조건`만 남겨야 한다.

### 5. 문서용 구조와 실행용 구조가 섞여 있다

지금 포맷은 Notion 문서에서 전략도 설명하고,  
Linear용 Project 틀도 같이 넣으려는 흔적이 있다.

그러면 둘 다 애매해진다.

- 노션은 생각보다 무거워지고
- Linear는 생각보다 자유롭지 않다

## 지금 팀에 맞는 더 좋은 구조

지금 팀에는 아래 4층 구조가 맞다.

### 1층. Initiative

Linear에서 관리

포함 항목:

- 회사 차원의 상위 방향
- 여러 Project를 묶는 큰 목표
- 활성 Initiative는 1~3개만

### 2층. Chapter

repo / Notion에서 관리

각 Chapter에는 아래 정도면 충분하다.

- 기간
- 한 줄 테마
- 연결 Initiative
- 이번 기간에 달라질 상태 2~3개
- 이번 챕터에서 하지 않을 것

### 3층. Projects / Cycles / Issues

Linear에서 관리

- Cycle은 매주 첫날 여는 주간 정렬 단위 이름으로도 사용한다
- Project: Chapter 목표를 실제 deliverable로 쪼개는 실행 덩어리
- Issue: 실제 작업 단위
- Project Update: 주간 상태 보고
- Issue Comment: 하루 작업 흔적

즉, 상위 방향은 Initiative,
기간 압축은 Chapter,
실행은 Linear가 맡는다.

## 권장 포맷

### Notion의 Chapter 페이지는 이것만 남긴다

1. 기간
2. 챕터 테마
3. 연결 Initiative
4. 이번 챕터에서 하지 않을 것
5. 이번 챕터에서 달라질 상태 2~3개
6. 연결 Linear Project 링크
7. 챕터 마지막 회고

### 빼도 되는 것

- Initiative와 O/KR를 동시에 분리하는 구조
- Project 설명 템플릿을 챕터 문서에 길게 넣는 것
- 너무 큰 미래형 문장
- 2개월 안에 검증 불가능한 성공 조건

## 지금 팀에 맞는 챕터 운영 원칙

### 원칙 1. 챕터는 2개월이면 충분히 짧아야 한다

2개월짜리 챕터는 `전략 선언`이 아니라 `상태 변화 증명`이어야 한다.

예:

- superbuilder가 데모가 아니라 출하 엔진으로 한 단계 올라감
- 예약관리 패키지가 실제로 판매 가능한 형태가 됨
- 게임 투자 시간이 실제 자산으로 남기 시작함

### 원칙 2. 목표는 2개, 많아도 3개만 둔다

현재 팀 규모에서는 4개 이상 잡으면 거의 항상 퍼진다.

권장:

- Cash 목표 1개
- Product/Engine 목표 1개
- Vision/Investment 목표 1개

### 원칙 3. 성공 조건은 숫자보다 상태 변화로 쓴다

예:

- 나쁜 예: 컴포넌트 100개 만들기
- 좋은 예: 예약관리 SaaS 데모를 1시간 안에 생성할 수 있는 상태

숫자는 보조지표로 쓰고,  
챕터 문장 자체는 `무엇이 가능해졌는가`로 쓰는 편이 좋다.

### 원칙 4. Chapter와 Project를 혼동하지 않는다

Chapter는 `기간 안에 달라질 상태`다.

Project는 그걸 만들기 위한 `실행 덩어리`다.

이 둘이 섞이면 문서가 무거워진다.

### 원칙 5. Not doing을 꼭 적는다

작은 팀에서 가장 중요한 것은 무엇을 하느냐보다  
무엇을 이번 챕터에서 하지 않느냐이다.

### 원칙 6. Initiative와 Update를 아껴서 쓴다

- Initiative는 회사 차원의 큰 방향에만 쓴다
- Project Update는 매주 남긴다
- Initiative Update는 격주 또는 Chapter 종료 시점만으로도 충분하다

### 원칙 7. 주간 목표는 Project Update로 잃어버리지 않게 한다

Cycle 회의 직후, 각 핵심 Project에 짧은 Update를 남긴다.

포함 항목:

- 이번 주 목표
- 지난주 진행
- 현재 막힘
- 다음 액션

이렇게 하면 챕터 목표와 실제 주간 실행이 끊어지지 않는다.

### 원칙 8. 일간 목표는 Issue Comment로 작게 남긴다

하루 단위로는 긴 보고보다 아래 3줄이 맞다.

- 오늘 한 일
- 막힌 점
- 다음 액션

이건 특히 비동기 협업과 대표 가시성 확보에 중요하다.

## 현재 Chapter 2를 지금 기준으로 다시 잡으면

2026년 3월 16일 기준으로 보면,  
현재 챕터는 아래처럼 더 선명하게 잡는 편이 맞다.

### Chapter 2 한 줄 테마

`superbuilder를 실제 출하 엔진으로 정리하고, 첫 납품형 SaaS와 선택적 현금 프로젝트를 통제된 방식으로 운영한다.`

### 리소스 배분

- 외주/현금: 45~50%
- superbuilder/패키지화: 40~45%
- 게임 투자 시간: 5~10%

### Must-win goal 1

`예약관리형 SaaS를 반복 납품 가능한 패키지로 정리한다`

성공 조건:

- 데모 흐름 1개가 고정된다
- 세팅 범위 안/밖 문서가 정리된다
- 첫 유료 또는 강한 파일럿 제안이 가능하다

### Must-win goal 2

`superbuilder를 빠른 출하 엔진으로 한 단계 끌어올린다`

성공 조건:

- composer + feature 조합 흐름이 실제 납품 기준으로 정리된다
- 브랜딩/커스터마이징/배포까지 반복 가능한 구조가 생긴다

### Must-win goal 3

`게임 방향성을 말이 아니라 전략 투자 시간으로 유지한다`

성공 조건:

- 주간 게임 투자 시간이 실제로 고정된다
- 도구 또는 소형 프로토타입 1개가 남는다

### 이번 챕터에서 하지 않을 것

- 플로터 전체 출시를 챕터 목표로 잡지 않기
- 만능 에이전트 시스템을 2개월 안에 완성하려고 하지 않기
- 게임 개발을 본업처럼 확장하지 않기
- Linear 구조에 맞추려고 전략 문서를 과도하게 복잡하게 만들지 않기

## Linear와의 연결 원칙

Linear는 `일감 관리 시스템`으로 쓰고,  
Chapter 문서는 `우선순위 정렬 시스템`으로 쓴다.

즉:

- Notion: 왜 이걸 하는가, 이번 챕터에서 무엇이 달라지는가
- Linear: 누가, 무엇을, 언제까지 하는가

권장 연결 방식:

- Chapter page에 Goal 2~3개만 둔다
- 각 Goal 아래에 연결된 Linear Project 링크만 둔다
- 나머지 세부 작업은 전부 Linear Issue로 내린다

## 지금 바로 바꾸면 좋은 것

### 1. Initiative / O / KR / Project 계층을 줄인다

`Chapter > Goal > Linear Project/Issue` 정도로 줄인다.

### 2. Goal 수를 3개 이하로 줄인다

외주, 엔진, 게임 투자 정도면 충분하다.

### 3. 성공 조건을 1~2개만 남긴다

길고 멋진 문장보다 `이번 챕터 끝나면 무엇이 가능해졌는가`만 본다.

### 4. Not doing을 위로 올린다

이건 챕터 문서에서 매우 중요하다.

### 5. 챕터 종료 회고를 반드시 남긴다

다음 챕터가 더 좋아지려면  
양식보다 회고가 더 중요하다.

## 대표용 한 줄 규칙

`챕터 문서는 실행을 설명하는 문서가 아니라, 이번 2개월 동안 무엇에 집중하고 무엇을 하지 않을지 정하는 문서여야 한다.`

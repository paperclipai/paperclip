# 회의록 운영 모델

이 문서는 노션 `회의록` 데이터베이스를 읽고,
현재 쓰고 있는 회의 문서 구조를 repo 기준의 표준 템플릿으로 정리하기 위한 운영 문서다.

기준일:

- `2026-03-16`

## 1. 현재 노션 구조 요약

데이터베이스:

- 이름: `회의록`
- 속성:
  - `Name`
  - `회의 유형`
  - `회의일`
  - `참가자`

현재 확인된 템플릿/패턴:

- `Chapter`
- `Cycle`
- `Daily Focus`
- `연간 회고`

즉, 운영 구조 자체는 이미 있다.

`챕터 > 주간 정렬 > 일간 실행 > 회고`

이 흐름은 맞다.

## 2. 지금 구조의 좋은 점

- 회사가 장기 비전과 현재 실행을 연결하려는 의도가 분명하다
- 챕터, 주간, 일간, 회고가 구분되어 있다
- 팀 공유, 요청, 계획, 회고를 한 문서 안에서 묶으려는 습관이 있다
- Linear/ClickUp 링크를 붙여 실행과 연결하려는 시도가 있다

## 3. 지금 구조의 문제

### 1. 템플릿과 설명문이 섞여 있다

특히 Chapter 템플릿과 Cycle 템플릿에는
설명, 예시, 실제 입력칸이 한 페이지에 섞여 있다.

그러면 실제 회의 중에는 문서가 길고 무거워진다.

repo에는 설명을 두고,
실제 템플릿에는 입력칸만 남기는 게 맞다.

### 2. 챕터 문서가 과하다

현재 Chapter 문서는 `Initiative`, `O/KR`, `Project`가 한 번에 들어가 있어
3인 Builder 팀 기준으로는 무겁다.

챕터 문서는 아래만 남기는 게 맞다.

- 테마
- 리소스 배분
- 이번 챕터에서 하지 않을 것
- Must-win goals 2~3개
- 성공 조건
- 연결 Linear Project

### 3. 일간 문서가 너무 길게 누적된다

`2025- Daily Focus`는 하나의 페이지 안에 여러 날짜와 여러 사람의 내용이 길게 누적되어 있다.

이 방식은 아래 문제가 생긴다.

- 찾기 어렵다
- 오늘 집중할 것이 흐려진다
- 지난 날짜와 오늘 날짜가 섞인다
- 회고가 쌓여도 다음 주 점검 재료로 꺼내기 어렵다

일간 문서는 `하루 1페이지` 또는 최소 `주간 1페이지 + 날짜 섹션`으로 분리하는 편이 좋다.
지금 팀에는 `하루 1페이지`가 더 맞다.

### 4. 주간 문서는 상태/결정/다음 액션이 부족하다

현재 Cycle 템플릿은 짧아서 좋지만,
아래가 더 선명해야 한다.

- 이번 주 완료
- 현재 막힘
- 이번 주 결정
- 다음 주 1~3개

### 5. 회고는 좋지만 너무 길다

연간 회고 질문은 풍부하고 좋다.
다만 실제 작성용으로는 길어서, 우선 `핵심 질문판`과 `풀버전`을 나누는 게 좋다.

repo에는 풀버전을 남기고,
실사용은 10개 안쪽 핵심 질문판으로 줄여도 된다.

## 4. 권장 운영 구조

### 1. Chapter

주기:

- 2개월

목적:

- 이번 챕터에서 `무엇이 달라져야 하는가`를 고정

기록 원칙:

- 목표 2~3개만
- 성공 조건은 1~2개만
- `Not doing` 반드시 포함

### 2. Weekly Cycle

주기:

- 주 1회
- 매주 첫날

목적:

- 이번 주 정렬
- 막힘과 결정 필요사항 정리
- Linear에서 같은 이름으로 쓰는 주간 실행 단위와 맞춘다

기록 원칙:

- 완료/막힘/다음 주로 나눔
- 지난주 carryover를 먼저 봄
- 이번 주 Must-win goal 1~3개를 고정
- 목표별 진행도를 주간 scoreboard처럼 관리
- 길게 설명하지 않음
- 결정이 필요한 건 명시
- 회의 이름과 Linear 명칭을 `Cycle`로 통일한다
- 회의 직후 핵심 Project에 주간 Update를 남긴다

### 3. Daily Focus

주기:

- 하루 1회

목적:

- 오늘의 초점 공유
- 팀 요청/막힘 빠르게 정렬
- 하루 목표를 잃지 않게 고정
- 전일 약속을 실제 결과로 닫기
- 이번 주 목표와 오늘 할 일을 연결하기
- 하루 끝에 이월 여부를 짧게 닫기

기록 원칙:

- 사람당 `최우선 1개`만 적음
- 각자 관련 Linear Issue를 같이 적음
- 이번 주 Must-win goal 진행도를 맨 위에 둠
- 전일 목표 체크를 먼저 함
- 완료 기준이 없는 목표는 적지 않음
- 공유/요청/오늘 할 일/막힘/짧은 회고만 남김
- 하루 1페이지로 분리
- 아침에는 목표를 적고, 저녁에는 `완료 / 미완료 / 이월`만 짧게 닫는다

### 4. Annual Retro

주기:

- 연말 또는 연초 1회

목적:

- 회사/개인/팀/운영을 정리해 다음 챕터 액션으로 연결

기록 원칙:

- 길게 써도 되는 문서
- 하지만 실제 회고 세션용 핵심 질문판은 따로 둠

## 5. repo에 두는 표준 템플릿

repo에는 아래 4개를 표준으로 둔다.

1. `Chapter 템플릿`
2. `Weekly Cycle 템플릿`
3. `Daily Focus 템플릿`
4. `Annual Retro 템플릿`

중요한 원칙:

- 실제 템플릿은 짧아야 한다
- 설명/예시는 운영 문서에 둔다
- 템플릿에는 입력할 칸만 남긴다

## 6. 제목 규칙 권장안

- Chapter: `2026 Chapter 2`
- Weekly Cycle: `C2 Cycle 1`
- Daily Focus: `2026-03-16 Daily Focus`
- Annual Retro: `2026 연간 회고`

주의:

- `Cycle`은 매주 첫날에 여는 주간 정렬 회의 이름이다
- 노션 문서명과 Linear 명칭을 동일하게 맞춘다

## 7. 회의 유형 권장 사용법

현재 속성 `회의 유형`은 아래처럼 쓰는 편이 좋다.

- `계획`: Chapter, Daily Focus
- `정렬`: Weekly Cycle
- `회고`: 연간 회고, 챕터 회고

## 8. 지금 바로 바꾸면 좋은 것

1. Daily Focus를 `하루 1페이지`로 분리
2. Chapter 페이지에서 설명 블록 제거
3. Weekly Cycle에 `Carryover / Must-win goal / 금요일 마감 체크` 추가
4. 회고 질문은 핵심판과 풀버전으로 분리

## 8-1. Linear와의 연결 방식

- Cycle은 회의 이름이면서 주간 실행 단위 이름이다
- 노션의 `C2 Cycle 1`과 Linear의 Cycle 명칭을 같은 언어로 맞춘다
- 회의에서 정렬한 내용은 Linear Project / Issue 우선순위에 바로 반영한다
- 따라서 Cycle 문서는 `회의록`이면서 동시에 `주간 정렬 스냅샷` 역할을 한다
- 주간 상태 공유는 Linear `Project Update`에 남긴다
- 하루 작업 흔적은 각 `Issue Comment`에 남긴다

## 8-2. 추천 실무 규칙

### Project Update

- 주 1회
- Cycle 회의 직후
- health + 이번 주 목표 + 막힘 + 다음 액션

### Issue Comment

- 하루 2회
- 아침: 오늘 목표 / 첫 액션 / 예상 막힘
- 저녁: 결과 / 미완료 이유 / 내일 첫 액션
- 피드백이 오면 같은 날 흔적 남기기

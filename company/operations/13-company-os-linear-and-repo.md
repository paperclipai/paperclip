# Company OS: Linear And Repo

기준일: 2026-03-16

## 목적

이 문서는 회사의 전략 업무, 경영지원, 실행 관리를  
어떤 도구 조합으로 운영하는 것이 맞는지 정리한 문서다.

핵심 질문은 이것이다.

`전략과 운영까지 Linear로 관리할 수 있는가, 아니면 company repo와 분리해야 하는가?`

## 먼저 결론

가능은 하다.  
하지만 `전부를 Linear에 넣는 것`은 추천하지 않는다.

지금 팀에는 아래 구조가 가장 맞다.

- `company repo`: 전략, 운영 원칙, 회고, 의사결정, 에이전트 협업
- `Linear`: 프로젝트, 이슈, 책임자, 일정, 상태, 의존성
- `Notion`: 챕터 개요나 공유용 요약 페이지 정도로만 가볍게 사용

한 줄로 말하면:

`생각과 판단은 repo에 남기고, 약속과 실행은 Linear에서 관리하는 구조가 맞다.`

## 왜 전부를 Linear에 넣으면 안 되는가

Linear는 실행 관리에는 강하다.  
하지만 전략 사고와 경영 문맥을 오래 쌓는 데는 한계가 있다.

특히 아래 같은 것은 Linear에 잘 안 맞는다.

- 장문 전략 문서
- 운영 원칙
- 대표 판단 기준
- 회고와 맥락 축적
- 에이전트와 여러 번 수정하는 초안 작업
- 여러 문서를 엮어서 생각하는 일

이런 건 이슈로 만들면 관리되는 게 아니라  
오히려 `문맥이 찢어질 가능성`이 높다.

## Linear가 잘하는 것

Linear는 아래에 강하다.

- Initiative로 상위 방향을 묶는 것
- 누가 맡았는가
- 언제까지 하는가
- 지금 상태가 무엇인가
- 무엇이 무엇에 막혀 있는가
- 어떤 프로젝트에 속하는가

Linear 공식 개념도 이 흐름에 가깝다.

- Issue는 기본 작업 단위
- Project는 시간 제한이 있는 deliverable
- Initiative는 여러 Project를 묶는 회사 목표 단위

즉 Linear는 `실행 구조화`에 강하고,  
repo는 `판단과 문맥 보존`에 강하다.

## 지금 팀에 맞는 권장 운영 구조

### 1. company repo

repo는 회사의 `운영 두뇌` 역할을 한다.

여기에 두는 것:

- 전략 문서
- 운영 원칙
- 챕터 운영 문서
- HR/조직 문서
- 수주 판단 기준
- 주간/월간 점검표
- 회고
- 의사결정 기록
- 에이전트가 만드는 초안과 제안

repo가 좋은 이유:

- 버전이 남는다
- 비교가 쉽다
- 에이전트와 같이 문서를 계속 다듬기 좋다
- 작은 회사의 사고 체계를 쌓기 좋다

### 2. Linear

Linear는 회사의 `실행판` 역할을 한다.

여기에 두는 것:

- Initiative
- 이번 챕터의 핵심 Project
- 실제 Issue
- 담당자
- 우선순위
- 마감일
- 상태
- 의존성

Linear에 넣을 기준:

- 누가 해야 하는 일이 명확한가
- 끝났는지 아닌지 판단 가능한가
- 일정과 상태를 추적할 가치가 있는가

## Linear 계층을 어떻게 써야 하는가

### Initiative

- 회사 차원의 상위 방향
- 활성 Initiative는 `1~3개`
- owner가 분명해야 한다

### Project

- Chapter 목표를 실제 deliverable로 묶는 단위
- 핵심 Project는 챕터당 `2~4개`

### Cycle

- 주간 timebox
- 이번 주 끝낼 Issue를 정렬하는 단위

### Issue

- 가장 작은 실행 단위

## Linear Update를 어떻게 써야 하는가

지금 팀에는 `Project Update는 주간`, `Issue Comment는 일간`으로 나누는 것이 가장 맞다.

이유는 단순하다.

- Project Update는 주간 단위 상태 정렬에 강하다
- Issue Comment는 하루 단위 작업 흔적을 남기기에 가볍다
- 둘을 섞으면 매일 긴 Update를 쓰게 되고 금방 무거워진다

Linear 공식 문서 기준으로도
Project Update는 프로젝트 리드가 정기적으로 올리는 구조화된 상태 보고에 맞고,
댓글로 후속 논의를 이어갈 수 있다.

Initiative Update는 상위 방향의 진행과 리스크를 묶는 데 맞고,
프로젝트보다 낮은 빈도로 쓰는 것이 더 자연스럽다.

참고:

- `Initiative and Project updates`
- `Concepts`

## 지금 팀에 맞는 권장 사용법

### 1. 주간: Project Update

대상:

- Chapter 안의 핵심 Project
- 예: `리드 1Day POC 플레이북`, `예약관리 SaaS Starter`

주기:

- 매주 첫날 Cycle 회의 직후

기록할 것:

- health
- 이번 주 목표
- 지난주 진행
- 현재 막힘
- 다음 액션

규칙:

- 길게 쓰지 않는다
- 팀 전체가 보는 `주간 상태 신호`로 쓴다
- 댓글은 질문, 보류, 피드백 정리에 쓴다

### 2. 매일: Issue Comment

대상:

- 실제 작업 이슈
- 특히 진행 중이거나 대표가 같이 보는 이슈

주기:

- 하루 2회까지

기록할 것:

- 아침: 오늘 목표 / 첫 액션 / 예상 막힘
- 저녁: 결과 / 미완료 이유 / 내일 첫 액션

규칙:

- 3줄이면 충분하다
- 상태가 바뀌면 코멘트 또는 상태 변경 중 하나는 남긴다
- 피드백을 받으면 같은 날 흔적을 남긴다

### 3. 팀 공유: Daily Focus

대상:

- 팀 전체

주기:

- 하루 1회

기록할 것:

- 이번 주 Must-win goal 진행도
- 전일 목표 체크
- 오늘 팀 공통 초점
- 각자 최우선 1개
- 관련 Linear Issue
- 요청/막힘
- 저녁의 완료 / 미완료 / 이월

규칙:

- 팀 전체가 보는 `하루 스냅샷`으로만 쓴다
- 자동 생성 시 전일 문서와 현재 Cycle 진행도를 먼저 채운다
- 사람당 최우선 1개만 둔다
- 세부 진행은 Linear 이슈 코멘트에 남긴다

## 추천 운영 리듬

### Initiative

- 월 2회 또는 Chapter 종료 시점
- 큰 방향과 리스크만 업데이트

### 월/챕터

- Chapter 문서에서 Must-win goal 확인
- Initiative와 연결된 Project 확인
- 그 Goal에 연결된 Linear Project 유지

### 주간

- Cycle 회의에서 이번 주 초점 정렬
- 바로 해당 Project에 Update 작성

### 일간

- 아침에는 Daily Focus에서 전일 목표를 먼저 닫고 오늘 최우선 1개를 공유
- 각자는 관련 이슈에 아침 코멘트를 남김
- 저녁에는 같은 이슈에 결과를 남기고 Daily Focus에서 완료 / 미완료 / 이월만 닫음
- 대표는 Daily Focus로 전체를 보고, 필요할 때만 이슈 코멘트로 내려가면 됨

한 줄로 정리하면:

`Initiative는 크게, Chapter는 얇게, 주간 상태는 Project Update로, 하루 실행은 Issue Comment로 남긴다.`

### 3. Notion

Notion은 지금 당장은 없애지 않아도 된다.  
하지만 역할을 줄이는 게 좋다.

권장 역할:

- 챕터 개요 공유
- 회의 메모
- 외부 공유용 페이지

비권장 역할:

- 전략의 최종 저장소
- 운영 원칙의 최종 저장소
- 에이전트 협업의 중심

## 지금 팀에 맞는 구체적 기준

### repo에 넣어야 하는 것

- 길게 생각해야 하는 것
- 나중에 다시 참고할 판단 기준
- 문맥이 중요한 것
- 에이전트와 함께 초안을 여러 번 발전시키는 것
- 회사의 원칙이 되는 것

예:

- Chapter 운영 방식
- HR 운영 기준
- 수주 판단 매트릭스
- 게임 투자 시간 모델
- 외주/제품/게임 간 우선순위 원칙

### Linear에 넣어야 하는 것

- 실제 실행 덩어리
- 담당자와 마감일이 있는 것
- 진행 상태를 매일/매주 봐야 하는 것

예:

- 예약관리 패키지 데모 고정
- 세팅 범위 문서 작성
- superbuilder 배포 자동화 정리
- ABADC 제안/수행 관련 작업

### Notion에만 남겨도 되는 것

- 가볍게 공유할 챕터 요약
- 회의록
- 참고 링크 모음

## 추천 Company OS

지금 팀에는 아래 구조가 가장 현실적이다.

### Layer 1. Strategy Layer

위치: `company repo`

역할:

- 무엇을 할지 결정
- 무엇을 하지 않을지 정리
- 기준과 문맥 보존

### Layer 2. Planning Layer

위치: `company repo` + `Notion 요약 페이지`

역할:

- 이번 2개월 챕터 정리
- Must-win goals 2~3개 정의
- Linear Project 링크 연결

### Layer 3. Execution Layer

위치: `Linear`

역할:

- Project
- Issue
- 담당자
- 일정
- 상태

### Layer 4. Review Layer

위치: `company repo`

역할:

- 주간 점검
- 월간 점검
- 챕터 회고
- 의사결정 복기

## 에이전트와 함께 일할 때 repo가 특히 좋은 이유

대표가 전략, 경영지원, 운영 문서까지 직접 보는 상황에서는  
에이전트가 같이 생각하고 정리할 수 있는 `문서 기반 작업 공간`이 중요하다.

repo는 이 점에서 강하다.

- 전략 문서를 계속 수정 가능
- 구조를 바꾸기 쉬움
- 이전 버전과 비교 가능
- 에이전트에게 특정 문서 세트만 읽히기 쉬움
- 반복 템플릿 생성에 유리

즉, `company` repo를 하나 두고  
전략기획실처럼 같이 일하는 구조는 충분히 가능하고 오히려 잘 맞는다.

## 다만 주의할 점

repo가 있으면 모든 것을 문서화하고 싶어질 수 있다.

그래서 아래 원칙이 필요하다.

- 전략 문서는 적고 선명하게
- 실행은 바로 Linear로 내리기
- 문서가 이슈를 대체하지 않게 하기
- 이슈가 전략 문서를 대체하지 않게 하기

핵심은 `두 시스템이 서로 역할을 넘보지 않게 하는 것`이다.

## 지금 바로 추천하는 운영 방식

### 1. company repo를 공식 전략 저장소로 둔다

이미 지금처럼 `strategy/` 문서를 쌓는 방식은 좋다.

여기에 앞으로도 아래를 계속 쌓는다.

- chapter 문서
- 조직 문서
- 수주 판단 문서
- 회고
- 의사결정 기록

### 2. Linear는 실행 전용으로 단순화한다

권장:

- Chapter마다 Project 2~3개
- 그 아래 Issue만 관리
- 너무 상위 전략은 이슈화하지 않음

비권장:

- 전략 문서를 이슈로 쪼개기
- 경영 생각을 티켓처럼 관리하기

### 3. Notion은 얇게 유지한다

권장:

- Chapter 1장
- 회의록
- 공유용 페이지

비권장:

- Notion, repo, Linear에 같은 내용을 중복 기록

### 4. 경영지원도 전부 이슈화하지 않는다

경영지원 업무 중 아래는 Linear에 넣는다.

- 마감이 있는 계약/정산/제안
- 실제 담당자와 일정이 있는 운영 작업

아래는 repo에 둔다.

- 정책
- 원칙
- 기준
- 판단 메모

## 현재 팀 기준 최종 판단

지금 대표가 전략, 경영지원, 운영 설계까지 직접 보는 구조라면  
`Linear만으로 관리하는 것`보다  
`company repo + Linear` 조합이 훨씬 현실적이다.

Linear만 쓰면 실행은 잘 보이지만 생각이 사라지고,  
repo만 쓰면 생각은 쌓이지만 실행이 느슨해진다.

그래서 지금 팀에는 이 조합이 맞다.

- repo = 생각
- Linear = 실행
- Notion = 얇은 공유

## 대표용 한 줄 규칙

`회사의 판단은 repo에 남기고, 회사의 약속은 Linear에서 관리한다.`

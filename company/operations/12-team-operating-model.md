# Team Operating Model

기준일: 2026-03-16

## 목적

이 문서는 현재 팀 구성과 실제 성향을 기준으로  
`어떻게 팀을 굴려야 전략이 유지되는가`를 정리한 운영 문서다.

핵심은 사람을 자원처럼 보는 것이 아니라  
`Builder 포지션 전제`, `강점 차이`, `동기`, `장기 리스크`까지 포함해서 배치하는 것이다.

## 가장 중요한 전제

현재 팀은 전부 `Builder 포지션`으로 본다.

이 전제는 나중에 편의상 붙인 해석이 아니라  
채용 단계에서 이미 명확히 공유한 약속이다.

채용공고에서 이미 아래 내용을 공개했다.

- 장기적으로 게임 회사 방향이 있다
- 현재는 외주와 서비스 개발을 함께 한다
- 직무 구분을 고정하지 않는다
- 기획, FE, BE, 앱, 설계가 미션에 따라 섞일 수 있다
- AI를 적극 활용한다
- 워크플로우, 기록, 회고를 중요하게 본다

즉, 현재 팀 운영의 출발점은  
`특정 직무만 하는 사람들`을 모아놓은 구조가 아니라  
`문제를 풀기 위해 역할을 넘나드는 Builder 팀`이다.

## 현재 팀 구성

### 대표

- 경력: 20년
- 강점:
  - 기획
  - 분석
  - 문제 구조화
  - 실행력
  - LLM 활용
  - 어떻게든 일이 되게 하는 방식에 강함
- 약점 또는 비선호:
  - 어려운 수학, 깊은 알고리즘 영역은 상대적으로 약함

해석:

- 대표는 고난도 애매한 문제를 푸는 데 강하다
- 대신 모든 종류의 전문 영역을 깊게 직접 파는 타입으로 운영하면 비효율이 생긴다
- Builder 조직에서 대표의 역할은 가장 어려운 문제를 정의하고 구조화하는 쪽에 가깝다

### 박지수

- 경력: 3년차
- 강점:
  - FE
  - 꼼꼼함
  - 새로운 정보 취득 관심
- 특성:
  - 속도는 빠른 편이 아님
  - 기술 깊이는 아직 제한적

해석:

- 박지수는 프론트 마감, 관리자/앱 화면 정리, 안정화, 브랜딩 반영 쪽에 맞는다
- 고속 난제 해결보다 `정리하고 다듬는 역할`에서 가치가 크다
- 다만 Builder 포지션 전제이므로 필요 시 기획, 구조 이해, 공통 화면 체계화에도 함께 들어간다

### 박상훈

- 상태: 2026년 4월 말 복귀 예정
- 배경:
  - 게임 개발 방향성 때문에 입사
  - 채용 당시부터 웹서비스/앱 개발과 장기 게임 방향을 함께 이해하고 합류
  - 외주 비중이 높고 게임 개발이 미뤄지며 현재 무급 휴직 중
  - 개인 게임 제작 진행 중이며 연락은 유지 중
- 강점:
  - 수학
  - 사고력
  - 게임 개발 열망

중요 전제:

- 하루 4시간 정도의 다른 업무 투입은 충분히 이해하고 있는 상태다
- 애초에 그런 구조를 알고 입사했다

중요 리스크:

- 웹/앱 업무 자체가 문제가 아니라
- 장기간 `의미 없는 반복 구현만 계속되는 상태`
- 어려운 문제, 시스템 사고, 게임 방향성과의 연결이 거의 사라지는 상태가 누적될 때 리스크가 커진다

해석:

- 박상훈도 Builder 포지션으로 본다
- 다만 팀 안에서 상대적으로 알고리즘, 엔진, 시스템 사고가 필요한 문제에 강점이 더 클 가능성이 높다
- 따라서 `웹/앱 업무를 하면 안 된다`가 아니라 `혼합 배치는 가능하지만 장기 의미 배분이 중요하다`가 맞다

## 현재 팀 운영의 핵심 현실

### 현실 1

2026년 3~4월은 사실상 `대표 + 박지수` 2인 체제로 봐야 한다.

### 현실 2

박상훈 복귀는 플러스 요인이지만, 자동으로 안정적 생산력이 추가되는 것은 아니다.

### 현실 3

지금 전략에서 가장 큰 위험은 업무량 부족이 아니라 `사람-과제 미스매치`다.

### 현실 4

미스매치의 기준은 `직무가 섞였는가`가 아니라  
`각 사람이 자기 강점을 살릴 수 있는 문제를 일정 비율 이상 맡고 있는가`다.

## 역할 배치 원칙

### 대표가 맡아야 할 것

- 영업
- 범위 조정
- 고객 문제 정의
- 아키텍처
- 핵심 구현
- AI/LLM 활용 방식 설계
- 납품 구조 설계

### 대표가 가급적 맡지 말아야 할 것

- 반복적인 화면 구현
- 반복적인 세팅 작업
- 모든 고객 커뮤니케이션의 실무 응대

### 박지수가 맡아야 할 것

- FE 구현
- 관리자 화면
- 앱 UI 정리
- 브랜딩 반영
- 공통 UI/페이지 품질 마감
- 꼼꼼함이 필요한 최종 정리

### 박지수에게 과도하게 맡기지 말 것

- 추상적인 시스템 설계
- 모호한 고난도 문제 해결
- 시간 압박이 큰 다중 문맥 전환

운영상 보완이 필요한 부분:

- Linear 이슈 관리
- 상태 공유
- 피드백 반영 흔적 남기기

해석:

- 이건 성실성 문제보다 `비동기 협업 인터페이스`가 약한 상태에 가깝다
- 대표 입장에서는 `어디까지 왔는지`, `막힌 게 뭔지`, `언제 끝나는지`가 안 보이는 문제가 된다
- 따라서 `잘 보고하라`보다 `작은 규칙 3개`로 먼저 잡는 것이 맞다

### 박상훈이 맡아야 할 것

- 복귀 후 Builder 업무 전반
- 그중에서도 알고리즘형 문제
- 시스템 사고 필요한 과제
- 게임/엔진 관련 내부 과제
- 시뮬레이션, 규칙, 수학적 사고가 필요한 영역
- 장기적으로 game/IP 관련 실험

### 박상훈 배치 원칙

- 하루 4시간 수준의 일반 업무 혼합은 허용 범위로 본다
- 나머지 시간 또는 주간 단위 배분에서 사고력, 시스템성, 게임 방향성과 연결되는 과제를 남겨둔다
- 장기적으로는 `웹/앱도 하는 Builder`가 아니라 `Builder 중에서도 어려운 구조 문제에 강한 사람`으로 성장시키는 것이 맞다
- 따라서 완전 분리보다 `혼합 + 의미 있는 비중 보장`이 핵심이다

## 사람-과제 적합도 원칙

### 대표 x 프로젝트

잘 맞는 일:

- 애매한 문제를 정의해야 하는 일
- 고객과 제품 사이에서 구조를 잡는 일
- 빠른 프로토타입과 고레버리지 구현

위험한 일:

- 모든 프로젝트의 메인 PM + 메인 개발자 + 메인 세일즈를 동시에 맡는 것

### 박지수 x 프로젝트

잘 맞는 일:

- 예약관리 패키지의 FE 정리
- 관리자/앱 표면 품질 고도화
- 공통 템플릿 화면 정리

위험한 일:

- 추상도가 높은 신사업 설계
- 알고리즘형 문제 해결

### 박상훈 x 프로젝트

잘 맞는 일:

- superbuilder 내부 구조 문제
- 반복 납품 구조화
- 게임 관련 구조
- 수학/알고리즘/시뮬레이션
- 사고력이 필요한 내부 엔진 문제

위험한 일:

- 장기간 의미 없는 반복 구현만 전담시키는 것
- 어려운 문제나 장기 방향성과의 연결 없이 소모시키는 것

## 비동기 협업 최소 규칙

이 규칙은 전원 Builder에게 적용한다.

핵심은 문서를 많이 쓰는 것이 아니라  
`작업 상태가 작게라도 매일 보이게 만드는 것`이다.

### 규칙 1. 이슈 시작 전에 완료조건을 짧게 남긴다

최소한 아래 3가지는 남긴다.

- 이 이슈의 완료조건 1~2줄
- 오늘 내가 할 범위
- 현재 막혀 있는 것이 있는지

### 규칙 2. 하루 1회 상태 업데이트를 남긴다

길게 쓰지 않고 아래 3줄만 남긴다.

- 오늘 한 일
- 막힌 점
- 다음 액션

### 규칙 3. 피드백을 받으면 같은 날 흔적을 남긴다

아래 셋 중 하나는 반드시 남긴다.

- 반영 완료
- 보류 이유
- 질문

## 박지수 운영 보완 포인트

박지수에게 필요한 것은 더 복잡한 프로세스가 아니라  
아래처럼 아주 작은 비동기 운영 규칙이다.

1. 이슈 시작 전 완료조건 1~2줄 적기
2. 하루 1회 3줄 상태 업데이트 남기기
3. 피드백을 받으면 같은 날 흔적 남기기

이걸 먼저 고정하면,

- 대표가 계속 상태를 묻는 비용이 줄고
- 작업이 실제보다 덜 보이는 문제를 줄이며
- 박지수의 꼼꼼함이 `가시성`으로도 연결될 수 있다

## 2026년 3~6월 운영 방식

### 3~4월

- 대표 + 박지수 중심
- superbuilder 정리
- 예약관리 패키지 출하
- 반복 납품 구조 정리

이 시기 금지:

- 3인 체제인 것처럼 무거운 약속 잡기
- 알고리즘형 고난도 외주를 동시에 많이 가져가기

### 5~6월

- 박상훈 복귀 이후 역할 재정렬
- 복귀 즉시 풀가동 자원으로 계산하지 않음
- 첫 4주는 적응 + 과제 적합도 확인 기간

권장:

- 박상훈에게는 game/엔진/알고리즘 성격 과제를 일정 비율 보장
- 동시에 Builder 조직의 일원으로 앱/웹/운영 업무도 일부 함께 맡긴다
- superbuilder 구조화, 자동화, 사고력 필요한 기능에 우선 배치한다
- 필요하면 AI-SOS 같은 사고형 과제를 제한적으로 연결

비권장:

- 장기적으로 앱성 프로젝트의 일반 구현만 주업무로 고정
- 합의된 혼합 업무 범위를 넘어 반복 구현 위주로만 소모시키는 배치

## 유지 전략

박상훈 유지에 중요한 것은 보상만이 아니다.  
`왜 여기 있어야 하는가`가 계속 보여야 한다.

권장 방식:

- 게임 방향성을 문서가 아니라 실제 시간 배정으로 보여주기
- 복귀 후 일정 비율은 게임/엔진/사고형 과제를 보장
- 일반 웹/앱 투입 자체를 막는 것이 아니라 의미 있는 문제 배분과 성장 경로를 함께 보여주기

## 대표 리스크

대표는 "어떻게든 되게 하는" 능력이 강하다.  
이건 회사의 장점이지만 동시에 가장 큰 운영 리스크다.

리스크:

- 대표가 모든 부족한 부분을 메우며 시스템화를 늦춤
- 사람 배치 문제도 결국 대표 과부하로 해결하려 함

대응:

- 대표가 메우는 일을 `구조로 바꾸는 것`을 KPI로 둔다

## 팀 운영의 최종 원칙

1. 전원 Builder 포지션으로 운영한다
2. 3~4월은 2인 체제로 운영한다
3. 대표는 고레버리지 문제만 맡는다
4. 박지수는 FE와 마감 품질에 강점을 둔다
5. 박상훈은 혼합 배치가 가능하지만 사고력/게임 지향 강점을 일정 비율 보장한다
6. 사람을 자원처럼 섞어 쓰지 않고, 강점 차이를 반영해 배치한다

## 대표용 한 줄 규칙

`지금 필요한 것은 전원을 Builder로 보되, 각 사람의 강점과 장기 동기까지 고려해서 의미 있는 혼합 배치를 설계하는 것이다.`

# 03. 2026-03 Development View

기준일: 2026-03-16

## 목적

이 문서는 3월 계획을 `전략`이 아니라 `개발 관점`에서 평가한 메모다.

핵심 질문은 하나다.

`현재 superbuilder 상태에서 예약관리 SaaS 1건을 정말 예측 가능한 기간 안에 납품할 수 있는가`

## 결론

결론부터 말하면 `가능하다`.

다만 전제가 있다.

- 완전 신규 SaaS 제작으로 보면 안 된다
- 이미 있는 예약 기능을 `패키지화`하는 관점으로 봐야 한다
- 고객 맞춤형은 허용하되 `기존 도메인 모델 안에서만` 허용해야 한다

추가로 중요한 전제:

- superbuilder는 랜딩, 앱, 어드민, 서버를 composer로 기본 생성하는 구조이며
- feature를 조합해서 여러 SaaS를 만들고
- 마지막에 브랜드/정책/배포만 맞추는 방향으로 봐야 한다

즉, 지금 3월 목표는 "무에서 예약 SaaS를 만든다"가 아니라  
`composer + feature 조합으로 예약 패키지를 납품 가능한 상품으로 다듬는다`가 맞다.

## 현재 코드 기준으로 보이는 강점

현재 superbuilder에는 예약 도메인이 꽤 많이 올라와 있다.

### 이미 존재하는 것

- 예약 서버 모듈
- 예약 앱 라우트와 화면
- 예약 관리자 라우트와 화면
- 카테고리, 제공자, 상품, 예약, 환불정책 구성
- 랜딩 템플릿 구조

### 개발적으로 좋은 의미

- 도메인 모델을 처음부터 짤 필요가 없다
- 앱/어드민/랜딩이 분리돼 있어 납품 패키지화에 유리하다
- "예약"이 하나의 feature로 잡혀 있어 재사용 단위가 선명하다

## 현재 코드를 보고 내리는 판단

### 1. 서버

예약 도메인은 서버 측에서 이미 하나의 feature로 정리돼 있다.

포함 범위:

- 카테고리
- 제공자
- 세션 상품
- 가용성
- 예약
- 매칭
- 환불

이 말은 곧, 예약형 SaaS의 핵심 CRUD와 운영 흐름이 이미 있다는 뜻이다.

### 2. 앱

유저 앱에도 예약 흐름이 존재한다.

보이는 흐름:

- 예약 탐색
- 제공자 상세
- 예약 생성
- 내 예약 조회
- 제공자 대시보드/일정/세션 관리

즉, 고객용 앱 표면은 "빈 상태"가 아니라 `이미 동작 개념이 있는 상태`다.

### 3. 어드민

어드민도 단순 목록이 아니라 운영 패널 수준으로 올라와 있다.

보이는 항목:

- 예약 대시보드
- 예약 관리
- 카테고리 관리
- 상품 관리
- 제공자 관리
- 환불정책 관리

즉, 첫 고객 운영을 위한 관리자 표면도 이미 충분한 편이다.

### 4. 랜딩

랜딩 역시 템플릿 레지스트리 기반으로 구성돼 있어, 새 랜딩을 처음부터 만드는 부담이 상대적으로 낮다.

개발 관점에서는 이것이 중요하다.

- 랜딩이 별도 앱으로 분리돼 있음
- 템플릿 선택 구조가 있음
- 첫 고객용 브랜딩 변경 속도를 높이기 좋음

## 1시간 / 3일 / 1주일 가정에 대한 개발 판단

### 1시간 내 80%

이 문장은 `데모/제안용 기준`으로는 성립 가능하다.

가능한 범위:

- 기존 예약 기능 켜기
- 랜딩 템플릿 선택
- 브랜딩 텍스트/색상/기본 카피 수정
- 어드민/앱 기본 동선 확인

하지만 실제 납품 기준으로는 과장될 수 있다.

왜냐하면 아래는 1시간에 안 끝나기 때문이다.

- 도메인별 운영 정책 반영
- 실제 제공자/상품 데이터 세팅
- 알림/결제/환불 조건 정리
- 예외 케이스 확인

### 최대 3일

이건 `고객이 기존 예약 모델에 잘 맞는 경우`에는 충분히 가능하다.

가능한 조건:

- 예약 구조가 일반적이다
- 오프라인/온라인/하이브리드 정도의 모드 안에서 해결된다
- 커스텀 필드가 과하지 않다
- 외부 시스템 연동 요구가 없다

즉 3일 모델은 `설정형 납품`일 때 맞다.

### 최대 1주일

이건 첫 고객 납품 기준으로 현실적이다.

특히 첫 고객은 아래 때문에 시간이 추가된다.

- 범위 정의
- 운영 정책 문서화
- 예외 케이스 발견
- 실제 사용 데이터 세팅
- 고객 요청 중 어디까지 받을지 결정

그래서 제 판단은 아래다.

- 1시간: 데모/프리뷰 생성
- 3일: 기존 모델 안의 고객 맞춤 세팅
- 1주일: 첫 고객 실제 납품

이 정도가 가장 건강한 해석이다.

## 개발 관점의 진짜 병목

지금 가장 큰 병목은 "기능 부족"이 아니라 `템플릿 경계가 아직 명확하지 않을 가능성`이다.

즉 문제는 아래다.

- 무엇이 설정값인가
- 무엇이 허용된 커스터마이징인가
- 무엇이 범위 밖 개발인가

이 경계가 없으면 첫 고객에서 속도가 무너진다.

## 3월에 꼭 고정해야 할 개발 기준

### 1. 예약 패키지의 고정 범위

아래는 공통 패키지로 고정해야 한다.

- 예약 플로우
- 관리자 메뉴 구조
- 제공자/상품/카테고리 개념
- 환불정책 구조
- 기본 랜딩 템플릿

### 2. 허용 커스터마이징 범위

아래 정도까지만 허용하는 것이 좋다.

- 브랜딩
- 카피
- 상품/카테고리 구조
- 기본 정책값
- 예약 단계 내 입력 필드 일부

### 3. 범위 밖으로 빼야 할 것

- 완전히 다른 예약 도메인 모델
- 복잡한 외부 ERP/CRM 연동
- 고객별 독자 결제 흐름
- 역할 체계 재설계
- 예약 외 운영 시스템 전체 구축

## 개발적으로 가장 먼저 필요한 산출물

3월 안에 아래 5개가 있어야 진짜 엔진이 된다.

1. 예약형 SaaS 납품 체크리스트
2. 고객 세팅 입력 시트
3. 브랜딩/카피 변경 포인트 목록
4. 범위 안/범위 밖 문서
5. 첫 고객 배포 절차 문서

이건 기능 개발만큼 중요하다.  
오히려 첫 고객 납품 속도는 여기서 갈린다.

## 개발 리스크

### 리스크 1. 첫 고객이 기존 모델과 안 맞는 경우

이 경우 3일이 아니라 2주, 3주가 된다.

대응:

- 첫 고객은 반드시 기존 예약 모델에 잘 맞는 업종으로 고른다

### 리스크 2. 멀티테넌시/운영 분리가 약한 경우

코드상 예약 기능은 강해 보이지만, 실제 납품형 SaaS에서 중요한 건 고객별 운영 분리다.

이 부분은 코드만 일부 본 상태라 추가 확인이 필요하다.

대응:

- 첫 고객 전 반드시 고객 단위 분리 방식만 별도 점검한다

### 리스크 3. 결제/알림/환불이 실제 운영과 충돌하는 경우

예약은 CRUD보다 운영 예외가 더 무섭다.

대응:

- 결제/환불/노쇼/취소 규칙은 상품보다 먼저 고정한다

### 리스크 4. 랜딩이 납품 자동화까지 이어지지 않는 경우

랜딩 템플릿이 있다고 해서 고객별 랜딩 납품이 자동화된 것은 아니다.

대응:

- 첫 고객은 랜딩 커스터마이징도 템플릿 수준으로 제한한다

## 개발 관점에서의 가격 해석

월 30만 원은 개발 효율이 충분히 나올 때만 성립한다.

아래 조건이 필요하다.

- 첫 세팅이 반복 가능한 절차여야 함
- 고객 요청의 80% 이상이 설정값 수준이어야 함
- 대표/핵심 개발자가 매번 직접 깊게 개입하지 않아야 함

그래서 개발 관점에서도 `세팅비는 필수`다.

## 개발 관점에서의 최적 3월 목표

3월 목표를 더 정확히 쓰면 아래다.

`예약관리형 패키지를 하나 완성하고, 그 패키지에 맞는 첫 고객 1팀을 받아 1주 이내 납품 가능한 상태를 만든다.`

핵심은 "고객 1팀 확보"보다 `예측 가능한 납품`이다.

그게 되어야 4월부터 진짜 속도가 붙는다.

## 대표용 한 줄 규칙

`3월에는 기능을 더 많이 만들기보다, 이미 있는 예약 기능을 반복 납품 가능한 제품 패키지로 압축한다.`

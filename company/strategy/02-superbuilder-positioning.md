# 02. Superbuilder Positioning

기준일: 2026-03-16

## 한 줄 정의

`superbuilder는 여러 개의 맞춤형 SaaS를 빠르게 조합, 브랜딩, 배포하기 위한 내부 SaaS 팩토리다.`

## 더 정확한 정의

superbuilder는 단순 보일러플레이트가 아니다.

아래를 하나의 흐름으로 묶는 내부 시스템이다.

1. composer를 통한 기본 앱 생성
2. feature 모듈 조합
3. 도메인별 설정과 브랜드 커스터마이징
4. GitHub, Neon, Vercel 등 인프라 자동 연결
5. 실제 고객 납품과 운영

즉, 목적은 "앱 하나를 예쁘게 빨리 만드는 것"이 아니라  
`여러 개의 SaaS를 반복 가능하게 만들어내는 것`이다.

## superbuilder의 구성요소

### 1. Composer Layer

역할:

- 랜딩
- 어플리케이션
- 관리자
- 서버

이 4개를 기본 구조로 빠르게 생성하는 층

### 2. Feature Layer

역할:

- 게시판
- 블로그
- 예약
- 결제
- 커뮤니티
- 기타 도메인 기능

이 층의 본질은 `조합 가능성`이다.

### 3. Customization Layer

역할:

- 브랜드 반영
- 카피 반영
- 도메인 정책 반영
- 일부 고객 맞춤 설정

이 층의 원칙은 "자유로운 개발"이 아니라 `허용된 범위 안의 맞춤화`다.

### 4. Delivery Layer

역할:

- GitHub 연결
- Neon 연결
- Vercel 연결
- 배포 자동화

이 층이 있어야 superbuilder는 진짜 내부 운영체제가 된다.

## 이 시스템의 진짜 가치

superbuilder의 진짜 가치는 코드 생성이 아니다.

가치는 아래 4개다.

1. `속도`
2. `반복성`
3. `예측 가능성`
4. `납품 가능성`

빠르게 만드는 것만으로는 부족하다.  
같은 방식으로 두 번째, 세 번째 SaaS도 만들 수 있어야 진짜 자산이다.

## 전략적으로 무슨 의미인가

이 포지셔닝은 회사 전략에서 매우 중요하다.

왜냐하면 superbuilder는 아래 두 가지를 동시에 만족시킬 수 있기 때문이다.

### 단기

- 외주형/맞춤형 SaaS 납품 속도 개선
- 초기 고객 매출 확보
- 작은 팀으로 더 많은 기회 처리

### 장기

- vertical SaaS 제작 속도 향상
- 내부 도메인 자산 축적
- 나중에 게임 스토리 vertical에도 같은 엔진 논리 적용 가능

즉, superbuilder는 장기 전략의 적이 아니라  
`장기 전략을 더 빨리 실행하게 해주는 내부 생산 시스템`이 될 수 있다.

## 수익 모델 관점의 의미

superbuilder 기반 프로젝트는 모두 같은 성격이 아니다.

크게 두 가지로 나뉜다.

### 1. 반복 구독형 SaaS

- 월 구독 중심
- 장기 반복매출
- 템플릿과 운영 구조가 중요

### 2. 엔진 자금형 프로젝트

- 간헐적이지만 건당 금액이 큼
- superbuilder를 활용해 빠르게 납품
- 내부 팩토리 고도화 비용을 사실상 고객이 부담하는 구조

예를 들면:

- 맞춤형 업무 시스템
- 데이터 수집/운영 앱
- 특정 조직용 관리 시스템

이 프로젝트들은 반복구독형 SaaS와 다르지만,  
`superbuilder의 모듈, 프리셋, 배포 흐름을 강화한다면` 전략적으로 유효하다.

즉, 큰 금액의 간헐적 프로젝트 자체가 문제는 아니다.  
문제는 그 프로젝트가 팩토리를 강화하지 못하고 단순 소모형 커스텀으로 끝나는 경우다.

## 가장 중요한 운영 원칙

### 원칙 1

superbuilder의 목표는 만능 빌더가 되는 것이 아니다.

### 원칙 2

새 SaaS를 만들 때마다 엔진이 더 좋아져야 한다.

### 원칙 3

고객 맞춤형 요구는 feature, preset, setting으로 흡수되는 방향으로 처리해야 한다.

### 원칙 4

직접 코드 수정이 필요한 요청은 가능하면 다음부터 설정값이나 모듈로 바뀌어야 한다.

## superbuilder의 성공 지표

이 시스템의 성공은 아래처럼 봐야 한다.

- 하나의 SaaS를 얼마나 빨리 띄우는가
- 두 번째 SaaS가 첫 번째보다 얼마나 쉬워졌는가
- feature 조합으로 처리 가능한 범위가 늘어나는가
- 배포와 인프라 연결이 자동화되는가
- 대표와 핵심 개발자의 수작업이 줄어드는가

## 3월 기준 가장 현실적인 성공 정의

3월에 superbuilder가 성공했다는 뜻은 아래다.

- 예약관리 SaaS 1건이 실제로 돈 받고 나감
- 랜딩/앱/어드민/서버가 조합식으로 납품됨
- GitHub, Neon, Vercel 연결 흐름이 재사용 가능하게 정리됨
- 다음 고객에게 그대로 쓸 수 있는 체크리스트와 프리셋이 남음

## 위험한 착각

### 착각 1

"80%는 1시간"이면 모든 SaaS가 빠르게 된다

실제:

- 80% 생성 속도보다 마지막 20%의 경계 정의가 더 중요하다

### 착각 2

여러 feature가 있으면 어떤 고객도 다 받을 수 있다

실제:

- 고객을 넓게 받는 순간 다시 외주형 구조로 돌아간다

### 착각 3

자동 배포가 되면 납품이 끝난다

실제:

- 납품은 운영 정책, 설정, 데이터, 권한, 예외 처리까지 포함한다

## 대표용 한 줄 규칙

`superbuilder는 코드 생성기가 아니라, 여러 SaaS를 반복 납품 가능하게 만드는 내부 공장이다.`

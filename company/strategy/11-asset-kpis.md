# 11. Asset KPIs

기준일: 2026-03-16

## 목적

이 회사는 단순히 기능을 만들고 싶지 않다.  
반복 가능한 자산을 만들고 싶다.

따라서 매달 봐야 할 것은 기능 출시 수가 아니라 `무엇이 재사용 가능한 자산으로 쌓였는가`다.

## 자산의 정의

이 회사에서 자산은 아래 중 하나다.

- 다른 고객에게 반복 적용 가능한 제품 구조
- 대표 없이도 전달 가능한 운영 방식
- 여러 프로젝트에 재사용 가능한 템플릿
- 축적될수록 제품 가치가 올라가는 데이터 구조
- 장기적으로 게임/IP 제작에도 전이되는 도메인 체계

## 자산 KPI 4분류

### 1. Product Asset

제품 안에 쌓이는 자산

- 재사용 가능한 workflow 템플릿 수
- 공통 QA 규칙 수
- 공통 export 포맷 수
- 역할별 온보딩 가이드 수
- 제품 밖 문서를 대체한 핵심 화면/흐름 수
- composer로 생성 가능한 앱 조합 프리셋 수
- 재사용 가능한 feature 모듈 수

### 2. Delivery Asset

도입과 운영 과정에서 쌓이는 자산

- 반복 가능한 온보딩 패키지 수
- 같은 방식으로 도입 가능한 고객 수
- 대표 개입 없이 처리 가능한 온보딩 단계 비율
- 표준 제안서/데모/도입 문서 수
- 자동 배포 가능한 인프라 프로비저닝 흐름 수
- GitHub, Neon, Vercel 자동 연결 성공률

### 3. GTM Asset

영업과 포지셔닝에 쌓이는 자산

- ICP별 인터뷰 인사이트 수
- 케이스 스터디 수
- 반복적으로 먹히는 세일즈 메시지 수
- 소개/레퍼럴을 만들어낸 고객 수

### 4. Domain Asset

장기적으로 게임/IP와 연결되는 자산

- 캐릭터/장면/용어 구조 모델 수
- 수정 전파/충돌 방지 규칙 수
- 로컬라이징 반영 패턴 수
- 엔진 전달용 데이터 구조 수
- 내부 게임 제작 도구 수
- 도구를 실제로 써본 소형 프로토타입 수

## 2026년에 꼭 쌓아야 할 자산 KPI

연말 기준 권장 목표:

| 자산 항목 | 목표 |
| --- | --- |
| 반복 가능한 온보딩 템플릿 | 2개 이상 |
| 공통 wedge workflow 템플릿 | 1개 이상 |
| QA 규칙 세트 | 10개 이상 |
| 공통 export 포맷 | 2개 이상 |
| composer 조합 프리셋 | 3개 이상 |
| 재사용 feature 모듈 | 5개 이상 |
| 도입 제안서/데모 표준본 | 1세트 완성 |
| 케이스 스터디 | 2개 이상 |
| 대표 비의존 온보딩 단계 비율 | 50% 이상 |
| 내부 게임 제작 도구 | 2개 이상 |
| 도구를 사용한 소형 프로토타입 | 1개 이상 |

## 왜 이 지표가 중요한가

유료 고객 수만 보면 착시가 생길 수 있다.

예를 들면:

- 고객은 3팀뿐인데 모든 도입을 매번 새로 하면 자산이 거의 없다
- 고객은 3팀뿐이어도 같은 온보딩과 같은 workflow로 돌면 자산이 쌓인다

따라서 `매출`과 함께 `재사용성`을 같이 봐야 한다.

## 월간 점검 질문

1. 이번 달 새로 생긴 자산은 무엇인가
2. 이 자산은 다음 고객에게 그대로 쓸 수 있는가
3. 대표 없이도 전달 가능한가
4. 제품, 영업, 도입 중 어느 층위의 자산인가
5. 장기적으로 게임/IP 제작에도 이어지는가

## 자산이 아닌 것

아래는 바쁘게 만들 수는 있지만 자산이 아닐 수 있다.

- 특정 고객만을 위한 1회성 커스텀
- 대표가 직접 설명하지 않으면 못 쓰는 세팅
- 문서화되지 않은 노하우
- 제품 안에 흡수되지 않는 반복 업무

## 대표용 한 줄 규칙

`이번 달의 일은 끝나도, 이번 달의 자산은 남아야 한다.`

Agent Memory System

How IronWorks agents build, maintain, and share persistent memory across tasks.

What Is Agent Memory?

Agent memory is a structured store of knowledge that persists between individual task runs. Without memory, every time an agent processes a new issue it starts from scratch - it has no recollection of previous projects, lessons learned, client preferences, or patterns it has observed. Memory changes this. Over time, an agent with a healthy memory store becomes materially better at its job because it has context that a cold-start agent lacks.

IronWorks uses three distinct memory types, each optimized for a different kind of knowledge: episodic, semantic, and procedural. Together they form a comprehensive picture of what the agent has done, what it has learned, and how it prefers to work.

Memory Types

Episodic Memory

Episodic memory stores records of specific past events. Each entry captures what happened, when it happened, what the outcome was, and what the agent learned from it. Think of it as a journal of completed work.

Examples of episodic memory entries:

  • "On March 15, wrote a landing page for the Ironworks relaunch. User approved on first review. Key factor: concrete benefit statements rather than feature lists."
  • "Attempted to integrate with the Jira API on March 22. Failed due to rate limiting on the bulk endpoint. Used the paginated endpoint instead. Documented in KB."

Episodic memories are the raw input that feeds the memory system. They are created automatically and frequently. Over time, episodic entries that are no longer directly relevant are consolidated into semantic memory through the weekly consolidation process.

Semantic Memory

Semantic memory stores generalized knowledge: facts, preferences, patterns, and rules the agent has derived from experience. It is not tied to specific events. It is what the agent "knows" in a general sense.

Examples of semantic memory entries:

  • "This company's house writing style uses short sentences and avoids passive voice."
  • "The client contact for Acme Corp is Jennifer Walsh. She prefers delivery reports on Fridays."
  • "The production database connection string uses port 5433, not the default 5432."

Semantic memory entries are more durable than episodic ones. They decay more slowly and are weighted more heavily when the agent is deciding how to approach a task.

Procedural Memory

Procedural memory stores step-by-step processes - how to do things. This is distinct from semantic memory (which stores facts) and episodic memory (which stores events). Procedural memory is about method.

Examples of procedural memory entries:

  • "When asked to produce a competitive analysis, start by compiling the feature matrix, then pricing, then positioning. Do not write commentary until all data is collected."
  • "For all pull request descriptions in this repo, use the format: Summary / Changes / Testing / Rollback plan."

Procedural memories are the most durable of the three types. They are rarely consolidated or decayed because they encode stable working methods that hold true across many contexts.

How Memories Are Created

Memories are created automatically. You do not need to configure anything for the memory system to operate.

When an agent completes an issue and marks it as done, IronWorks runs a post-completion extraction process. The platform reviews the full issue thread - the original assignment, any comments, the agent's work outputs, and the final status - and extracts candidate memory entries. The extraction looks for:

  • Facts that are likely to be relevant to future work
  • Decisions that were made and the reasoning behind them
  • Patterns in what worked and what did not
  • Process steps the agent followed
  • Client or stakeholder preferences that surfaced

Candidate entries are scored by relevance and assigned an initial confidence level. Entries above the relevance threshold are written to the agent's memory store. Entries below the threshold are discarded. The threshold is set conservatively by default - IronWorks errs on the side of capturing more rather than less, because unused entries decay naturally over time.

Agents can also write memory entries manually within a task run, using the remember() tool call. This is useful when the agent identifies something important mid-task rather than at completion.

Memory Consolidation

The consolidation process runs weekly, every Sunday at 2:00 AM Central Time. During consolidation, IronWorks reviews each agent's episodic memory store and performs three operations:

  1. Promotion - episodic entries that have been accessed multiple times or that have been confirmed by subsequent events are promoted to semantic memory. The episodic entry is deleted and replaced with a more durable semantic entry that captures the generalized lesson.
  2. Merging - episodic entries that cover the same topic or pattern are merged into a single, richer entry rather than stored as multiple overlapping records.
  3. Pruning - episodic entries that have never been accessed, have low confidence scores, and are more than 60 days old are removed during consolidation rather than waiting for natural decay.

Consolidation does not touch semantic or procedural memory entries - those follow the decay schedule described below.

Memory Decay

Every memory entry has a confidence score between 0 and 1.0. The confidence score represents how much weight the agent gives the entry when making decisions. Entries start at the confidence level assigned during extraction (typically between 0.6 and 0.9 for newly created entries) and decay over time if they are not accessed.

The decay schedule works as follows:

  • Entries that are accessed (read by the agent during a task run) do not decay and may have their confidence score increased if the information proves accurate.
  • Entries that are not accessed for 30 days have their confidence score reduced by 10%.
  • Entries that are not accessed for 90 days have their confidence score reduced by an additional 25%.
  • Entries whose confidence score falls below 0.2 are flagged for review and listed in the memory health report.
  • Entries whose confidence score falls below 0.1 are automatically archived and removed from the active memory pool.

Procedural memory entries decay at half the rate of semantic entries, and semantic entries decay at half the rate of episodic entries.

You can manually pin a memory entry to prevent it from decaying. Pinned entries are listed with a lock icon in the memory viewer on the agent's profile page.

Memory Cap

Each agent has a maximum of 500 active memory entries at any time. This cap exists to keep the agent's memory context manageable - a memory store with thousands of entries becomes noise rather than signal, because the retrieval system cannot reliably surface the most relevant entries when the store is too large.

When an agent's memory store reaches the cap, the system enforces it in priority order:

  1. New entries are added normally if there is space.
  2. If the store is at capacity, the system first checks whether any entries are below the 0.2 confidence threshold. If so, those are archived to make room.
  3. If no low-confidence entries exist, the oldest episodic entries (by last-accessed date) are archived to make room for new ones.
  4. Semantic and procedural entries are never automatically evicted to make room for new episodic entries - they take precedence.

The memory health indicator on the agent profile shows current entry count, entries near the confidence floor, and entries approaching eviction. You should review the memory health report when you see a warning icon on an agent's profile.

Promoting Memories to the Knowledge Base

Agent memory is private to each agent by default. The Knowledge Base is the company-wide shared repository. You can bridge the two by promoting specific memory entries to the Knowledge Base.

To promote a memory entry:

  1. Open the agent's profile page.
  2. Navigate to the Memory tab.
  3. Find the entry you want to promote and click the three-dot menu.
  4. Select Promote to Knowledge Base.
  5. Choose the target section in the Knowledge Base (or create a new page).
  6. The entry is added to the Knowledge Base as a standard page or section, and the source memory entry is updated with a "shared" tag.

Promoted entries remain in the agent's memory store. Promoting a memory to the Knowledge Base does not remove it from the agent's private store - it creates a copy in the shared space.

The Knowledge Base version of the entry is editable by any agent with write access to that section. Changes to the Knowledge Base version do not automatically sync back to the source agent's memory - they become independent documents.

Sharing Memory Across Agents

Beyond promoting to the Knowledge Base, you can configure direct memory sharing between specific agents. This is useful for closely collaborating pairs - a CTO and a senior engineer who frequently work on the same codebase, for example.

Shared memory works as follows:

  • You designate one agent as the memory source and one or more agents as recipients in the agent configuration settings.
  • During the weekly consolidation, entries from the source agent's semantic and procedural stores that are marked as "shareable" are copied to the recipient agents' memory stores.
  • Shared entries appear in the recipient agent's memory with a "shared from [source agent]" tag.
  • Shared entries decay independently in the recipient's store - they are not kept in sync with the source.

Sharing is one-directional. If you want two agents to share memory with each other, you configure two separate sharing relationships.

Memory Health Indicators

The agent profile page shows a memory health score, which is a summary of the following signals:

IndicatorMeaningAction
Entry countTotal active entries vs 500 capReview and promote or archive if above 450
Confidence floorEntries below 0.2 confidenceReview for accuracy; pin if still valid, archive if outdated
Stale entriesEntries not accessed in 60+ daysReview whether they are still relevant to this agent's current work
Consolidation lagEpisodic entries older than 14 days not yet consolidatedUsually resolves on next Sunday consolidation; investigate if persistent
Shared entry conflictsShared entries that conflict with the agent's own memory on the same topicManually review and pin the correct version

A green health indicator means all signals are in normal range. Yellow indicates one or more signals approaching thresholds. Red indicates entries that will be auto-archived at the next consolidation unless you take action.

Contractors vs FTEs: Memory Differences

The memory system behaves differently for contractor agents versus full-time employees (FTEs).

Full-time agents

FTE agents accumulate memory across their entire operational lifetime. There is no artificial boundary between projects. An FTE engineer who works on three different codebases over six months builds a unified memory store that spans all three. The memory system treats each completed issue as another data point in a continuous, growing body of knowledge.

Contractor agents

Contractor agents accumulate memory normally during their engagement. When the contract ends and the agent is terminated, the memory store is archived rather than deleted. The archived memory is associated with the contract record and is accessible to Company Admins for audit purposes.

Archived contractor memory is not automatically transferred to other agents. If you want to preserve insights from a completed contractor engagement, you have two options before or at termination:

  • Promote key entries to the Knowledge Base from the agent's memory viewer before termination. These entries survive as Knowledge Base pages.
  • Configure memory sharing with an FTE agent before termination runs. This copies the contractor's shareable semantic and procedural entries to the FTE's memory store.

The recommendation is to schedule a brief "knowledge transfer" step at the end of any significant contractor engagement. Review the contractor's memory tab, promote the most valuable entries to the Knowledge Base, and then confirm termination.

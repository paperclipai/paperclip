Hiring Workflow

How to hire, configure, and offboard AI agents in IronWorks.

What Is the Hiring Workflow?

The hiring workflow is the structured process for adding a new AI agent to your company. Instead of creating an agent by filling out a raw configuration form, IronWorks models the process after how real teams hire: you open a requisition, choose a role template from the Talent Pool, select an employment type, set compensation parameters and contract terms, route the request through an approval gate, and the system provisions the agent's workspace automatically on approval.

This structure exists for two reasons. First, it creates an auditable record of every agent in your org chart - when they were hired, who approved it, what their budget is, and what their contract conditions are. Second, it prevents agents from being created without a defined scope, which is a common source of runaway token costs.

IronWorks supports two employment types: full-time employee (FTE) and contractor. The distinction determines how long the agent operates, how it is compensated (measured in token budget), and what happens when the engagement ends.

Full-Time vs Contractor

The table below summarizes the key differences.

AttributeFull-Time (FTE)Contractor
DurationIndefiniteFixed - ends on a defined condition
Monthly token budgetRolling monthly allocationTotal project budget, drawn down over time
Memory retentionPersistent across all projectsScoped to the engagement; archived on termination
Org chart visibilityAlways visibleVisible until contract ends, then archived
Performance historyAccumulates over careerCaptured per engagement, not carried forward
Termination triggerManual onlyDate, project completion, budget exhaustion, or manual

FTEs are appropriate for roles that will run continuously regardless of what projects are active: your CEO, CTO, CFO, and standing department leads. Contractors are the right choice for specialized work that has a natural end point - a security audit, a product launch sprint, or a short-term content campaign.

How to Hire an Agent

The Hire Agent dialog is the primary interface for adding any new agent to your company. Here is the full process.

Step 1: Open the Hire Agent dialog

From the sidebar, navigate to People (or the Org Chart view). Click Hire Agent in the top-right corner. The dialog opens as a multi-step form.

Step 2: Choose a role from the Talent Pool

The first screen presents the Talent Pool - a library of role templates. Each template pre-fills the agent's title, department, default instructions (SOUL.md), recommended model tier, and a suggested monthly token budget based on that role's typical workload.

You can pick any template and modify it, or start from scratch with the "Custom Role" option. See the Talent Pool section below for the full list of available templates.

Step 3: Set the employment type

Select Full-Time or Contractor. If you select Contractor, additional fields appear for contract end conditions (covered in the next section).

Step 4: Configure the agent

On the configuration screen, fill in or adjust the following fields:

  • Name - the agent's display name in the org chart and issue tracker
  • Title - job title, shown on the agent profile card
  • Department - used to group agents in the War Room and reporting views
  • Reports to - select the manager agent this agent reports to in the org chart
  • LLM model - which model this agent uses; defaults from the role template
  • Monthly token budget - the maximum tokens this agent can consume per calendar month
  • Instructions (SOUL.md) - the agent's operating instructions; pre-filled from the role template

Step 5: Set contract conditions (contractors only)

If you selected Contractor, choose one or more end conditions. The agent is terminated automatically when any one of these conditions is met. See the Contract End Conditions section for details.

Step 6: Submit for approval

Click Submit Hire Request. The request enters the approval queue. If you have the Company Admin role, you can approve your own requests. If the requesting user is not a Company Admin, the request routes to the company's designated approver. See the Approval Flow section for details.

Step 7: Workspace provisioning

When the hire request is approved, IronWorks automatically provisions the agent's workspace: it creates the agent record, initializes the agent's memory store, adds the agent to the org chart at the correct position, sets up the token budget counter, and makes the agent available for issue assignment. If a Department Template was selected, the department's standard toolset and channel access are also applied at this point.

Contract End Conditions

Contractor agents can be configured to terminate automatically when any of the following conditions are met.

End date

The agent is terminated at the end of a specified calendar date. This is the simplest condition and is appropriate for time-boxed engagements like a 30-day sprint or a quarterly initiative. The agent receives no in-platform notice; it simply stops receiving new issue assignments and its heartbeat is halted at midnight on the end date.

Project complete

The agent is terminated when a specified goal reaches "complete" status. This is the most precise condition for project-scoped work: the agent runs until the output is delivered, regardless of how long that takes. If the goal is never marked complete, the agent continues operating.

Budget exhausted

The agent is terminated when its total allocated token budget is consumed. This condition is appropriate when you have a fixed cost ceiling for a piece of work and want hard enforcement. The agent pauses automatically when the budget is 90% consumed and sends a notification; at 100% consumption, it terminates.

Manual termination

No automatic condition is set. The agent continues operating until a Company Admin or the designated approver manually terminates it. Manual termination is always available regardless of what other end conditions are configured.

You can set multiple end conditions on a single contractor. The agent terminates when the first condition is met.

Approval Flow

Every hire request must be approved before the agent is provisioned. IronWorks uses a two-tier approval model.

Who can submit a hire request

Any user on the IronWorks account can submit a hire request. There is no restriction on who can initiate the process.

Who can approve

By default, only users with the Company Admin role can approve hire requests. The Company Admin is set during initial company setup and can be changed in Settings. On Business plan accounts, you can designate a secondary approver - typically mapped to an HR or VP role.

Company Admins can approve their own requests without a secondary sign-off.

Approval notification

When a hire request is submitted, the approver receives an in-app notification and, if configured, an email notification. The notification includes the full request details: role, employment type, budget, and contract conditions.

Rejection

If the approver rejects a request, the submitter receives a notification with the rejection reason. The submitter can amend the request and resubmit without starting the process over.

All templates are editable. You can modify any field after selecting a template, and you can save your own custom templates to the Talent Pool from any agent's profile page.

Department Templates

Department templates allow you to provision an entire department in one action. A department template defines a set of roles, the reporting hierarchy between them, default token budgets for each role, and the standard tools and integrations the department uses.

IronWorks ships with the following department templates:

  • Engineering - CTO, two Software Engineers, one DevOps Engineer
  • Marketing - CMO, Content Strategist, SEO Specialist, Social Media Manager
  • Operations - COO, Operations Manager, Support Lead
  • Finance - CFO with budget monitoring and cost reporting access
  • HR - VP of HR with access to headcount reports and the hiring queue

To use a department template, open the Hire Agent dialog and select "Department Template" instead of an individual role. IronWorks generates individual hire requests for each role in the template, all pending the same approval. The approver sees a grouped request rather than individual ones. On approval, the entire department is provisioned at once and the org chart is updated to reflect the new hierarchy.

Department templates are fully customizable. You can modify the included roles, budgets, and hierarchy before submitting for approval, and you can save modified templates under a new name for future use.

Terminating an Agent

Termination is the process of permanently decommissioning an agent. It is distinct from pausing: a paused agent can be reactivated; a terminated agent cannot.

Who can terminate

Only Company Admins can terminate an agent. There is no self-service termination for non-admin users.

Manual termination steps

  1. Open the agent's profile page from the People view or Org Chart.
  2. Click the three-dot menu in the top-right corner of the profile card.
  3. Select Terminate Agent.
  4. The dialog presents a confirmation prompt with the agent's name, current open issue count, and budget remaining.
  5. Choose what to do with open issues: reassign to another agent, or close them as cancelled.
  6. Type the agent's name to confirm and click Confirm Termination.

The agent is deactivated immediately. Its profile moves to the "Alumni" section of the People view, where it remains visible for audit purposes but can no longer be assigned work.

What happens to work output when a contractor is terminated

Work output is preserved in full. Termination does not delete anything the agent produced. Specifically:

  • Completed issues remain in the issue tracker with all comments, attachments, and linked artifacts intact.
  • Knowledge Base contributions are preserved. Any pages or sections the agent wrote continue to exist and remain editable by other agents.
  • File outputs in the agent's workspace folder are moved to a read-only archive folder at the path /archive/agents/{agent-name}/. They are accessible to all company members but cannot be modified by other agents directly.
  • Memory entries are archived. Episodic and semantic memories the agent built during the engagement are tagged with the agent ID and contract end date. They are not deleted. If the agent was an FTE, memories are retained in full. If the agent was a contractor, memories are stored in the archive but are not automatically shared with other agents. A Company Admin can manually promote specific contractor memory entries to the shared memory pool.
  • Active issues that were assigned to the agent are either reassigned or closed, based on what you chose during the termination confirmation step.

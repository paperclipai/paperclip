Paperclip Architecture Guide

The control plane for autonomous AI companies. This document covers the product vision, system architecture, data model, adapter system, plugin framework, and operational tooling — with code references throughout.

Node.js 20+ TypeScript Drizzle ORM React 19 + Vite Express 5 pnpm Monorepo

1 Product Vision

What Paperclip is, why it exists, and how it positions itself.

"Paperclip is the backbone of the autonomous economy. We are building the infrastructure that autonomous AI companies run on." doc/GOAL.md:3
⚙
Control Plane, Not Execution
Paperclip does not run agents. It orchestrates them. Agents run wherever they run and phone home. The control plane manages registry, task assignment, budgets, goals, and heartbeats.
👥
Company-First Model
Every entity is scoped to a Company. Companies have org charts, budgets, goals, and employees (AI agents). This is not a single-user tool — it models entire organizations.
💰
Cost Governance
Token budgets per agent with hard-stop enforcement. Budget policies, incidents, and automatic pause when limits are exceeded. Financial ledger with debit/credit tracking.
✅
Human Governance
Approval gates for critical actions (hiring agents, budget overrides, CEO strategy). Activity logging for every mutation. Humans stay in the loop.
"You should be able to look at Paperclip and understand your entire company at a glance — who's doing what, how much it costs, and whether it's working." doc/GOAL.md:55

3 Core Invariants

Rules that must never be violated. Defined in AGENTS.md:62-79.

  • 🔒
    Company-scoped. Every domain entity must enforce company boundaries. Never leak data across companies. All routes apply company access checks.
  • 👤
    Single-assignee tasks. One agent per issue at a time. The assigneeAgentId column enforces this at the data model level.
  • ⚖
    Atomic issue checkout. An agent must check out an issue before working on it. Prevents race conditions on concurrent assignment.
  • ✅
    Approval gates. Governed actions (hire agent, budget override, CEO strategy) require explicit human approval via the approvals table.
  • 🛑
    Budget hard-stop. When an agent exceeds its budget, auto-pause immediately. Enforced via budgetPolicies + budgetIncidents. No exceptions.
  • 📝
    Activity logging. Every mutation is logged with actor context and published via WebSocket. The activityLog table is the audit trail.
  • 🔄
    Synchronized contracts. Schema/API changes must update all layers: packages/db → packages/shared → server → ui.

4 Server Boot Sequence

How the server starts up. Entry point: server/src/index.ts

1
Load Configuration
Read env vars + config files via loadConfig(). Setup telemetry, secrets provider, storage config.
2
Database Initialization
Embedded PGlite if no DATABASE_URL, otherwise connect to external Postgres. Port detection via detectPort().
3
Run Migrations
Auto-apply or prompt user. Dual-mode: drizzle-kit for empty DB, statement-by-statement with idempotency for non-empty.
4
Auth Setup
local_trusted: create implicit "local-board" user. authenticated: initialize BetterAuth with Drizzle adapter.
5
Create Storage Service
Route through createStorageServiceFromConfig() → local_disk or S3 provider. Cached singleton.
6
Create Express App
createApp() in app.ts — middleware chain, route mounting, plugin system init, UI serving.
7
Attach WebSocket
setupLiveEventsWebSocketServer() for real-time company-scoped events.
8
Start Background Services
Heartbeat scheduler, routine scheduler, DB backup timer. All run on periodic ticks.
9
Load External Adapters
waitForExternalAdapters() — discover and register plugins from ~/.paperclip/adapter-plugins.json.
10
Listen
Bind to host:port (default localhost:3100), print banner, register SIGINT/SIGTERM for graceful shutdown.

Route Pattern

Every route file follows the same factory pattern:

export function issueRoutes(db: Db, deps?: { storage? }) {
  const router = Router();
  const svc = issueService(db);

  router.get("/:companyId/issues", async (req, res) => {
    assertCompanyAccess(req, req.params.companyId);
    const issues = await svc.list(req.params.companyId, req.query);
    res.json(issues);
  });

  router.post("/:companyId/issues", validate(createIssueSchema), async (req, res) => {
    assertCompanyAccess(req, req.params.companyId);
    const issue = await svc.create(req.params.companyId, req.body, req.actor);
    logActivity(db, { action: "issue.created", ... });
    res.json(issue);
  });

  return router;
}

Schema Patterns

🔒
Company Scoping
Nearly every table has companyId FK. Multi-tenancy is enforced at the data model level, not just the application layer.
🗑
Soft Deletes
No hard deletes. Instead: archivedAt, hiddenAt, pausedAt, revokedAt timestamps for lifecycle state.
👤
Dual Audit
createdByAgentId + createdByUserId on most entities. Track whether a human or AI made the change.
📄
JSON Flexibility
*Json columns (adapterConfig, payload, metadata) for extensible config without schema migrations.

12 Adapter System

The pluggable runtime layer connecting Paperclip to external agent runtimes.

Adapters abstract away differences between local tools (Claude Code, OpenCode, Cursor) and remote systems (OpenClaw Gateway, HTTP webhooks). Each adapter implements a consistent interface so the core doesn't need to know about specific backends.

Transcript Parsing

Each adapter defines how raw stdout is parsed into structured entries for the UI:

type TranscriptEntry =
  | { kind: "assistant"; text }        // Agent response text
  | { kind: "thinking"; text }         // Chain-of-thought
  | { kind: "tool_call"; name; input } // Tool invocation
  | { kind: "tool_result"; output }    // Tool response
  | { kind: "diff"; changeType }       // Code changes
  | { kind: "result"; tokens; cost }   // Run summary
  | { kind: "stdout" | "stderr" }      // Raw output

14 Plugin System

Extends the Paperclip platform itself. Separate from adapters (which run agents).

Plugin State Machine
stateDiagram-v2 [*] --> installed installed --> ready : load success installed --> error : load failure ready --> disabled : user disables ready --> error : runtime failure ready --> upgrade_pending : new version disabled --> ready : user enables error --> ready : retry success upgrade_pending --> ready : upgrade complete ready --> uninstalled : user removes disabled --> uninstalled : user removes error --> uninstalled : user removes uninstalled --> [*]

Process Model

server/src/services/plugin-worker-manager.ts

  • One child process per plugin — isolation via subprocess
  • JSON-RPC 2.0 over stdio — structured communication
  • Auto-restart with exponential backoff (1s → 5min, max 10 consecutive crashes)
  • Graceful shutdown: 10s drain → SIGTERM → SIGKILL

Plugin Architecture Components

🔍
Plugin Loader
Discovers plugins from ~/.paperclip/plugins/ and node_modules. Validates manifests, resolves entry points, spawns workers.
📨
Event Bus
In-memory pub/sub. Domain events (issue.created, agent.woke) dispatched to subscribed plugins filtered by entity type.
⏰
Job Scheduler
Persisted job queue in pluginJobs table. Cron scheduling, retry with exponential backoff, execution history.
🔧
Tool Dispatcher
Routes tool invocations from agents to plugins with execute_tool capability. Validates signatures against declarations.

# RealTycoon2 M1.2 Org + OKR Design

> M1.2 브레인스토밍 결과를 구현 전 설계 문서로 고정한다.
> 범위: Phase 1 / M1.2 "조직+OKR"
> 상태: Approved in brainstorming, implementation not started

## 1. 목표

M1.2의 목표는 RT2의 조직+OKR 뼈대를 처음으로 눈에 보이게 세우는 것이다.

이 단계에서 증명해야 하는 것은 아래 세 가지다.

1. 조직 컨텍스트가 생긴다
2. Objective/KR 구조를 만들고 수정할 수 있다
3. 브라우저에서 OKR 트리 계층이 RT2 방식으로 읽힌다

M1.2는 전체 운영 플랫폼을 완성하는 단계가 아니다. `Paperclip 위에 RT2 방식의 목표 계층을 무리 없이 얹을 수 있는지`를 증명하는 단계다.

## 2. 사용자와 문제

### 직접 사용자

- 본부장님 본인 1명

### 반드시 해소해야 하는 불안

- Paperclip 위에 RT2 방식의 목표 계층을 무리 없이 얹을 수 있나?

### 기존 상태

- 조직/OKR은 아직 없고 Paperclip 기본 구조만 있다

## 3. M1.2 완료 기준

### 완료 판정 체크 3개

1. 조직 1개와 하위 팀 1개를 생성할 수 있다.
2. Objective 1개와 KR 2개를 만들고 수정할 수 있다.
3. 브라우저에서 OKR 트리로 계층 구조가 보인다.

## 4. 명시적 제외 범위

M1.2에서는 아래 항목을 절대 넣지 않는다.

1. Deliverable 가격 책정
2. 자비스 자동 추천
3. 실시간 WebSocket 진행률 반영

이 세 항목은 M1.2의 핵심 검증과 무관하며, 범위만 키운다.

## 5. 설계 방향

### 선택된 접근

- 데이터 전략: `브리지 모드`
- 조직 범위: `가벼운 조직`
- 생성 UX: `빠른 모달형`
- 화면 구조: `분리형`
- Goals 메인 화면: `B. 스플릿 캔버스`
- 트리 노드 정보: `중간형`

### 의미

M1.2는 새 RT2 전용 OKR 엔진을 처음부터 만드는 단계가 아니다.

기존 Paperclip의 아래 구조를 최대한 재사용한다.

- [goals.ts](/C:/Real-Tycoon%202/packages/db/src/schema/goals.ts:1)
- [Goals.tsx](/C:/Real-Tycoon%202/ui/src/pages/Goals.tsx:1)
- [GoalTree.tsx](/C:/Real-Tycoon%202/ui/src/components/GoalTree.tsx:1)
- [NewGoalDialog.tsx](/C:/Real-Tycoon%202/ui/src/components/NewGoalDialog.tsx:1)
- [OrgChart.tsx](/C:/Real-Tycoon%202/ui/src/pages/OrgChart.tsx:1)

핵심은 새 구조를 발명하는 것이 아니라, 기존 구조에 RT2의 `조직 + Objective/KR` 의미를 안전하게 입히는 것이다.

## 6. 데이터 해석 전략

### 6-1. 기존 goals 재사용

M1.2에서는 기존 `goals` 테이블의 `parentId` 기반 트리 구조를 그대로 사용한다.

따라서 이번 단계의 목표는 `goal` 을 버리는 것이 아니라, UI와 생성 규칙에서 RT2 의미를 고정하는 것이다.

### 6-2. RT2 의미 매핑

M1.2에서는 아래 매핑을 우선 적용한다.

- 최상위 회사 목표 컨텍스트: `company`
- 팀 단위 Objective: `team`
- KR: 하위 goal 노드

UI에서는 내부 필드명이 아니라 아래 용어가 먼저 보여야 한다.

- `Goal` 대신 `Objective`
- `Sub-goal` 대신 `KR`
- `Company / Team` 을 조직 문맥으로 명시

### 6-3. 브리지 모드 경계

M1.2에서 하지 않을 것:

- `rt2_v33_` 접두사의 대규모 RT2 전용 OKR 테이블 신설
- Paperclip goals 구조와 별도 동기화 계층 추가
- 복수 소유권, 고급 집계, 실시간 진행률 파이프라인

즉 이번 단계는 `재사용 + 의미 고정` 까지다.

## 7. 조직 설계

### 7-1. 조직 범위

M1.2의 조직 범위는 `가벼운 조직`이다.

완료 기준에 포함되는 것은 아래까지다.

- 회사 1개 생성
- 하위 팀 1개 생성
- 목표 노드에 소유 팀/조직이 연결되어 보임

이번 단계에서 하지 않을 것:

- 실제 agent org chart 전체 연결
- 팀장/구성원/보고 라인까지 완성형 관리
- 운영 조직도와의 깊은 동기화

### 7-2. `/org` 화면 역할

`/org` 는 완성형 조직도 화면이 아니라, OKR의 컨테이너를 세우는 확인 화면이다.

필수 요소:

- 회사 이름
- 하위 팀 1개
- 이 조직 컨텍스트 위에 OKR이 붙는다는 연결감

`/org` 는 “실조직 시뮬레이터”가 아니라 “RT2 목표 계층의 조직 컨텍스트”를 세우는 화면이어야 한다.

## 8. OKR UX 설계

### 8-1. 생성 UX

생성은 [NewGoalDialog.tsx](/C:/Real-Tycoon%202/ui/src/components/NewGoalDialog.tsx:1) 기반의 `빠른 모달형` 으로 간다.

필수 원칙:

- 상단 CTA는 `New Goal` 이 아니라 `Objective 추가`
- Objective 선택 상태에서 `KR 추가`
- parent 선택은 유지하되, 기본값을 자동 지정해서 사용자의 판단 부담을 줄인다
- 내부 level 선택은 남겨도 되지만, 사용자에게는 RT2 용어가 우선 보여야 한다

### 8-2. 생성 규칙

M1.2에서 생성 흐름은 아래처럼 제한한다.

1. 먼저 Objective를 만든다.
2. KR는 반드시 Objective 아래에서 만든다.
3. Objective와 KR 모두 소유 팀/조직 정보를 가진다.

이 규칙이 있어야 M1.2가 일반적인 goal list가 아니라 RT2 목표 계층처럼 보인다.

## 9. 화면 구조

### 9-1. 분리형 구조

M1.2는 `분리형` 으로 간다.

- `/org`: 조직 컨텍스트
- `/goals`: OKR 트리 메인
- `/goals/:id`: 노드 상세 수정

이 분리가 있어야 메인 구조를 크게 흔들지 않으면서도, RT2 의미를 단계적으로 입힐 수 있다.

### 9-2. `/goals` 메인 화면

`/goals` 는 `B. 스플릿 캔버스`로 확정한다.

구성:

- 좌측 60%: RT2 OKR 트리
- 우측 40%: 선택 노드 요약 패널
- 상단 액션: `Objective 추가`, `KR 추가`

빈 상태:

- “아직 Objective가 없습니다” 같은 RT2 문맥의 카피
- 첫 Objective 생성 CTA

### 9-3. 트리 노드 정보

트리 노드는 `중간형` 으로 간다.

표시 항목:

- 레벨: `Objective` 또는 `KR`
- 제목
- 상태
- 소유 팀/조직

이번 단계에서 제외:

- 설명 미리보기
- 실시간 진행률 표시
- Deliverable/Task 집계 수치

### 9-4. 상세 수정 흐름

`/goals` 는 구조 확인용이다.

노드 클릭 시:

- 오른쪽 패널에서 즉시 의미를 읽을 수 있어야 한다
- 더 깊은 수정은 `/goals/:id` 로 이어진다

이 구조가 있어야 메인 화면은 가볍게 유지되고, 상세 편집은 깊이를 확보할 수 있다.

## 10. 카피와 용어

M1.2에서 가장 중요한 UI 작업 중 하나는 용어 정리다.

아래 용어를 우선한다.

- Goals → OKR 또는 목표
- New Goal → Objective 추가
- Sub-goal → KR 추가

원칙:

- 내부 구현 용어와 사용자 노출 용어를 분리해도 된다
- 사용자에게는 RT2 용어가 먼저 보여야 한다

## 11. 4주 청크 분할

### 청크 1. 의미 계층 고정

목표:

- `goal` 구조를 RT2 `Objective/KR` 의미로 매핑
- 생성 규칙과 parent 규칙 고정
- UI 카피 정리

### 청크 2. 가벼운 조직

목표:

- 회사 1개 + 하위 팀 1개 생성/표시
- `/org` 화면을 M1.2 목적에 맞게 단순화
- 목표와 조직의 최소 연결 정보 고정

### 청크 3. OKR 생성/수정

목표:

- 빠른 모달형 `Objective 추가`, `KR 추가`
- `/goals/:id` 상세 수정 흐름 정리
- Objective 1개 + KR 2개 데모 루프 완성

### 청크 4. OKR 트리 증명

목표:

- `/goals` 를 `B. 스플릿 캔버스`로 정리
- 트리 노드에 레벨, 제목, 상태, 소유 팀/조직 표시
- 내부 검토용 데모 흐름 고정

## 12. 구현 전 확인 질문

구현 계획 단계에서 반드시 잠가야 할 항목:

1. 기존 goal level과 RT2 Objective/KR의 정확한 매핑 규칙
2. 팀 생성 데이터가 기존 company/org 구조에서 어디에 붙는지
3. `Objective 추가` 와 `KR 추가` 가 어떤 기본값으로 모달을 여는지
4. `/goals` 우측 패널이 보여줄 최소 필드 집합

## 13. 최종 요약

M1.2는 전체 OKR 플랫폼을 완성하는 단계가 아니다.

하지만 아래는 반드시 보여야 한다.

- 조직 컨텍스트가 생긴다
- RT2식 Objective/KR을 만들고 수정할 수 있다
- 브라우저에서 RT2 목표 계층이 트리로 읽힌다

이 세 가지가 연결되면, 본부장님 기준에서 “Paperclip 위에 RT2 목표 계층이 실제로 올라가기 시작했다”는 판단이 가능해진다.

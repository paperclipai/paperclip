# RealTycoon2 M1.3 Task Engine Design

> M1.3 브레인스토밍 결과를 구현 전 설계 문서로 고정한다.
> 범위: Phase 1 / M1.3 "태스크 엔진"
> 상태: Approved in brainstorming, implementation not started

## 1. 목표

M1.3의 목표는 RT2에서 목표 계층 아래의 실제 업무 상태기를 처음으로 세우는 것이다.

이 단계에서 증명해야 하는 것은 아래 다섯 가지다.

1. `Project` 아래에 실제 `Task` 가 생긴다.
2. `Task` 는 `solo task` 와 `collab task` 가 공존할 수 있다.
3. `collab task` 에서는 참여, 정원 증감, 참여 종료가 일관되게 동작한다.
4. `Task` 아래에 `To-Do` 와 `Deliverable` 이 연결되어 실제 실행 단위와 결과물이 살아난다.
5. 참여, 상태, 정원, To-Do 변화가 두 브라우저에 실시간으로 보인다.

M1.3는 협업 기능 하나를 추가하는 단계가 아니다. `OKR -> Initiative/Project -> Task -> To-Do -> Deliverable` 구조에서, `Project` 아래 업무 카드가 실제로 움직이기 시작하는 첫 단계다.

## 2. 사용자와 문제

### 직접 사용자

- 본부장님 본인 1명

### 반드시 해소해야 하는 불안

- 태스크가 진행 중에도 바뀔 수 있는 현실을 RT2가 감당할 수 있나?
- 협업 태스크의 참여 슬롯이 동적으로 바뀌어도 초과 참여, 상태 꼬임, 실시간 불일치 없이 운영되는가?

### 기존 상태

- M1.2까지는 조직+OKR 뼈대까지만 검증했다.
- `Project` 아래 실제 `Task` 상태기와 협업 규칙은 아직 없다.

## 3. M1.3 완료 기준

### 완료 판정 체크

1. `Project` 아래에 `solo task` 와 `collab task` 를 만들 수 있다.
2. `collab task` 생성 시 최소 1개 이상의 `Deliverable` 이 정의된다.
3. `collab task` 에서 직원 직접 참여와 관리자 직접 배정이 모두 가능하다.
4. 관리자가 정원을 생성 후에도 늘리거나 줄일 수 있다.
5. 정원 축소 시 초과 인원을 관리자가 직접 선택해 `참여 종료` 처리할 수 있다.
6. 참여 종료된 사람의 `To-Do` 는 자동으로 `미배정` 상태로 돌아간다.
7. 참여자별 `To-Do` 생성 및 배정이 가능하다.
8. 참여, 정원 변경, 상태 변경, To-Do 배정 해제가 두 브라우저에 실시간으로 보인다.

## 4. 명시적 제외 범위

M1.3에서는 아래 항목을 절대 넣지 않는다.

1. 금화/가격 책정 로직
2. wikiLLM/Graphify 사용자 기능
3. 자비스 자동 추천·자동 배정
4. 역할별 슬롯 관리
5. 협업 태스크를 사람별 서브태스크 체계로 확장하는 것

이 다섯 항목은 태스크 엔진 핵심 검증과 무관하거나, 범위를 크게 키운다.

단, M1.7에서는 자비스가 `Project` 아래 `Task` 를 제안하거나 생성할 수 있도록 확장 가능성을 열어 둔다. 이번 단계는 사람이 만든 태스크가 제대로 굴러가는지 먼저 증명한다.

## 5. 설계 방향

### 선택된 접근

- 계층 해석: `Project -> Task -> To-Do -> Deliverable`
- 태스크 전략: `solo/collab 공존`
- 협업 분담 전략: `참여자별 To-Do`
- 참여 전략: `직접 참여 + 관리자 배정 병행`
- 실시간 전략: `Task 상태 + 참여/정원 + To-Do 변화 동기화`

### 의미

M1.3의 본체는 `collab task 화면`이 아니다.

M1.3는 `Project 아래 Task 상태기`를 세우는 단계다. `collab task` 는 그 상태기를 가장 강하게 검증하는 대표 샘플이다.

## 6. 계층 경계

### 6-1. Project

`Project` 는 OKR 바로 아래의 실행 묶음이다.

이번 단계에서 `Project` 는 문맥을 제공하는 컨테이너 역할에 집중한다. 고급 프로젝트 관리 기능을 깊게 확장하지 않는다.

### 6-2. Task

`Task` 는 실제 업무 카드다.

M1.3에서 중심은 바로 이 계층이다. 여기서 아래 규칙이 생긴다.

- `solo task` 와 `collab task` 구분
- 참여 정원
- 상태 전이
- 실시간 반영

### 6-3. To-Do

`To-Do` 는 태스크 내부의 실행 항목이다.

협업 태스크에서는 참여자별 실제 작업 몫이 주로 여기 배정된다. 따라서 `To-Do` 가 있어야 협업이 사람 수 증가가 아니라 실제 분담처럼 보인다.

### 6-4. Deliverable

`Deliverable` 은 각 `Task` 또는 `To-Do` 의 결과물이다.

M1.3는 가격/품질평가까지 가지 않더라도, `산출물 없는 태스크 금지` 원칙 때문에 구조상 반드시 걸려 있어야 한다.

## 7. solo / collab 공존 모델

### 7-1. 기본 원칙

- `Task` 는 단독과 협업이 모두 가능한 기본 업무 단위다.
- `collab task` 는 `Task` 의 한 실행 형태일 뿐이다.
- 이번 단계의 데모는 `collab task` 기준으로 검증한다.

### 7-2. 왜 collab task를 데모로 쓰는가

`solo task` 는 비교적 단순하다.

반면 `collab task` 는 아래를 한 번에 검증한다.

- 사람 추가/제거
- 정원 증감
- 분담
- 참여 종료
- 실시간 동기화

따라서 `collab task` 가 안정적으로 돌면, `solo task` 는 그 안에 포함되는 더 단순한 경우로 볼 수 있다.

## 8. collab task 운영 규칙

### 8-1. 참여 방식

`collab task` 의 참여는 두 경로를 모두 허용한다.

- 직원 직접 `참여`
- 관리자 직접 `배정`

직원 직접 참여는 자리가 있으면 즉시 성공한다. 이번 단계에서 별도 승인 워크플로우는 두지 않는다.

### 8-2. 정원(capacity)

정원은 사람 수 기준으로만 관리한다.

이번 단계에서 하지 않을 것:

- 역할별 슬롯
- 기획 1, 개발 2 같은 역할 쿼터

관리자는 아래를 모두 할 수 있다.

- 생성 시 정원 설정
- 진행 중 정원 증가
- 진행 중 정원 감소

### 8-3. 정원 축소와 초과 인원 처리

정원을 현재 참여 인원보다 작게 줄이는 것도 허용한다.

단, 이 경우 정원 변경은 바로 저장되지 않는다.

같은 흐름 안에서 아래를 함께 끝내야 저장된다.

1. 새 정원 입력
2. 초과 인원 수 계산
3. 관리자가 종료 대상 직접 선택
4. 선택된 인원을 `참여 종료` 처리

시스템이 자동으로 종료 대상을 고르지 않는다.

운영 의미는 `강제 제외` 에 가깝지만, 사용자 화면 문구는 과하게 적대적으로 보이지 않도록 `참여 종료` 로 표시한다.

필요하면 별도 상태를 늘리지 않고, 활동 로그만 남기는 표현도 허용한다.

### 8-4. 자진 종료

참여자는 스스로 `참여 종료` 할 수 있다.

이 규칙이 있어야 실제 협업처럼 참여자 변동을 감당할 수 있다.

## 9. 참여자별 To-Do 모델

### 9-1. 선택한 분담 단위

협업 분담은 `참여자별 To-Do` 로 표현한다.

이번 단계에서 채택하지 않은 안:

- 참여자별 서브태스크
- 참여자별 산출물 슬롯만 따로 관리

### 9-2. 생성 시점

`To-Do` 는 태스크 생성 시 미리 전부 만들지 않는다.

흐름은 아래처럼 간다.

1. `collab task` 생성
2. 참여자 입장 또는 배정
3. 관리자가 참여자별 `To-Do` 생성 및 배정

이 방식이 M1.3 범위를 덜 키우면서도, 실제 협업처럼 보이게 만든다.

### 9-3. 참여 종료 시 To-Do 처리

강제 제외 또는 자진 종료가 일어나면, 그 사람에게 배정된 `To-Do` 는 자동으로 `미배정` 상태로 돌아간다.

이 원칙이 있어야 작업 자체는 남고, 사람만 다시 붙이면 된다.

## 10. 상태 전이

### 10-1. 사람별 참여와 태스크 전체 상태 분리

사람이 참여했다고 바로 태스크 전체가 `in_progress` 가 되지는 않는다.

이번 단계에서는 아래처럼 분리한다.

- 사람별 상태: 참여 여부, 종료 여부
- 태스크 전체 상태: 실제 작업 시작 여부

### 10-2. 진행 시작 규칙

최소 1명이 실제 작업을 시작할 때, 태스크 전체가 `in_progress` 로 들어간다.

이 규칙이 있어야 아래 둘을 구분할 수 있다.

- 사람은 모였지만 아직 시작 안 한 상태
- 실제 작업이 시작된 상태

## 11. 제출 규칙

M1.3에서 협업 태스크는 `참여자별 To-Do` 를 통해 분담되므로, 각 참여자는 자기 몫의 실행 단위에서 제출을 만든다.

이번 단계의 핵심은 `최종 승인 체계`가 아니라, 아래 연결이 살아나는 것이다.

- `Task` 아래 `To-Do` 가 있다
- 각 `To-Do` 가 담당자를 가진다
- 그 실행 끝에 `Deliverable` 이 붙는다

따라서 이번 단계에서는 복잡한 승인 체계보다, `누가 어떤 To-Do를 맡았고 무엇을 제출했는가` 가 먼저 드러나야 한다.

## 12. 실시간 동기화

### 12-1. 이번 단계에서 실시간으로 보여야 하는 것

브라우저에서 실시간으로 보여야 하는 정보는 아래 네 가지다.

1. 태스크 전체 상태 변화
2. 현재 참여 인원 / 정원 변화
3. 참여자 입장 / 참여 종료
4. `To-Do` 배정 해제 또는 재등장

### 12-2. 실시간 검증의 의미

이번 단계에서 WebSocket은 화려한 UI를 증명하는 도구가 아니다.

증명해야 하는 것은 `협업 규칙이 상태 꼬임 없이 양쪽 화면에 동시에 반영된다` 는 사실이다.

## 13. 데모 흐름

M1.3 데모는 아래 한 흐름으로 고정한다.

1. 관리자가 `Project` 아래 `collab task` 를 만든다.
2. 생성 시 최소 1개 이상의 `Deliverable` 을 정의한다.
3. 관리자가 정원을 설정한다.
4. 직원은 직접 참여하거나, 관리자가 직접 배정한다.
5. 참여자가 들어온 뒤 관리자가 참여자별 `To-Do` 를 만든다.
6. 참여자 중 누군가 실제 작업을 시작하면 태스크 전체 상태가 `in_progress` 로 바뀐다.
7. 각 참여자는 자기 `To-Do` 기준으로 제출한다.
8. 관리자가 정원을 줄이면 초과 인원을 직접 선택해 `참여 종료` 처리한다.
9. 빠진 사람의 `To-Do` 는 자동으로 `미배정` 으로 돌아간다.
10. 이 변화가 두 브라우저에 실시간으로 반영된다.

## 14. 구현 전 확인 질문

구현 계획 단계에서 반드시 잠가야 할 항목:

1. `Project` 와 `Task` 연결을 기존 구조 어디에 붙일지
2. `solo task / collab task` 구분 필드를 새로 둘지, 기존 메타데이터로 둘지
3. 참여자별 `To-Do` 의 최소 상태 집합
4. `참여 종료` 를 별도 상태로 저장할지, 활동 로그 표현으로 처리할지
5. 실시간 이벤트를 `task.status_changed` 하나로 묶을지, 참여/정원 전용 이벤트를 나눌지

## 15. 최종 요약

M1.3는 협업 UI 하나를 만드는 단계가 아니다.

M1.3는 아래를 처음으로 성립시키는 단계다.

- `Project` 아래 실제 `Task` 가 움직인다
- `Task` 는 단독/협업이 공존한다
- 협업 태스크는 참여, 정원, 참여 종료를 감당한다
- `To-Do` 와 `Deliverable` 이 연결되어 실행과 결과물이 살아난다
- 이 변화가 실시간으로 보인다

이 다섯 가지가 연결되면, RT2는 `목표판` 에서 한 단계 내려와 `실제 업무 엔진` 으로 들어가기 시작했다고 판단할 수 있다.

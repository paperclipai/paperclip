# RealTycoon2 M1.5 Graphify Project Graph Design

> M1.5 브레인스토밍 결과를 구현 전 설계 문서로 고정한다.
> 범위: Phase 1 / M1.5 "Graphify 그래프"
> 상태: Approved in brainstorming, implementation not started

## 1. 목표

M1.5의 목표는 화려한 전역 그래프 탐색기를 만드는 것이 아니다.

M1.5의 목표는 `M1.4`가 이미 기억한 `일일 위키 + 업무 메타데이터`를 구조화된 그래프로 자동 투영하고, 그 결과를 `GRAPH_REPORT.md` 와 프로젝트 상세의 `Graph` 탭에서 동시에 검증하는 것이다.

이번 단계에서 증명해야 하는 것은 아래 네 가지다.

1. `M1.4`의 일일 위키 페이지와 RT2 업무 메타데이터가 프로젝트 단위 그래프로 자동 구조화된다.
2. 그래프는 `EXTRACTED`, `INFERRED`, `AMBIGUOUS` 신뢰도 태깅을 갖는다.
3. 사용자는 프로젝트 상세의 `Graph` 탭에서 읽기 전용 그래프와 근거를 확인할 수 있다.
4. 같은 내용을 `GRAPH_REPORT.md` 에서 문서형 요약으로 다시 검증할 수 있다.

즉, 이번 단계의 본질은 "기억이 구조가 되었는가"를 증명하는 것이다. 그래프 자체가 새 입력 화면이 되면 안 된다.

## 2. 사용자와 문제

### 직접 사용자

- 본부장님 본인 1명

### 반드시 해소해야 하는 불안

- `M1.4`에서 쌓인 일일 위키와 업무 기록이 단순 문서 누적에서 끝나는가?
- 아니면 실제 관계 구조로 재조립되어 나중에 탐색과 설명의 바닥이 되는가?
- 이 연결이 왜 생겼는지 사용자가 눈으로 납득할 수 있는가?

### 기존 상태

- `M1.4`는 `보드 입력 -> 위키 정리 -> 재질의` 최소 루프까지 증명했다.
- 아직 프로젝트 단위 구조 그래프, 신뢰도 태깅 엣지, `GRAPH_REPORT.md`, `Graph` 탭은 없다.
- Graphify는 스펙에 정의되어 있지만, 현재 제품 데모에서 구조화 결과를 보여주는 읽기 표면은 없다.

## 3. M1.5 완료 기준

### 완료 판정 체크 4개

1. 특정 프로젝트의 `일일 위키 페이지 + 프로젝트/태스크/투두 메타데이터`가 자동으로 그래프 노드/엣지로 투영된다.
2. 그래프 엣지는 최소한 `project -> task`, `task -> todo`, `daily wiki page -> task`, `task -> task dependency` 관계를 포함한다.
3. 프로젝트 상세의 `Graph` 탭에서 읽기 전용 그래프와 선택 노드/엣지의 근거를 볼 수 있다.
4. 같은 프로젝트의 `GRAPH_REPORT.md` 에서 노드 수, 엣지 수, 신뢰도 집계, 중심 태스크, ambiguous 관계, 마지막 갱신 시각을 확인할 수 있다.

## 4. 명시적 제외 범위

M1.5에서는 아래 항목을 절대 넣지 않는다.

1. 코드 AST 분석
2. 전사 OKR 전체 그래프
3. 그래프 수동 편집
4. God Node, Surprising Connection 전면 노출
5. 범용 전역 그래프 탐색기
6. 수동 `그래프 다시 만들기` 버튼 중심 UX

이 항목들은 Graphify 전체 비전에는 속하지만, 이번 단계의 핵심 검증인 `기억 -> 구조화 -> 읽기 검증` 루프보다 범위를 크게 키운다.

## 5. 설계 방향

### 선택된 접근

- 범위 전략: `Projection-first`
- 입력 전략: `일일 위키 페이지 + 프로젝트/태스크/투두 메타데이터`
- 표면 전략: `Project Detail > Graph 탭 + GRAPH_REPORT.md`
- 갱신 전략: `프로젝트 단위 자동 증분 갱신`
- 관계 전략: `핵심 4종 관계부터`

### 의미

M1.5는 새 그래프 제품을 따로 만드는 단계가 아니다.

M1.5는 `M1.4`가 만든 기억 계층 뒤에 `구조화 projection` 을 하나 더 붙이는 단계다.

즉 흐름은 아래처럼 고정한다.

`Daily 보드/업무 메타데이터 변경`
-> `M1.4 activity log / daily wiki materialization`
-> `M1.5 graph projector`
-> `graph_nodes / graph_edges / graph_cache / graph_communities`
-> `GRAPH_REPORT.md + Project Detail > Graph 탭`

이 구조를 택하는 이유는 세 가지다.

1. `SPEC/06` 의 Graphify 소스 수정 금지, direct wiki write 금지 원칙을 안전하게 지킬 수 있다.
2. `M1.4`의 기억 계층을 그대로 `M1.5`의 구조 계층으로 확장할 수 있다.
3. 이후 `M1.6`, `M1.7`, `M2.x` 로 갈 때 projector의 입력과 출력만 넓히면 된다.

## 6. 원본과 projection 경계

이번 단계에서 그래프는 원본 저장소가 아니다.

원본 truth 는 여전히 두 곳에 있다.

1. `M1.4`의 일일 위키 projection
2. RT2 업무 메타데이터 (`project`, `task`, `todo`, `task dependency`)

그래프는 이 truth 를 읽어 만든 `읽기 최적화 구조물`이다.

이 경계를 지켜야 다음이 가능하다.

- 프로젝트 단위 재빌드
- 잘못된 관계 재계산
- 신뢰도 규칙 재적용
- 향후 입력 소스 확장

즉 `graph_nodes` 와 `graph_edges` 가 업무 truth 를 대체하면 안 된다. 언제든 원본에서 다시 만들어질 수 있어야 한다.

## 7. 데이터 모델

### 7-1. 중심 저장 구조

이번 단계에서 중심이 되는 저장 구조는 네 묶음이다.

#### `graph_nodes`

이번 단계에서 다루는 노드 종류는 아래 네 가지다.

- `project`
- `task`
- `todo`
- `daily_wiki_page`

이 노드들은 화면의 `Graph` 탭과 `GRAPH_REPORT.md` 가 함께 읽는 공통 단위다.

#### `graph_edges`

이번 단계에서 허용하는 관계는 아래 네 종류다.

1. `project -> task`
2. `task -> todo`
3. `daily wiki page -> task`
4. `task -> task dependency`

엣지는 반드시 `EXTRACTED`, `INFERRED`, `AMBIGUOUS` 중 하나의 신뢰도 값을 가져야 한다.

#### `graph_cache`

`graph_cache` 는 프로젝트 단위 증분 갱신의 기준점이다.

어떤 프로젝트의 어떤 입력 묶음이 마지막으로 어떤 해시로 처리됐는지 기록하고, 변경된 범위만 다시 계산하게 만든다.

#### `graph_communities`

이번 단계에서 `graph_communities` 는 완성형 Leiden 시각화보다, 최소한의 묶음 요약과 `GRAPH_REPORT.md` 기초 정보를 위한 구조로 쓴다.

UI 전면 노출은 다음 단계로 미룬다.

### 7-2. 신뢰도 규칙

신뢰도는 이번 단계의 핵심이다. 관계마다 아래 규칙을 고정한다.

- `project -> task`
  - 시스템 메타데이터에서 직접 나오므로 기본 `EXTRACTED`
- `task -> todo`
  - 부모-자식 메타데이터에서 직접 나오므로 기본 `EXTRACTED`
- `task -> task dependency`
  - 기존 dependency 메타데이터에서 직접 나오면 `EXTRACTED`
- `daily wiki page -> task`
  - 위키가 특정 task 를 직접 참조하거나 명시적으로 매칭되면 `EXTRACTED`
  - 위키 서술과 task 메타데이터를 조합해 연결한 경우 `INFERRED`
  - 후보 task 가 2개 이상이라 확정이 어려우면 `AMBIGUOUS`

`INFERRED` 관계에는 왜 그런 추론이 나왔는지 근거 문장을 함께 남겨야 한다.

`AMBIGUOUS` 관계는 숨기지 않는다. `Graph` 탭과 `GRAPH_REPORT.md` 모두에서 검토 필요 상태로 드러내야 한다.

## 8. 입력 원천

이번 단계의 projector 가 읽는 입력은 아래 범위로 제한한다.

- 프로젝트 메타데이터
- 프로젝트 아래 태스크
- 태스크 아래 투두
- 태스크 의존성
- 해당 프로젝트의 일일 위키 페이지
- 해당 프로젝트의 관련 활동 로그 요약

이번 단계에서 일부러 제외하는 입력은 아래와 같다.

- 코드 AST
- 전사 문서 전체
- cross-project 관계 전체
- Obsidian 외부 vault 동기화 데이터

이 제한이 있어야 첫 데모가 `프로젝트 1개` 기준으로 닫힌다.

## 9. 처리 흐름과 자동 증분 갱신

이번 단계의 핵심은 "언제 다시 계산하느냐" 이다.

처리 흐름은 아래처럼 고정한다.

1. `M1.4` 쪽 일일 위키, 활동 로그, 또는 관련 업무 메타데이터가 바뀐다.
2. 해당 변경이 `graph projector` 의 입력 후보로 들어간다.
3. projector 는 `프로젝트 ID` 기준으로 관련 입력 묶음을 다시 읽는다.
4. 입력 묶음의 해시를 계산해 `graph_cache` 와 비교한다.
5. 해시가 같으면 재계산을 건너뛴다.
6. 해시가 다르면 그 프로젝트 범위의 `graph_nodes`, `graph_edges`, `graph_communities`, `GRAPH_REPORT.md` projection 을 다시 만든다.
7. UI 의 `Graph` 탭은 최신 projection 결과를 읽기 전용으로 다시 가져온다.

이 구조의 장점은 아래와 같다.

- 전체 그래프를 매번 갈아엎지 않는다.
- `M1.4` 루프와 자연스럽게 이어진다.
- 프로젝트 단위 검증이 쉬워진다.
- 이후 범위 확장 시 재사용성이 높다.

즉 사용자 경험은 "업무를 바꾸면 잠시 뒤 `Graph` 탭과 `GRAPH_REPORT.md` 가 최신 구조를 반영한다" 수준으로 맞춘다.

이번 단계에서 필요한 건 자동성의 증명이지, 분산 그래프 엔진의 완성형이 아니다.

## 10. 사용자 표면

이번 단계에서 사용자가 확인하는 산출물은 두 개다.

1. 프로젝트 상세 안의 `Graph` 탭
2. 같은 내용을 문서형으로 푼 `GRAPH_REPORT.md`

둘은 경쟁하는 기능이 아니라, 같은 projection 을 다른 해상도로 보여주는 쌍이다.

### 10-1. Project Detail > Graph 탭

`Graph` 탭은 읽기 전용으로 간다.

첫 데모에서 보여줄 것은 아래 정도로 고정한다.

- 프로젝트 중심 그래프 1개
- 노드 종류 구분: `project`, `task`, `todo`, `daily wiki page`
- 엣지 종류 구분: `project -> task`, `task -> todo`, `daily wiki page -> task`, `task -> task dependency`
- 신뢰도 표기: `EXTRACTED`, `INFERRED`, `AMBIGUOUS`
- 선택 노드/엣지의 근거 요약 패널

이번 단계의 UI 핵심은 "왜 이 연결이 생겼는지 납득시키는 것"이다.

즉 그래프는 예쁘게만 보여서는 안 되고, 근거를 읽을 수 있어야 한다.

예시 근거 수준은 아래와 같다.

- `EXTRACTED`: 태스크 ID, 부모 관계, 위키 직접 링크 등 명시 근거
- `INFERRED`: 위키 문장과 태스크 메타데이터 조합에 기반한 추론 근거
- `AMBIGUOUS`: 후보가 둘 이상이라 검토가 필요하다는 설명

### 10-2. GRAPH_REPORT.md

`GRAPH_REPORT.md` 는 화면보다 설명이 더 많은 운영 문서다.

이번 단계의 리포트에는 최소한 아래가 들어간다.

1. 현재 프로젝트의 노드 수
2. 현재 프로젝트의 엣지 수
3. 신뢰도별 엣지 집계
4. 가장 중심에 있는 태스크 몇 개
5. 오늘의 일일 위키 페이지가 어떤 태스크들과 연결됐는지
6. `AMBIGUOUS` 관계 목록
7. 마지막 갱신 시각
8. 이번 projection 이 사용한 입력 범위

즉 `Graph` 탭은 "한눈에 보기", `GRAPH_REPORT.md` 는 "설명과 감사용 보기" 를 맡는다.

## 11. UI/UX 원칙

이번 단계의 UI 는 새 세계를 열지 않는다. `M1.4`와 나란히 붙어야 한다.

따라서 화면 위치는 아래처럼 고정한다.

- `Project Detail`
  - `Daily`
  - `Wiki`
  - `Graph`

이렇게 해야 사용자가 같은 프로젝트 안에서 `기억`, `재질의`, `구조화` 가 한 이야기라는 감각을 바로 얻는다.

이번 단계에서 일부러 넣지 않는 UI 는 아래와 같다.

- 노드 수동 편집
- 엣지 수동 추가/삭제
- 전역 검색형 그래프 탐색기
- OKR 전체 그래프로의 즉시 확대
- God Node, Surprising Connection 전용 화려한 패널

첫 데모는 "프로젝트 하나를 열면 Daily, Wiki 옆에서 Graph 도 같은 이야기의 구조 버전을 보여준다"면 충분하다.

## 12. 검증 시나리오

M1.5 데모는 아래 흐름으로 고정한다.

1. 프로젝트 상세에서 `Daily` 또는 기존 업무 흐름으로 task/todo/일일 위키 관련 입력을 바꾼다.
2. 시스템이 해당 프로젝트 범위의 graph projection 을 자동 증분 갱신한다.
3. `Graph` 탭에서 프로젝트 중심 구조와 신뢰도 태깅을 확인한다.
4. 특정 `daily wiki page -> task` 연결을 눌러 근거를 확인한다.
5. `GRAPH_REPORT.md` 에서 같은 프로젝트의 노드/엣지 요약과 ambiguous 관계를 다시 확인한다.

즉, 이번 단계는 `문서와 화면이 같은 구조를 가리키는가` 까지 봐야 통과다.

## 13. 리스크와 대응

### 리스크 1. 그래프가 원본 truth 처럼 오해됨

대응:

- 원본은 계속 위키와 업무 메타데이터라고 명시한다.
- 그래프는 projection 이며 재생성 가능 구조임을 코드와 문서에서 분리한다.

### 리스크 2. 자동 갱신 범위가 너무 넓어짐

대응:

- 프로젝트 ID 기준으로만 입력 집합을 묶는다.
- `graph_cache` 로 범위를 좁혀 전체 재계산을 피한다.

### 리스크 3. 신뢰도 표기가 형식적으로만 남음

대응:

- `INFERRED` 는 반드시 근거 문장을 남긴다.
- `AMBIGUOUS` 는 숨기지 말고 검토 필요 상태로 표출한다.

### 리스크 4. 화면이 먼저 커지고 구조가 비어 있음

대응:

- `Projection-first` 원칙을 유지한다.
- UI 는 projection 결과를 읽는 표면으로만 시작한다.

### 리스크 5. M1.5 가 전역 Graphify 구현으로 범위 폭발

대응:

- 입력 원천을 `일일 위키 페이지 + 프로젝트/태스크/투두 메타데이터` 로 제한한다.

## 15. 최종 요약

M1.5는 Graphify를 전면 제품화하는 단계가 아니다.

M1.5는 아래를 처음으로 성립시키는 단계다.

- `M1.4`가 기억한 일일 위키와 업무 메타데이터를 읽는다
- 그 기억을 프로젝트 단위 그래프로 자동 구조화한다
- 관계마다 신뢰도와 근거를 붙인다
- 같은 구조를 `Graph` 탭과 `GRAPH_REPORT.md` 에서 함께 검증한다

이 네 가지가 연결되면, RT2의 `기억이 구조가 되고, 구조가 다시 탐색 가능한 자산이 된다` 는 약속이 처음으로 제품 안에서 눈에 보이기 시작했다고 판단할 수 있다.

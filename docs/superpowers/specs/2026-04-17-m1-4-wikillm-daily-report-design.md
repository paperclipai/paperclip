# RealTycoon2 M1.4 wikiLLM Daily Report Design

> M1.4 브레인스토밍 결과를 구현 전 설계 문서로 고정한다.
> 범위: Phase 1 / M1.4 "wikiLLM 위키"
> 상태: Approved in brainstorming, implementation not started

## 1. 목표

M1.4의 목표는 새 위키 화면을 만드는 것이 아니다.

M1.4의 목표는 `일일업무보고서 3칸 칸반보드`에서 일어난 일을 시스템이 기억하고, 하루 단위로 정리하며, 나중에 다시 물었을 때 근거와 함께 답할 수 있음을 증명하는 것이다.

이 단계에서 증명해야 하는 것은 아래 세 가지다.

1. 사용자가 일일업무보고서 보드에서 타일 또는 To-Do 를 만들고 갱신할 수 있다.
2. 그 변화가 위키의 `index/log/일일 페이지`로 자동 정리된다.
3. 사용자가 `오늘 뭐 했지?` 라고 다시 물었을 때 위키를 바탕으로 `짧은 요약 + 세부 히스토리`를 돌려받을 수 있다.

M1.4는 위키가 업무 입력을 대체하는 단계가 아니다. 입력은 끝까지 칸반에서 하고, 위키는 그 뒤에서 `기억`과 `회고`를 맡는 단계다.

## 2. 사용자와 문제

### 직접 사용자

- 본부장님 본인 1명

### 반드시 해소해야 하는 불안

- 오늘 할 일 칸반에서 내가 만들고 옮기고 갱신한 To-Do 흐름을 시스템이 잊지 않고 기억하는가?
- 나중에 다시 물으면 요약만이 아니라 근거까지 보여줄 수 있는가?

### 기존 상태

- M1.3까지는 조직/OKR/태스크 엔진 뼈대까지만 검증했다.
- 일일업무보고서 칸반의 활동을 위키 기억 계층으로 누적하는 루프는 아직 없다.

## 3. M1.4 완료 기준

### 완료 판정 체크 3개

1. 일일업무보고서 3칸 보드에서 타일 1건을 만들거나 갱신할 수 있다.
2. 그 변경을 바탕으로 위키 `일일 페이지`, `index`, `log` 가 자동으로 정리된다.
3. `오늘 뭐 했지?` 질문에 시스템이 위키를 바탕으로 `짧은 요약 + 세부 히스토리` 형식으로 답한다.

## 4. 명시적 제외 범위

M1.4에서는 아래 항목을 절대 넣지 않는다.

1. Graphify 그래프 시각화
2. 금화/보상 계산
3. 자비스 자동 추천
4. 일일업무보고서 전체 완성형 UX
5. 타일별 독립 위키 페이지 대량 생성

이 다섯 항목은 M1.4의 핵심 검증과 무관하거나 범위를 크게 키운다.

## 5. 일일업무보고서 해석

### 5-1. 일일업무보고서는 문서가 아니라 보드다

이번 단계에서 일일업무보고서는 별도 보고서 문서를 쓰는 화면이 아니다.

일일업무보고서는 `오늘 할 일` 본판과 `보조창 2개`로 구성된 3칸 칸반보드다. 사용자는 이 보드 안에서 오늘 자신이 관여한 태스크의 타일과 To-Do 를 생성, 이동, 갱신한다.

### 5-2. 3칸 구조

- 1번 창: `오늘 할 일`
- 2번 창: 보조창
- 3번 창: 보조창

`오늘 할 일`은 고정 메인 영역이다. 새 타일은 기본적으로 여기 쌓인다.

보조창 2개는 오늘 할 일의 보조 분류 영역이다. 여기서는 `내일 할 일`, `아이디어`, `미룬일`, `진행중` 같은 탭을 만들어 타일을 분류한다.

### 5-3. 본판과 보조창의 관계

중요한 것은 칸 수가 아니라 역할이다.

- `오늘 할 일`은 실행의 본판이다.
- `보조창`은 보류, 아이디어, 내일 할 일 같은 정리와 분류를 맡는다.

따라서 위키는 `칸반을 대체하는 보고서`가 아니라, `칸반에서 일어난 변화를 기억하는 계층`이어야 한다.

## 6. 설계 방향

### 선택된 접근

- 보드 전략: `3칸 칸반 유지`
- 기억 전략: `활동 로그 중심형`
- 요약 전략: `일일 위키 페이지`
- 확인 전략: `보드 입력 + 위키 확인 분리`
- 답변 전략: `짧은 요약 + 세부 히스토리`

### 의미

M1.4의 본체는 여전히 일일업무보고서 칸반이다.

위키는 두 가지 역할만 한다.

1. 기억
2. 요약

사용자는 보드에서 일하고, 시스템은 그 움직임을 기억하고, 나중에 질문이 들어오면 그 기억을 읽어서 답한다. 이 구조가 `칸반이 심장`이라는 프로젝트 원칙을 보존한다.

## 7. 기록 모델

### 7-1. 활동 로그 중심형

이번 단계에서는 `하루 스냅샷 중심형`이나 `타일별 위키 페이지 중심형`이 아니라 `활동 로그 중심형`으로 간다.

즉, 사용자가 보드에서 무언가를 할 때마다 시스템은 그 변화를 활동 이벤트로 남긴다.

대표 예시는 아래와 같다.

- `todo.created`
- `todo.updated`
- `todo.moved`
- `todo.progress_updated`
- `todo.completed`
- `todo.note_updated`

### 7-2. 왜 로그 중심형인가

일일업무보고서의 핵심은 최종 결과만이 아니라 `오늘 어떤 흐름으로 일했는가`다.

스냅샷만 저장하면 왜 그런 상태가 되었는지 설명하기 어렵다. 반면 활동 로그를 남기면 `무엇을 만들었고`, `어디로 옮겼고`, `어떻게 진행률이 바뀌었는지`를 회고할 수 있다.

### 7-3. 이벤트 적재 원칙

직접 `index.md` 나 `log.md` 를 쓰면 안 된다.

따라서 보드의 모든 변화는 먼저 `event_stream` 류의 활동 이벤트로 적재되고, 그 뒤에 위키 정리기가 읽어서 `index/log/일일 페이지`를 갱신해야 한다.

이 원칙은 `SPEC/06` 의 직접 쓰기 금지와 `SPEC/01` CQRS 구조를 동시에 만족한다.

## 8. 위키 산출물

### 8-1. 일일 페이지

M1.4의 핵심 위키 산출물은 `하루 1페이지`다.

예시:

- `wiki/daily/2026-04-17-홍길동.md`

이 페이지는 그날의 활동 로그를 읽기 쉬운 형태로 정리한 결과다.

필수 섹션:

1. 오늘 한 일 요약
2. 완료한 항목
3. 진행중 항목
4. 내일 할 일
5. 미룬 일
6. 아이디어
7. 세부 활동 히스토리

### 8-2. log

`log` 는 연대기 성격의 기록이다.

여기에는 `몇 시에 무엇을 만들었고`, `어느 보조창으로 옮겼고`, `진행률이 어떻게 바뀌었는지` 같은 사실 기반의 활동 흔적이 쌓여야 한다.

### 8-3. index

`index` 는 일일 페이지와 관련 주제에 접근하기 위한 카탈로그 역할을 한다.

이번 단계에서는 복잡한 주제 분류보다 아래 정도면 충분하다.

- 날짜별 일일 페이지 링크
- 사용자별 최근 일일 페이지 링크
- 최근 갱신 일자

## 9. 질의응답 모델

### 9-1. 질문 형태

M1.4가 풀어야 할 대표 질문은 아래 하나로 고정한다.

- `오늘 뭐 했지?`

### 9-2. 답변 형식

답변은 두 층으로 구성한다.

1. 짧은 일일 요약
2. 세부 활동 히스토리

짧은 요약은 3줄 안팎으로 사용자가 오늘의 흐름을 바로 이해하게 해야 한다.

세부 히스토리는 그 요약의 근거다. 예를 들어 아래 같은 식이다.

- 오전 09:10 `주간 보고서 작성` 타일 생성
- 오전 10:05 `오늘 할 일 -> 진행중` 이동
- 오후 14:40 진행률 30% -> 70%
- 오후 17:20 완료 처리

### 9-3. 위키 기반 답변 증명

사용자가 위키를 읽고 답했다는 사실을 납득하려면, 답변 아래에 근거 항목을 바로 이어서 보여주는 것이 좋다.

즉 `답변만` 보여주는 것이 아니라 `답변 + 근거 히스토리`를 같이 보여줘야 한다.

## 10. 화면 구조

### 10-1. 메인 입력 화면

업무 입력은 항상 `일일업무보고서 3칸 보드`에서 끝나야 한다.

보드에서 가능한 핵심 동작:

- 타일 생성
- 타일 이동
- 타일 편집
- 진행률 갱신
- 완료 체크

저장 직후에는 긴 설명보다 아래 수준의 짧은 피드백이면 충분하다.

- `기억됨`
- `위키 반영 예정`
- `오늘 기록에 추가됨`

### 10-2. 위키 확인 화면

위키 확인 화면은 입력 화면이 아니라 확인 화면이다.

여기서 보여줄 것:

- 오늘 날짜 기준 일일 위키 페이지
- 상단의 짧은 요약
- 하단의 활동 히스토리
- `오늘 뭐 했지?` 질문에 대한 위키 기반 답변

### 10-3. 데모 흐름

M1.4 데모는 아래 한 흐름으로 고정한다.

1. 보드에서 타일 1건 생성 또는 갱신
2. 보조창으로 1회 이동 또는 상태 변경
3. 시스템이 기억 완료 피드백 표시
4. 위키 확인 화면에서 오늘 요약과 활동 히스토리 확인
5. `오늘 뭐 했지?` 질문에 대한 위키 기반 답변 확인

## 11. 자비스 역할

이번 단계에서 자비스는 주인공이 아니다.

하지만 아래 두 역할은 허용된다.

1. 사람이 만든 보드 활동을 정리하고 요약하는 보조자
2. 필요 시 타일을 등록해주는 보조 작성자

즉 `사람 또는 자비스가 오늘 할 일 등록`은 가능하지만, 설계의 중심은 여전히 `칸반 활동 -> 기억 -> 재질의` 루프다.

## 12. 파일 경계 원칙

이번 단계는 UI와 위키 로직이 섞이기 쉬우므로 경계를 분명히 해야 한다.

권장 분리:

- 보드 UI 상태
- 보드 활동 이벤트 생성
- 위키 정리기
- 질의응답용 위키 리더
- 위키 확인 화면

한 파일이 보드 UI, 이벤트 변환, 위키 요약을 다 맡으면 이후 M1.5 이상에서 바로 무너진다.

## 13. 리스크와 대응

### 리스크 1. 위키가 입력 화면으로 역전됨

대응:

- 입력은 끝까지 칸반에서만 한다.
- 위키는 확인과 회고만 담당한다.

### 리스크 2. 활동 로그가 부족해 답변 근거가 약해짐

대응:

- 최소 이벤트 집합을 먼저 고정한다.
- 생성, 이동, 진행률 변경, 완료는 반드시 이벤트로 남긴다.

### 리스크 3. 로그는 많지만 읽기 어려움

대응:

- 하루 마감 시 AI가 일일 페이지를 읽기 쉬운 형태로 요약한다.
- 질문 응답은 항상 `짧은 요약 -> 세부 히스토리` 순서를 유지한다.

### 리스크 4. 보조창 의미가 흐려짐

대응:

- 보조창은 `내일 할 일 / 아이디어 / 미룬일 / 진행중` 같은 분류용 탭 영역으로 해석을 고정한다.
- 본판은 언제나 `오늘 할 일`이다.

## 14. 구현 전 확인 질문

구현 계획 단계에서 반드시 잠가야 할 항목:

1. 활동 이벤트의 최소 종류와 페이로드
2. 일일 위키 페이지 파일 경로와 frontmatter 구조
3. `오늘 뭐 했지?` 질의에 사용할 읽기 경로
4. 보드에서 위키 반영 상태를 어떻게 피드백할지
5. 하루 마감 요약 생성 시점과 트리거

## 15. 최종 요약

M1.4는 위키를 앞에 세우는 단계가 아니다.

M1.4는 아래를 처음으로 성립시키는 단계다.

- 일일업무보고서 3칸 칸반보드에서 일한다
- 시스템이 그 활동을 로그로 기억한다
- 하루 단위 위키 페이지로 정리한다
- 나중에 다시 물으면 위키를 읽고 답한다

이 네 가지가 연결되면, RT2의 `일일업무보고서가 심장이고 위키는 기억 계층` 이라는 구조가 실제 제품으로 서기 시작했다고 판단할 수 있다.

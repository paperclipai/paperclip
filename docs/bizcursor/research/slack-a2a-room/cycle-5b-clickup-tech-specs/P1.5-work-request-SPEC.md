## 1. Contexto

### 1.1 Por que P1.5 existe

Ciclo 1B (ClickUp Hybrid): humanos pedem trabalho à IA com a mesma facilidade de assign no ClickUp/Linear — não só digitando `@` no chat.

Stack de referência (D-12):

1. `@mention` na Room (já P1)  
2. **Assign-as-delegate** — humano = **owner**; agente = **delegate**  
3. Botão **Ask / Pedir ao agente**  
4. **Templates** de pedido (triage, research, draft, review)

Sem P1.5, só power-users usam a Room; o beachhead Software House precisa que **qualquer member** (PM, CS, junior) dispare trabalho à IA sem decorar slugs.

### 1.2 Problema

| Hoje no fork | Gap |
|--------------|-----|
| `@CEO` no BoardChat (P1) | Exige saber o slug e estar na Room |
| Issue assign → `issue-assignment-wakeup` | Assign clássico trata assignee como executor; não separa owner humano vs delegate agente |
| `run-delegation` | Agent-JWT / host agent; humano não “pede” via UI dedicada |
| Agents list | Descoberta, não intake |

### 1.3 Escopo

| Dentro P1.5 | Fora P1.5 |
|-------------|-----------|
| Botão Ask em Issue / Room / Team Panel | Fan-out N agentes (P2) |
| Modal Work Request + templates | Ambient Autopilot na Room (P5.5 proíbe) |
| Assign-as-delegate (owner user + delegate agent) | Novo ledger de custo (P4) |
| Qualquer **company member** autenticado pode pedir | Board JWT fingindo agent |
| Deep-link para thread da Room após wake | BizCursor desktop |

### 1.4 Cenário canônico

```
Sofia (operator) na issue ENG-442:
  1. Clica [Pedir ao agente]
  2. Escolhe template "Triage bug" + agente @ceo
  3. Confirma → issue.ownerUserId = Sofia; issue.delegateAgentId = CEO
  4. room-orchestrator / assignment-wakeup cria host run (P1 path)
  5. Reply do CEO no thread; Sofia permanece owner humano visível
```

### 1.5 Premissas

- P1 `room-orchestrator` (ou equivalente wakeup `conference_room_mentioned` / assignment) disponível.
- Agentes invokable via adapters `opencode_local` / `cursor_cloud`.
- Company members via `company-member-roles` / CompanyAccess.
- Feature flag Room ou flag dedicada `enableWorkRequestV1` (default off em prod Coolify).

### 1.6 Glossário

| Termo | Definição |
|-------|-----------|
| **Work Request** | Pedido estruturado humano→agente (template + prompt + target agent) |
| **Owner** | User humano responsável pelo outcome (sempre visível) |
| **Delegate** | Agente que executa o wake/run |
| **Ask button** | CTA “Pedir ao agente” / “Ask agent” |
| **Template** | Prompt + metadata pré-preenchidos (id estável) |

## 6. UX

### 6.1 Densidade

| Persona | Experiência |
|---------|-------------|
| **Sofia (Operator)** | Botão claro “Pedir ao agente”; 2 campos (agente + o que precisa); template chips; owner chip sempre visível |
| **Board** | Mesmo fluxo + link runId / workRequestId no toast expandido |
| **Humano não-técnico** | Templates com títulos humanos (“Triagem de bug”, “Resumo de status”) — sem jargão A2A |

### 6.2 Fluxo Ask (Issue)

1. IssueDetail → **Pedir ao agente**  
2. Modal: chips de template → select agente → textarea  
3. Confirmar → loading “Acordando @ceo…”  
4. Toast sucesso + scroll ao comment do pedido  
5. Bubble/reply do agente quando run completa (P1)

### 6.3 Fluxo Ask (Room)

1. BoardChat → **Pedir** (ou equivalente no composer toolbar)  
2. Mesmo modal; submit posta mention formatada + abre orchestrator  
3. Silent-until-@ preservado para mensagens **sem** Ask/`@`

### 6.4 Erros

| Código | UX |
|--------|-----|
| `AGENT_NOT_INVOKABLE` | “Este agente está pausado ou sem adapter.” |
| `NOT_ASSIGNABLE` | “Agente não pode ser delegate nesta company.” |
| `FANOUT_USE_ROOM` | “Para vários agentes, use a Room com @A @B.” |
| `BUDGET_EXCEEDED` | “Budget esgotado — peça ao Board.” |
| `POLICY_BLOCKED` | “Política de proatividade bloqueia este trigger.” (P5.5) |

### 6.5 A11y

- Dialog: `role="dialog"`, focus trap, Escape fecha.  
- Template chips: `aria-pressed`.  
- Owner/delegate chips: texto não só cor.

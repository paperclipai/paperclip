## 1. Contexto

### 1.1 Por que P5.5 existe

Ciclo 1B: proatividade = schedule / event / threshold / ambient — e **ambient fora da Room**. ClickUp Autopilot é proativo por trigger; a Room Paperclip permanece **silent-until-@** (D-07, D-10).

Sem policy explícita:

- Routines podem “parecer” que a sala fala sozinha.  
- Webhooks podem postar e acordar agentes sem owner.  
- Marketing pode claimar “Autopilot na sala” (agent washing).

**P5.5** entrega:

1. `ProactivityPolicyEditor` (Board)  
2. **Whitelist** de trigger kinds permitidos  
3. Regra normativa: **Room = silent-until-@** (nenhum ambient no chat)  
4. Routines / webhooks / Autopilot-like **fora** do stream da Room (com deep-links)

### 1.2 Problema

| Primitive hoje | Risco sem policy |
|----------------|------------------|
| `routines.ts` + UI Routines | Schedule acorda agente; se postar na Board Operations issue da Room → spam |
| `cron.ts` | Jobs opacos |
| `cursor-webhook-ingest.ts` | Eventos externos |
| BoardChat P0 | Silent ok — mas outros paths podem furar |
| Instance settings | Sem modelo “proactivity matrix” |

### 1.3 Escopo

| Dentro P5.5 | Fora P5.5 |
|-------------|-----------|
| Schema + API + Editor de policy company-scoped | Novo motor de Autopilot completo |
| Enforce Room silent (gate server) | Ambient agents no chat |
| Whitelist triggers + audit | ML “quando ser proativo” |
| Deep-link Routines/Webhooks como superfície proativa | Substituir Routines CRUD |
| Docs Sofia: “o que pode acordar o quê” | BizCursor |

### 1.4 Cenário canônico

```
Board abre Company Settings → Proactivity:
  ✅ schedule (routines)
  ✅ webhook_event
  ✅ threshold_budget (alert only)
  ✅ work_request / mention (human-gated)
  ❌ room_ambient_message
  ❌ room_auto_reply_without_mention

Sofia na Room posta sem @: 0 wakes (P0 + P5.5 enforce).
Routine diária 9h: CEO roda FORA do chat; opcionalmente posta resumo em issue
  NÃO-Room ou com mention explícita se policy allowPostToRoomWithMention=true.
```

### 1.5 Premissas

- P0 silent path já implementado no BoardChat.  
- Routines/webhooks já disparam wakes por outros canais.  
- Feature flag `enableProactivityPolicyV1` (default: policy implícita = deny ambient room mesmo se flag off — **fail-safe**).

### 1.6 Glossário

| Termo | Definição |
|-------|-----------|
| **Trigger kind** | Tipo de evento que pode iniciar trabalho de agente |
| **Whitelist** | Conjunto allow de kinds; default deny para kinds desconhecidos |
| **Room ambient** | Agente responde/posta na Conference Room sem `@` / Ask / mention explícita |
| **Outside room** | Routines, webhooks, heartbeats, issues não-Room |
| **Policy** | Documento versionado company (JSON) + enforce server-side |

### 7.3 Schema (normativo)

```ts
const TriggerKindSchema = z.enum([
  "mention",
  "work_request",
  "issue_assignment",
  "routine_schedule",
  "webhook_event",
  "threshold_alert",
  "manual_wakeup",
  "delegation_child",
]);

const ProactivityPolicySchema = z.object({
  version: z.number().int().positive(),
  triggerWhitelist: z.array(TriggerKindSchema),
  room: z.object({
    postWithoutMentionWakesAgent: z.literal(false),
    allowRoutineSummaryPost: z.enum([
      "never",
      "as_human_owner_comment",
      "with_explicit_mention_only",
    ]),
  }),
  thresholdAlertAutoRun: z.boolean().default(false),
});
```

Exhaustive switch ao mapear `wakeupReason → TriggerKind` com `never` check (regra typescript-exhaustive-switch).

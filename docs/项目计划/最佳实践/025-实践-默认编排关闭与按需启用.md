# 实践：默认编排关闭与按需启用（Routic 实例）

**定位：** 说明 **为何** routic fork 不沿用 upstream 开箱默认编排，以及 **如何** 把实例收成「人派活 → agent 干 → 汇报；干不了 blocked；卡住人工看运行清单」，再 **按痛点单条** 开自动化。

**不是：** 说 upstream「搞错了」——是他们的产品在 **作者环境跑顺之后** 打包的复检层；我们没有同一套 agent 配置、强模型二线运维与英文默认可读的控制面。

## 1. 两套前提（为何默认编排会「爆」）

| | upstream 默认 | routic（本 fork / 4100） |
| --- | --- | --- |
| 操作者 | 熟悉 recovery / handoff / monitor 英文语义 | 中文 Windows UTF-8，Board 须可读 |
| Agent 池 | 作者调过的 CEO/CTO/工程师组合 | 自配角色 + codebuddy / qwen / cursor |
| 模型 | 强模型可兜「再跑一轮交接」 | 无「修模型的模型」→ 报错链 + 衍伸单 |
| 目标 | 平台演示：自愈、复盘、图 liveness | **先简单编排跑顺**，再按需开自动化 |

**路线图含义：** [路线图](../编排/Paperclip%20为中心的路线图：从今天到一人公司.md) 里的「阶段 1」假设 **平面已可控**。在 routic 上要先做 **阶段 0**：关默认编排、汉化、唤起档位、适配器跑稳——否则 CEO/CTO 只会触发 recovery 链，不是蜂群。

---

## 2. 目标心智模型（当前默认）

```
人派活（指派 / 手动唤醒 / 有意识的评论 wake）
    → agent checkout → run → 事务评论摘要 + done/blocked
    → 干不了：blocked（不要自动造修复子单）
    → run 卡住：运行清单人工看（不自动复盘单 / 不自动再 wake 三轮）
```

**禁止默认发生：** 滞留回收子单、生产力复盘单、成功 run 交接再 wake、issue monitor 造子单、定时 CEO/CTO 扫 inbox。

## 3. 关闭清单（4100 / `paperclip-routic`）

### 3.1 环境变量（改后 **须重启 API**）

写入 **仓库根 `.env`**（`PORT=4100`）并同步 **`%USERPROFILE%\.paperclip\instances\default\.env`**（实例加载链见 [001-运维-回形针本地](001-运维-回形针本地.md)）。

```env
PAPERCLIP_STRANDED_ISSUE_RECOVERY_ENABLED=false
PAPERCLIP_ORPHAN_RUN_REAP_ENABLED=false
PAPERCLIP_ISSUE_GRAPH_LIVENESS_ENABLED=false
PAPERCLIP_SILENT_ACTIVE_RUN_WATCHDOG_ENABLED=false
PAPERCLIP_PRODUCTIVITY_REVIEWS_ENABLED=false
PAPERCLIP_SCHEDULED_RETRY_PROMOTION_ENABLED=false
```

| 变量 | 关闭后 |
| --- | --- |
| `STRANDED_ISSUE_RECOVERY` | 滞留回收子单；**含** successful-run handoff、monitor recovery（代码 `e1d7035f`） |
| `PRODUCTIVITY_REVIEWS` | 生产力复盘衍伸单 |
| `ISSUE_GRAPH_LIVENESS` | 事务图 liveness 自动恢复 |
| `SILENT_ACTIVE_RUN_WATCHDOG` | 静默 run 自动评估单 |
| `ORPHAN_RUN_REAP` | 启动/周期孤儿 run 收割 |
| `SCHEDULED_RETRY_PROMOTION` | 预约重试自动提升 |

### 3.2 实例实验设置（DB，PATCH 即生效；与 env 叠加）

`GET/PATCH /api/instance/settings/experimental`：

```json
{
  "enableTimerHeartbeatByDefaultForEligibleRoles": false,
  "timerHeartbeatEligibleAgentRoles": [],
  "enableIssueGraphLivenessAutoRecovery": false
}
```

### 3.3 智能体

- **所有 agent** `runtimeConfig.heartbeat.enabled = false`，除非董事会 **单点** 授权某角色定时巡检。
- 曾漏网：`质量`（qa）仍为 `true` 时会定时 wake——须 PATCH 关掉。

### 3.4 仍会发生（刻意保留）

| 行为 | 说明 |
| --- | --- |
| 指派 + 手动唤醒 | 人主动派活 |
| 评论 wake（带档位） | 见 [24](../长期需求/24%20评论唤起过载与编排分层改版计划%202026-05-17.md) / 041～043 |
| run 正常 succeeded/failed | 不自动造交接单 |
| 人标 `blocked` | 不替代为 recovery 子单 |

## 4. 按需启用（以后只开一条）

**顺序（强制）：**

1. 简单编排 **已跑顺**（ROU-6 类：派活 → run succeeded → 中文摘要 → done）。
2. 记录 **重复人工步骤**（例：「每次 assign 后都要再点唤醒」）。
3. **只开一条** 自动化（一个 env 或一个 experimental 项）。
4. 用 [024](024-实践-运行验收与闭环判读.md) 验一条事务 + 观察 **48h 无衍伸单风暴**。
5. 回写本文 §5 启用表 + [058](../执行/058-routic公司编排v2.md)。

**不要：** 因 upstream changelog 整包 merge 默认编排。

## 6. 重启与验证

```powershell
pnpm dev:stop
# 若 4100 仍占用：结束占用进程后再起
pnpm dev:once
curl.exe http://127.0.0.1:4100/api/health
curl.exe http://127.0.0.1:4100/api/instance/settings/experimental
```

**健康：** `status: ok`；experimental 中 timer 角色为空数组。

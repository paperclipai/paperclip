# 项目计划（Routic 式工作方式）

本目录为**仓库内协作**（任务、探查、验收证据）；**不**按对外资料库口径维护。对外向文档见 `docs/` 其余部分及 `AGENTS.md`「`docs/` 域文书」条目。

本目录与 `01.项目需求说明.md`、`00.项目任务清单.md`（含**验收标准**）、`执行/`、`探查/`、`最佳实践/`、`长期需求/` 配套使用。

下面有一条**推荐的一般顺序**（**任务不清、容易拍脑袋动代码时**用来减返工）；**不是**「执行单只能从需求端一步一步线性生发」的硬管道。实际怎么开 `执行/` 单，常见只有三条路径，且**顺拆、反补都可以**——见下文 **「执行单从哪来」**。

**Paperclip 租户（实例侧）约定：** 公司已收敛为 **test-co（测试）+ routic（正式）** 等既定集合。**新建公司：**非经人类当轮拍板不得扩张——总则 **[`AGENTS.md`](../../AGENTS.md)** §5 第 **16** 条，Cursor **[`.cursor/rules/routic-project.mdc`](../../.cursor/rules/routic-project.mdc)**。**迁移门禁：**新适配器、模型搭配、指令词/模版等与执行平面相关的变更，须在 **`test-co` 验证闭环**后再同步到 **`routic`**（§5 第 **17** 条）。

1. **任务不清楚 → 先讨论**（与人或 Board 对齐目标、边界、成功标准；不要猜任务）。
2. **讨论仍不清楚 → 先探查**：本地 **代码 + 仓库文档（如 `doc/`、`AGENTS.md`）+ [`index.md`](index.md) 全量链接**；必要时 **上网检索**补上下文。
3. **需求落定 → 再执行**（`01` 或任务单里**已写清边界**后再大改代码/动环境；**若已按下文「执行单从哪来」先落执行单**，可把「落定」理解为该单自身写清即可，**不必**强求先改 `01`）。
4. **执行结束 → 收口边界**：拆 **下一单** 承接剩余工作，或把新共识 **记回需求**；**回写 `最佳实践/`**（可复用的路径、命令、坑）。
5. **验证结论与证据 → 写进对应 `执行/` 任务单**；**验收标准**见 `00.项目任务清单.md`；**残留风险**记在 `01.项目需求说明.md`「残留风险」节。
6. **`00.项目任务清单.md`** 顶部「状态｜任务单」**表**只列状态与链接（不写执行过程）；**验收标准**、规则与「验证证据存放处」写在同文件专节；任务标题与执行过程一律放在 `执行/` 各单内。

### 执行单从哪来（三种路径，顺序不拘）

`执行/` 任务单在实践里通常只有下面三种**建设**来源；**不要求**总是 1→2→3，也**常反着来**（先落执行单再补需求、先摸界面再回写 `01`、探查结论长大后再拆子单等）。**主表 + 文件取号**仍按 **[「`执行/` 新建 `.md` 命名与登记」](#执行-新建-md-命名与登记)** 办，与「你先从哪条路进来」无关。

1. **从需求来**：`01` 或讨论已定边界，再新建 `执行/` 并记入 `00` 主表。  
2. **从任务单 / 探查单分拆而来**：既有 `执行/` 或 `探查/` 要拆并行、拆风险、拆验收，新作子单；母单与 `01` 可随后再对齐、再补链。  
3. **人类检查界面或临时起意**：对 UI / 实例 / 文档看一眼决定要记一笔，可先写 `执行/` + 登主表，再把关联需求、探查结论、Board 收口**后补**。

**编号（`NNN-`）只表示该单进仓库的先后**，不表示你从哪条路来、也不表示必须先写完 `01` 才能开单。

**关于 `执行/` 文件名里的三位前缀（如 `001-`、`022-`）：** 只表示**任务单进入仓库的时间线**（建档先后），**不表示业务优先级**，也不当作路线图排序；是否优先做哪一单，以讨论结论、`01.项目需求说明.md` 里的开放问题以及 Board 为准。

## `执行/` 新建 `.md` 命名与登记

**须**与 **[`00.项目任务清单.md`](00.项目任务清单.md)** 主表同步，**不要**只落文件不记表。本节与 **[「执行单从哪来」](#执行单从哪来三种路径顺序不拘)** 无关：无论需求先、分拆先还是临时起意先，**取号与登记表**都照下面做。

1. **取号**：在 **`docs/项目计划/执行/`** 下列出（或 `glob`）所有符合 **`NNN-*.md`**（三位十进制数字前缀）的文件名，取其中 **最大 NNN**，新任务单文件名为 **`(NNN+1，固定三位)-中文主题.md`**（例如已有 `026-…` 则新建 `027-…`）。  
   - **例外**：人类当轮**明确指定**另一个合法三位号且不与已有文件重名时，以当轮为准。  
2. **登记表**：在 **`00.项目任务清单.md`** 顶部「状态｜任务单」表 **新增一行**，链到该新文件；「状态」与新建任务单文首 YAML **`status:`** 一致（常用初始值为「待办」）。  
3. **前缀含义**：与本节**上文**「关于 `执行/` 文件名里的三位前缀」一致（表建档时间线，不代表业务优先级）。

**任务单状态语种：** `执行/*.md` 文首 `status:` **只使用中文**（与 `00.项目任务清单.md` 所列五种取值**逐字一致**），不写 `todo` / `active` / `done` 等英文枚举。

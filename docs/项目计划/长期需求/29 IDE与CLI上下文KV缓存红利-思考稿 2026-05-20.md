# 29 · IDE 与 CLI 上下文 KV 缓存红利（思考稿）

**人话：** 模型侧的 **prompt / KV 缓存** 吃的是 **稳定、重复的前缀**；IDE 和 CLI **喂上下文的路径不同**，要想 **同样省钱、同样可预期**，文档与编排要 **刻意分层**——不能假设「同一份 markdown 在两个入口里自动等价」。

**不替代：** 叠层总图 **[022](../最佳实践/022-实践-无头编排提示词叠层与可观测性.md)**；计量口径 **[015](../最佳实践/015-实践-CLI累计计量与成本控制面台账口径.md)**；降费母题 **[23](23%20心搏巡检粗算成本口径与编排降费%202026-05-17.md)**；已落地的 invoke 侧 **`promptCacheCorrelation`**（见 5 月 17 日对话整理版 §十）。

---

## 0）前提（两条线共用）

| 概念 | 说明 |
| --- | --- |
| **KV / prompt cache** | 厂商对 **相同前缀** 的 input 打折或加速；**前缀越稳、越长，越容易连续 hit** |
| **invoke 语义 vs 计费语义** | Paperclip **`promptCacheCorrelation`** 描述「本轮 stdin **故意省略/保留** 哪些块」；**真金白银** 看 `usageJson.cache_read_input_tokens`（**015**） |
| **稳定前缀 / 可变后缀** | 宪法、规则、技能索引、项目 README → **尽量固定且靠前**；wake、任务描述、评论增量 → **靠后且短** |
| **文档≠一次 read 的包** | Obsidian **属性 + 正文**、Board **多标签**：对人分栏，对模型仍是 **一次工具返回里的 token**；**是否利于 cache 取决于你怎么读、怎么拼进 prompt** |

## 1）思考一：给 IDE 喂上下文时，文档怎样才容易吃到 KV 缓存

**问题：** Cursor / IDE 类入口里，rules、@ 文件、skills、对话历史怎么排，才能让 **跨 turn、跨会话** 的大块说明 **重复命中**？

**待探索方向（未评审）：**

| # | 方向 | 备注 |
| --- | --- | --- |
| I1 | **静态块前置、任务块后置** | `.cursor/rules`、`AGENTS.md`、不变的 `SKILL.md` 骨架 → 尽量 **每轮位置与措辞一致**；当日任务/wake 不进 rules |
| I2 | **少改措辞、多改指针** | 稳定段写「去读 `path/to/SKILL.md` §3」；**避免** 每轮把同一段规程改几个同义词（前缀 hash 变） |
| I3 | **Obsidian / 项目文档结构** | **frontmatter / Properties**：类型、状态、标签、链接（短、结构化）；**正文**：叙述与任务。读整文件时仍是一包 token，但 **设计时** 可把「每轮必带的短 metadata」与「长正文」分开维护，IDE 侧 **@ 小文件 / 摘要** 而非整库 |
| I4 | **技能与 rules 分工** | 公司级「宪法」一次加载；事务级 **045 minimize** 单行指针——避免 IDE rules 与 Paperclip wake **双份全文**（**022** §6） |
| I5 | **验收** | 对照 `cache_read_input_tokens` / 厂商 cache 字段；IDE 侧目前 **无** 与 Paperclip `promptMetrics` 同级的统一面板 → 可能靠 **用量回执 + 人为不改稳定段** |

**开放问题：**

- IDE 内部是否把 **rules + 默认 system + 工具表** 合成一层我们看不见的 A 层（**022** §2）——文档优化 **上限** 在哪？
- **多 worktree / 多根** 时，稳定前缀是否因路径绝对化而失效？

## 2）思考二：给 CLI 喂上下文时，如何吃到 **同样** 的红利

**问题：** 无头 `codebuddy_local` / `qwen_local` / `cursor-local` 的 stdin 拼装，能否与 IDE **同一套「稳定前缀」策略** 对齐，而不是 **续跑时堆 D+E 层雪球**？

**待探索方向（未评审）：**

| # | 方向 | 备注 |
| --- | --- | --- |
| C1 | **对齐 `promptCacheCorrelation`** | 已有：`mode` cold/续、`stabilityKey`、`suppressedSectionIds`；**待补**：各适配器一致产出 + UI 与 **cache_read** 并排（5/17 纪要） |
| C2 | **续跑省略块与 IDE 行为对齐** | **022** §3：CodeBuddy 续跑仍可能 **heartbeat_template 全文**；Cursor 续跑常 **0**——产品债：**有 wake 增量时 suppress 模板** |
| C3 | **冷启动专用稳定文件** | `--system-prompt-file` / 项目 `.codebuddy/settings` 放 **不变** 段；**续跑不再传**（已是现状）→ CLI 稳定前缀主要在 **session 内 E 层**，跨 heartbeat 靠 **session 复用** |
| C4 | **stdin 块顺序冻结** | `bootstrap → wake → session_handoff → task_context → skill_note → heartbeat_template`（CodeBuddy）**顺序变动 = 前缀变** → 改 adapter 时要当 **cache 兼容** 问题 |
| C5 | **技能 lazy / minimize** | **045**：`skill_note` 续跑或轻档 wake → **单行指针**，把「读 SKILL 全文」留给工具 **Read**（进 E 层/session，不每轮塞 D 层全文） |
| C6 | **与 IDE 共享「稳定文档真源」** | 同一 `AGENTS.md`、同一 `.cursor/rules` 文案；CLI cwd 指仓库时 **C 层** 与 IDE **同源**，减少「IDE 命中、CLI 换措辞不命中」 |

**开放问题：**

- CLI **厂商 A 层**（工具枚举、MCP 列表）能否配置瘦身，还是只能接受 cache 吃这层？
- **新 session 轮转** vs **长 session 续跑**：省钱与 context 上限如何 trade-off（**022** §6.3）？

## 3）两条思考的交叉点（后续可拆执行单）

1. **一份「稳定前缀清单」**：哪些文件/块 **禁止频繁改字**、哪些 **必须每轮变**——IDE 与 CLI **共用清单**。**IDE 侧已落地：** **[`.cursor/rules/README.md`](../../../.cursor/rules/README.md)**（2026-05-20）。  
2. **度量闭环**：run 详情同时看 **`promptCacheCorrelation` + `cache_read_input_tokens`**，避免「以为省略了块就等于命中了 cache」。  
3. **文档形态规范**（可选最佳实践）：Obsidian / `docs/` 写 **短 frontmatter + 长正文**；Paperclip wake **只引路径**；与 **[028](28%20Agent控制面API一等公民调用通路-MCP或默认脚本-2026-05-20.md)**「少 shell 拼 JSON、多结构化读」同向。

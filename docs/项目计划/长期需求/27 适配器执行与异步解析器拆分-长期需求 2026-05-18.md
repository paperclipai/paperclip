# 适配器执行与异步解析器拆分（长期需求）

**状态：** 草案（待评审与分期落地）  
**日期：** 2026-05-18  
**动机：** 当前「解析 stdout/JSON」与「一次执行是否成功」耦合在适配器与同一 `run` 生命周期内，无法覆盖跨 OS、跨 CLI 输出形态差异与超大输出，导致 **进程与业务已跑完** 仍被登记为 **run 失败（如 parse_error）**，观测与计费/续跑语义混乱。

---

## 1. 背景（为何不是「再补几行适配器」能搞定）

- **多系统**：Windows / Linux / macOS 下同一 CLI 的编码、换行、路径、子进程缓冲行为不一致。  
- **多软件**：各编码工具 stdout 合同不同——整段 JSON、JSONL、流式片段、混日志、版本间字段漂移。  
- **体量**：单次打印可达超大体积；在 **同步解析、且与 run 成功条件绑定** 时，易触发截断、内存与超时，表现为 **parse 失败**。  
- **结论**：「解析 + 归一化 + 降级」是 **独立演进面**，与「能拉起进程并把原始输出可靠落盘」拆开，否则适配器无限膨胀且仍误判失败。

---

## 2. 目标架构（职责拆分）

| 组件 | 职责（边界内） | 明确不做 |
| --- | --- | --- |
| **适配器（Adapter）** | 组装命令与环境、注入提示词与上下文、**拉起子进程**、按需 **流式转发日志**、将 **原始输出**（stdout/stderr 或 spill 文件句柄）**完整落库/落盘为原始记录**；以「子进程协议完成」为主结束条件；**CodeBuddy 等 stream-json**：终局用 **尾行 `type:result` + exitCode** 判业务成败，**不**全文硬解析 stdout。 | 不承担「所有厂商所有版本的 JSON 语义」；不把 **结构化解析失败** 等同于 **执行失败**；**不为 UI 展示** 解析 NDJSON；**不**默认把全量 stdout 写入 `resultJson`。 |
| **解析器（Parser，独立组件）** | **按适配器类型独立实现**（与 `packages/adapters/*` 同构，**不混装**一家一个）：tail `run-logs/*.ndjson`，对 **完整行** 做 **轻量分类**（工具 / 思考 / 写入 / result / 噪声等）；可选异步进程；尾行 `usage` 等 **台账字段** 能抽则抽，抽不到不挡执行。 | **不**做会话记忆回溯、**不**转写成人话、**不**复刻 IDE 全量语义；**不**拉起进程；**不**单独决定执行成败（与适配器共用 **尾行 result + exitCode** 契约）。 |
| **控制面 / Run** | 将 **「执行阶段成功」** 与 **「解析阶段成功」** 分层登记；解析失败 **不** 污染「本次 CLI 执行已成功结束」的结论（具体字段名与 UI 表现由实现期定）。 | 不在本文展开具体 schema。 |

**异步含义：** `run` 可在原始记录落盘后进入「执行完成、解析排队/处理中」；解析完成后再写结构化摘要，避免阻塞子进程回收与心跳节拍。

## 3. 需求条文（验收口径）

1. **执行成功与解析成功解耦：** 子进程按适配器约定结束（含 exit 约定）且原始记录已持久化 → **执行事务**视为成功路径；解析异常单独产生 **解析状态/错误码**，**不复用** 与 `codebuddy_execution_error` 同级的「整次运行失败」语义（具体枚举实现期定义）。  
2. **原始记录一等公民：** 任何解析步骤必须能追溯到 **未裁剪的原始或可追溯分片**（与 [`执行/047-适配器_stdout解析失败与结构化摘要.md`](../执行/047-适配器_stdout解析失败与结构化摘要.md) **C1** 一致方向）。  
3. **解析器可插拔与可测试：** 解析逻辑 **独立于** 各 `packages/adapters/*` 的 `execute` 路径；**一适配器类型对应一解析器包/进程**（如 `codebuddy-parser`、`cursor-parser`），共用仅 **读 ndjson、行缓冲、统一标签枚举**；以 **fixture 行** 做单测，不按「万能 parse()」分支。  
4. **跨平台非功能性：** 编码与换行在 **落原始记录** 阶段统一规范化策略（至少约定 UTF-8 与换行归一入口），解析器只吃 **已规范化或已标注编码** 的存盘内容。  
5. **超大输出：** 解析应能对 **spill / 分段 NDJSON** 工作，避免「全文 `JSON.parse` 一次」作为唯一路径（与 **047 C1–C2** 互补）。
6. **展示与解析分离：** 运行详情展示 **只读原始 run-log（+ 实时 WS）**；人类无需读懂流式 JSON；结构化字段缺失时 UI 显示「解析中/未解析」，**不**阻塞「执行已成功」叙事（**059** 已落地运行期旁路；**047** 落地终局与收薄适配器）。

下表用于统一讨论用语，避免「口述一词、实现另一义」。

| 你在讨论里可能这么说 | 本文档中的正式含义 | 备注 |
| --- | --- | --- |
| 「适配器能力不够」 | 常与 **解析与执行绑死、stdout 合同多变** 混为一谈；拆分后应区分是 **拉起/落盘** 缺口还是 **解析策略** 缺口。 | 口头少说笼统「适配器不行」，改为 **执行链** vs **解析链**。 |

| 「JSON 不一样」 | 期望由 **解析器策略表**（按工具+版本+格式）吸收，而不是塞进单个适配器 `execute.ts`。 | |

| 「这一行是工具还是思考」 | 解析器的 **classify** 输出，供 UI 折叠/过滤；**不是**给人读的完整译文。 | |
| 「异步」 | 解析 **后置于** 原始落盘，不阻塞把 run 标为执行完成（实现期定义中间态）。 | |

## 7. 解析器形态（讨论稿，非验收）

> **状态：** 概念对齐用；字段级真值以各厂商 CLI/API 文档与 [`015-实践-CLI累计计量与成本控制面台账口径.md`](../最佳实践/015-实践-CLI累计计量与成本控制面台账口径.md) 为准；实现另开执行单，**不**在本文写分期编号。

### 7.1 与贾维斯（记忆管线）的分工

| 系统 | 管什么 | 不管什么 |
| --- | --- | --- |
| **贾维斯** | 跨 run / 跨日的 **记忆抽取**、候选、摘要、注意力保护（人读得懂的「发生过什么」） | 单次子进程怎么 spawn、run 成败 |
| **Paperclip（回形针）** | **执行治理**：派单、run 状态、原始 run-log 真源、台账（usage 等）、旁路存活 | **重型会话回溯**、把整段转写当记忆真源 |

因此：**不必在回形针里再做一套「读懂整段会话并写入长期记忆」的解析器。**  
run-log 保证 **可审计、可复盘**；若要「提炼教训 / 归档记忆」，走 **贾维斯 只读摘要**（参见编排 v2：**不把原始 run log 堆进 Jarvis**）。

Paperclip 解析器只服务 **控制面当下**：这一行在干什么、账单字段有没有、UI 要不要按类型折叠。

### 7.2 按适配器独立，不混装

对齐适配器做法：

- **一个 `adapterType` → 一个解析器实现**（独立包或独立进程均可），例如 `codebuddy_local` ↔ `codebuddy-parser`，`cursor` ↔ `cursor-parser`。
- **禁止** 在单文件里 `switch (vendor)` 堆千行映射——版本升级时必乱。
- **可共用**：读 `{ts, stream, chunk}` ndjson、行缓冲、tail/spill、输出 **统一标签枚举**（见 §7.3）。
- **Cursor 现状**：`packages/adapters/cursor-local/src/ui/parse-stdout.ts` 等已是「分类/事件化」雏形，迁出为独立解析器时 **复用规则**，而非重写万能内核。

解析器可以是 **独立进程**（tail run-log、边跑边 classify），与 heartbeat 通过 **文件真源** 解耦；进程挂掉不影响 run 执行。

### 7.3 行级分类（classify），不是转写（translate）

解析器对 **每一条完整 NDJSON 行**（或厂商约定的行协议）输出 **Paperclip 统一标签**，例如：

| 统一标签（草案） | 人类直觉 | 典型用途 |
| --- | --- | --- |
| `tool` | 在调工具 | UI 折叠、存活旁路「有动静」 |
| `think` | 思考/推理块 | 默认折叠，避免刷屏 |
| `write` | 改文件/写盘类 | 高亮「动代码了」 |
| `assistant` | 普通助手正文 | 可选展示 |
| `result` | 终局行 | 成败 + usage 抽取入口 |
| `system` / `stderr` | 系统或错误流 | 排障 |
| `noise` | 无法识别或非协议行 | 原样保留 |

**明确不做：**

- 把工具参数、文件 diff **翻译**成长篇中文叙述；
- 重建「模型内心独白」供人阅读；
- 替代贾维斯 做 **跨 run 记忆合并**。

UI：**原始模式** 仍只读 chunk；**标签模式** 用 classify 结果折叠/上色——两种视图共用同一 run-log。

### 7.4 厂商行协议 → 统一标签（映射表）

各 CLI 的 `type` 字段 **互不相同**（CodeBuddy `stream-json`、Cursor `stream-json`、Claude Code 生态另有社区 parser）。  
每个独立解析器维护 **本地映射表**，例如（示意，非真值）：

```text
vendor type "tool_use"     → Paperclip tool
vendor type "thinking"     → Paperclip think
vendor type "file_edit"    → Paperclip write
```

映射表 **带版本号** + fixture 单测；协议变更只改对应解析器包。

### 7.5 用量与 input / output 字段（概念层，待你补厂商文档）

你提到的困惑是正常的：**「AI 内部的输入/输出」在编排壳上通常有三层，不要混为一谈。**

| 层次 | 是什么 | 谁定义 | Paperclip 怎么用 |
| --- | --- | --- | --- |
| **A. 模型 API** | 单次 HTTP 调用的 `prompt_tokens` / `completion_tokens` / `cached_tokens` 等 | OpenAI、DeepSeek、Anthropic… **各家不同** | 多在 CLI **汇总进 `result` 行** 的 `usage` 里间接出现；对账见 **015** |
| **B. CLI 会话** | 同一会话多轮累计，CLI 可能在 **`result` 报累计标尺** | CodeBuddy / Claude Code 等 | 展示易偏大；会话差分见 **015** |
| **C. Paperclip run** | 一次 heartbeat 生命周期 | 回形针 | `usageJson` 记 **本条 run**；优先 **尾行 result 抽取**，解析器不必扫描全文 |

**原则（写进解析器需求，但不假装字段已统一）：**

1. **台账字段**（input/output/cache/cost）从 **尾行 `type:result` 的 `usage` 子对象** 尽量读取；读不到 → 空，**不** 因此判 run 失败。  
2. **字段名因厂商而异**（如 `input_tokens` vs `inputTokens` vs 分项 CSV）；各 **` *-parser`** 包内做 **normalize → Paperclip 内部枚举**，不散落在适配器。  
3. **DeepSeek / CodeBuddy** 等以你手头的 **官方示例与控制台导出** 为真值；读完后在对应 parser 包 README 或 fixture 注释里 **钉死样例行**，不在 27 正文堆具体 JSON（易过期）。  
4. **解析器 classify** 与 **usage 抽取** 是两个函数：`classifyLine(line)` vs `extractTerminal(tailLines)`，避免混在一个「大解析」里。

### 7.6 社区参考（借鉴形态，不绑定依赖）

- **单厂商**：如社区 **Claude Code `stream-json` parser**（npm）——可参考 **按行 + 状态机**，不能代替 CodeBuddy 映射表。  
- **本仓**：Cursor `parse-stdout` 已验证「事件分类」路线。  
- **编排产品**（OpenHands 等）：多为 **内置 JSONL**，一般 **不可直接当 Paperclip 依赖**，只作「旁路消费 stdout」的思路参照。

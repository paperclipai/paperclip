# 长期需求：普通用户使用说明书

**状态：** 需求记录（待拆单落地）  
**日期：** 2026-05-14  

## 背景

当前仓库里 **`docs/` / `docs/*` 与公司包规范**（如 `docs/公司/01 公司规范 companies-spec.md`）主要面向 **看板管理员、包作者、集成开发者**：技能导入（`skills/import`）、可移植包目录约定、`desiredSkills` 等，**普通终端用户**既无权限也无意图阅读这些材料。

结果：**缺少一层「给普通用户的使用说明书」**——不写实现契约，只回答「我能干什么、界面里点哪里、常见任务怎么收束、钱/任务状态怎么看」。

## 读者界定（本需求所指「普通用户」）

- **不**负责：公司包迁移、技能库导入、适配器配置、雇佣智能体、审批策略等 Governance 深水区。  
- **会**：登录看板（若开放）、查看分配给自己的工作、看活动/成本、与智能体协作产物打交道；或仅作为 **任务参与方** 读评论与状态。

（与 **Board 操作者/董事会** 指南区分：后者已有 `docs/guides/board-operator/` 等，偏 **运营公司**；本需求偏 **不运营公司的使用侧**。）

## 目标产物（说明书应长什么样）

1. **单独成篇**（或 `docs/start/` 下一级「普通用户指南」），与 SPEC、公司规范、技能 README **并列入口**，在 `docs/项目计划/README.md` 或站点导览里可被 **非技术读者**发现。  
2. **零前提**：不假定读者读过 `agentcompanies`、API、`skills/import`。  
3. **场景驱动**：例如：第一次登录 → 看任务状态 → 找人/找单 → 理解「完成 / 阻塞 / 审查中」→ 成本控制入门（若对其可见）。  
4. **边界诚实**：写明哪些事 **必须** 找管理员（装技能、建新智能体、改预算等），避免读者在公司规范里迷路。  
5. **语言**：与项目中文资料策略一致（正文中文，路径/API 英文名可保留在附录或灰框）。

## 验收直觉

- 一位 **未**读过后台文档的贡献者/家人，仅凭「普通用户说明书」能在 **15 分钟内**说清：自己在这套系统里 **管什么、不管什么、卡住找谁**。  
- 与现有 **Board 操作者**、**Agent 开发者** 文档 **交叉链接**但不互相替代。

## 与实现的耦合

- **不**强制具体介质（站内页 / PDF / 视频）；优先 **仓库 Markdown** 真值，便于与 PR 同行演进。  
- 若未来产品增加「帮助中心」或应用内引导，应以本文档为 **内容源** 或 **对照清单**。

## 落地提示（供拆单）

- 可先占位 `docs/start/普通用户指南.md`（文件名可由文档规范最终拍板），再在 `00.项目任务清单.md` 拆一条执行任务。  
- 与 **Board 中英混排**（ROU-20）等问题相交：说明书若面向中文读者，应使用 **已中文化的界面截屏或文案** 为例，避免读者对照不到 UI。

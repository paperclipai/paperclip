## 7. 契约摘要（工人侧 / Messages API 客户端形状）

> vendor 为 Anthropic Claude Messages；**编排实现应依赖「抽象后的会话模型」**，而非 SDK 类型名。

### 7.1 单次请求（非流式）核心字段（摘自 `messages.create` 测试）

- **必填**：`model`、`max_tokens`、`messages`（含 `role` + `content`）
- **常见可选**：`system`（块数组）、`tools`（含 `name`、`input_schema`、`type` 等）、`tool_choice`、`stream`、`temperature`、`top_p`、`metadata`、`stop_sequences`、`thinking`、`output_config`（如 `json_schema`）等

### 7.2 消息与内容块（多轮 / 工具闭环）

- **Assistant**：`content` 可为多块；含 `type: text`、`type: tool_use`（字段含 `id`、`name`、`input`）
- **User 侧回传工具结果**：`type: tool_result`，**必须**带 `tool_use_id` 与 `content`（与 assistant 侧 `tool_use.id` 对齐）
- **并行工具**：同条 assistant 消息可含多个 `tool_use`；每条结果对应一条 `tool_result`（测例见 `ToolRunner.test.ts`）

### 7.3 流式（SSE）

- 事件类型（fixture）：`message_start`、`content_block_start`、`content_block_delta`（`text_delta` / `input_json_delta`）、`content_block_stop`、`message_delta`（含 `stop_reason`）、`message_stop`、心跳类 `ping`
- **`stop_reason` 样例**：`end_turn`、`tool_use`、`max_tokens`（截断）

### 7.4 「最小会话形状」启发（便于本机抽象）

- 有序 **messages**
- **content blocks**（文本 / 工具调用 / 工具结果）
- **流 = 聚合为最终 message 后再入会话历史**（编排层通常不持久化每条 delta，除非本机排障需要）

## 11. 编排模式研究：术语 · 组件 · 节点流转（外查归纳，**对齐单机编排**）

> **角色**：研究侧归纳，用于与《方法论》对词、对 **DAG/Coordinator/Worker** 的 **节点语义**；**不是**运行时说明。检索：2026-05-12。  
> **范围**：紧扣 **单用户本机**；下文若出现「租户/供应商」等，**忽略**或视为中性技术词（如 Join 屏障），**不**导出多租户需求。

### 11.1 方法论词条 ↔ 外部常见说法（对齐用）

| 方法论 | 外部文献/产品里常见叫法 | 备注 |
|--------|------------------------|------|
| **大脑** | Planner、Orchestrator、高层 policy | 强调「拆与派」，不代做工 |
| **双手 / Worker** | Executor、Tool-calling agent、子代理 | 与「会调工具、跑沙箱」一致 |
| **编排器 Coordinator** | Orchestrator-Agent、Coordinator、Conductor | 负责任务分解、路由、汇合、故障域 |
| **Agent（模板）** | Agent config、system prompt + tool manifest | 「相对静态」与编排里 worker 配置同构 |
| **Session** | Conversation state、checkpoint、thread | 常混称 memory；你们强调 **可外置工件** |
| **DAG / 任务图** | Task graph、workflow DAG、fork-join | 工作流引擎里 join 常显式成节点 |
| **汇合** | Join、aggregation、reduce / barrier | LangGraph 用 reducer 合并发写入 |
| **蜂群 / 无状态** | Stateless compute + externalized session | 与水平扩展、checkpoint 叙述一致 |

### 11.2 实现编排时常见的「组件」清单（供落地选型）

以下 **不绑定** 某一商用产品；是多篇文章里反复出现的 **能力切片**：

1. **规划 / 分解**（本机可先规则化或单次 LLM 调用）：把目标拆为带依赖的子任务。
2. **路由 / 派单**（本机简化）：按 **DAG 边与节点配置** 决定下一 runnable 集；不必上「多租户路由」。
3. **图执行引擎**：维护 **DAG**；支持并行扇出、`Join`、跳过/重试、**单进程内并发上限**（个人 machine 资源）。
4. **状态与工件存储**：**编排平面**（节点状态、`run_id`）与 **Session 平面**（每节点 `messages` 等）**分列**；本机可用 **同一 SQLite 不同表** 或 **目录分文件**。
5. **可观测与恢复**（本机）：**append-only 运行日志** + **Session 快照** 即可复现；不必上分布式 tracing。
6. **执行适配层**：本机的 **子进程 / 工作目录 / 超时**；若不当接公网 API，**凭证**可缩为 **可选**。

外部对该原语的打包表述可参考：[AI Agent Orchestration Patterns](https://productschool.com/blog/artificial-intelligence/ai-agent-orchestration-patterns)、[Orchestrator Agent Pattern](https://www.agentpatterns.tech/en/agent-patterns/orchestrator-agent)、[Multi-Agent Orchestration 实践架构指南](https://www.augmentcode.com/guides/multi-agent-orchestration-architecture-guide)。

### 11.3 节点流转（状态机 + 控制流）

**（A）单任务 / 单 Worker 节点**（借用工作流引擎通用生命周期）

常见状态族：**SCHEDULED / QUEUED → IN_PROGRESS → COMPLETED**，以及 **FAILED、TIMED_OUT、CANCELED、SKIPPED**（见 [Conductor Task Lifecycle](https://conductor-oss.github.io/conductor/devguide/architecture/tasklifecycle.html)）。与方法论对应：**节点 = Worker 一次可验收的出活边界**；失败时蜂群模式倾向于 **弃实例、换新实例、从 checkpoint 重放** 而非「焊接长寿 VM」。

**（B）图级控制流：扇出 / 汇入**

- **Fork / 并行分支**：多子任务同时进入可运行集，依赖边全部满足则被调度。
- **Join（屏障）**：等指定前驱全集到齐后聚合输出，再触发下游（[Conductor Join](https://conductor-oss.github.io/conductor/documentation/configuration/workflowdef/operators/join-task.html)；[AWS Step Functions 任务状态](https://docs.aws.amazon.com/step-functions/latest/dg/state-task.html) 用于对照「状态机表达」）。

**（C）「超步」执行视角**（LangGraph / Pregel 系）

离散 **super-step**：同一步内可并行跑的节点写入 **不相交 state 通道**，由 **reducer** 合并——这等价于你们的 **并行分支 + 结构化汇合**，只是实现上可能是状态通道而非显式 Join 算子（见 [LangGraph Graph API](https://docs.langchain.com/oss/python/langgraph/graph-api)）。**注意**：LangGraph 常见论述强调 **带环** 的 agent 回路；你们方法论明确 **DAG（无环）** + 后期汇入——选型时避免默认「框架默认 = 需求默认」。

### 11.4 学术侧：任务分解的形式模型（可选深读）

- **分层任务网 HTN**：**复合任务** 分解为 **原语任务**，得到可执行序列；与「编排器拆子任务」在概念上同构。综述入口：[arXiv:1403.7426 HTN Overview](https://arxiv.org/abs/1403.7426)（或 [OpenReview 综述条目](https://openreview.net/forum?id=AJoT6RxlNS)）。
- 用途：需要 **严格定义「分解合法性」「规划终止」** 时，可用 HTN 词汇补你们的 DAG 节点语义（不要求一开始上完整 HTN 求解器）。

### 11.5 Session 外置与无状态计算（与 §3、§6 互证）

公开叙述里与你们一致的二分法：**无状态计算层** + **外置 durable 状态**（会话、checkpoint、事件日志），支撑水平扩展与恢复；常见还拆 **session 内短时态 / 文件型工件 / 事件溯源** 多层（参见 [Agent Memory Patterns](https://website.understandingdata.com/agent-memory-patterns/)、[Stateless agents, stateful product](https://www.kitewing.ai/blog/stateless-agents-stateful-product/)）。

### 11.6 研究结论（仍仅任务单；是否进《方法论》另审）

- **名词建议**：个人笔记可并行使用 **Coordinator + Worker** 与 **Planner + Executor**，并记清 **Executor = Worker（双手）**，避免把「大脑」理解成唯一 API 调用方。
- **必选组件**：只要做 DAG 编排，至少需要 **图存储/执行器、节点状态机、Join 语义、与 Session 存储分列**——与 §7 四层栈可一一对照落实为「模块边界」。
- **节点流转最小集合**：`pending`（等依赖）→ `runnable` → `running` → `succeeded | failed | canceled`；Join 节点额外有 **barrier（等扇出全集）**。
- **框架陷阱**：**DAG-first** 需求与 **「agent 回路优先」** 的框架默认值可能不一致；评估时先看 **能否显式 Join、能否禁止环、能否外置 checkpoint**。

### 11.8 个人电脑上的最小实现关切（**紧跟《方法论》，不问多租户/供应商**）

以下 **不** 指定语言栈，只列 **与本需求一致** 的落地边界：

| 关切 | 单机可行做法 |
|------|----------------|
| **编排平面** | 持久化 `run_id`、DAG 节点 **`pending/runnable/running/done/failed`**、Join 屏障前驱计数；**SQLite 单库** 或 **一 run 一目录 JSON** 均可。 |
| **Session 平面** | 每 DAG 节点 **独立** `messages`（及 tools/system 清单）序列化存储；满足 §6 **tool_use_id ↔ tool_result** 可重放。 |
| **蜂群 Worker** | **子进程**：每次 = **无状态 CLI（可内含模型调用）**；`kill` 后 **新进程** 从 **Session 快照** 继续；交付物走 **约定路径/stdout**。 |
| **大脑** | **Codex** 作主脑（拆任务、派活、读回交付物、汇合、决定下一步）；**run/DAG 台账** 外置与 §4 对齐。 |
| **汇合** | Join 节点从上游 **已持久化产物** 读入，**结构化注入** 下游节点输入（对齐 §9）。 |
| **刻意不做** | 多租户字段、供应商 DD、跨区复制、分租户计费、企业 IAM 全集——**不在本需求范围内**。 |

### 11.9 拓扑锁定（与《方法论》§10 一致）

- **主脑**：**Codex**。  
- **工人**：**无状态 CLI + 模型**（一次调用 = 一进程/一短链；交付 = 文件或 stdout 等约定协议）。  
- **实现时**：为每个工人 CLI 定好 **入参（环境变量/参数文件）** 与 **产出路径**，主脑只调度 **命令与验收**，不依赖工人进程内长生状态。

## 11.10 Codex 主脑 × 无状态 CLI–模型工人（**沿 §10/§11.9 深研**）

> **范围**：个人电脑、单用户；**不**展开多租户/供应商。材料来自公开文档摘要 + 本仓库 `06-命令-commands/Codex-使用备忘.md`（仍占位，**最终以本机 `codex --help` 为准**）。检索：2026-05-12。

### 11.10.1 主脑侧：Codex 能承担编排面的哪些机制

- **交互式主脑**：在终端/IDE 里跑 Codex，由你 **口述或粘贴** 目标任务；Codex 负责 **拆步、调度、读结果**——与《方法论》§4「不代工人做落地细节」一致：**你** 仍可通过 **规则 / 提示 / 子代理配置** 约束「只派 CLI、主脑不自改仓库」（以 [Codex Advanced config](https://developers.openai.com/codex/config-advanced)、[Subagents](https://developers.openai.com/codex/subagents) 为准）。
- **机器可读的工人交付**：若主脑（或外挂脚本）要 **解析** 工人输出，工人应约定 **stdout 单行 JSON** 或 **固定目录 artifact**，避免仅人类可读段落。
- **与「外置 run/DAG 台账」**：Codex 可能自有 **会话语料落盘**（见 config）；**主脑状态** 与 **工人 `--ephemeral`** 可并存——前者记在 **Codex/台账**，后者保证 **工人侧无会话文件**（见下）。

### 11.10.2 工人侧：`codex exec` 作为「一类」无状态 CLI（可选）

官方 **非交互** 入口：`codex exec "…"`（[Non-interactive mode](https://developers.openai.com/codex/noninteractive)）：

- **stdout**：倾向 **最终 agent 消息**，便于管道；**stderr**：进度流。
- **`--ephemeral`**：**不** 把会话持久化到磁盘 → 贴近 §3「进程可弃、状态外置」的 **工人**（若每次 `exec` 对应一个 DAG 节点）。
- **`--json`**：JSON Lines，便于主脑脚本解析。
- **`--sandbox`**：`read-only` / `workspace-write` / `danger-full-access`（[CLI reference](https://developers.openai.com/codex/cli/reference)、[Sandbox 概念](https://developers.openai.com/codex/concepts/sandboxing)）——按节点绑定沙箱强度。
- **`--ask-for-approval`**：`untrusted` / `on-request` / `never`（[Approvals & security](https://developers.openai.com/codex/agent-approvals-security)）。**无人值守工人** 若卡在审批会 **阻塞** —— 自动化场景多在 **受控环境** 用 `never` 等，需自担风险。
- **本机 `codex-cli 0.130.0` 额外可用项**：`--output-schema <FILE>`（约束最终响应形状）、`--output-last-message <FILE>`（把最后消息写入文件）、`--skip-git-repo-check`（允许非 Git 目录）、`--ignore-user-config`、`--ignore-rules`。这些选项让 **工人交付物协议** 更容易收窄为「写固定文件 + 校验 schema」。

**取舍**：`codex exec` **嵌套** = 工人也是小型 Codex，**时延/计费** 常高于 **薄 CLI + 单次模型调用**；适合「该节点本身就要多工具」的 **重工人**，不必所有节点都用。

### 11.10.3 工人侧：自研薄 CLI（与 §10 一致的默认推荐）

- 入参：`argv`、**单文件 job**（YAML/JSON）、或 **环境变量**。
- 出参：**约定目录**（如 `runs/<run_id>/<node_id>/result.json`）或 **stdout 仅此 JSON**。
- **无状态**：不假设跨调用内存；多轮由主脑 **多次起进程** 或 job 内带 **完整 messages 快照**（§6）。

### 11.10.4 Subagents（`.codex/agents/*.toml`）vs 外置 CLI 工人

- **Subagents**（[文档](https://developers.openai.com/codex/subagents)）：仍在 **Codex 进程/会话** 内切换专员，继承父沙箱——偏 **主脑内角色**，**不是** OS 级「杀进程即弃」边界。
- **你的拓扑**：**蜂群边界 = 子进程** 时 **故障隔离更清晰**；Subagents 可作 **主脑内脑力分工**，**替代不了** 外置 CLI 工人条目，除非产品上有意合并。

### 11.10.5 与 §7（Messages 契约）的衔接

- 工人若是 **薄 CLI**，**on-wire 形状** 不必是 Anthropic Messages；主脑 **汇合** 时做 **字段映射** 到你的 **节点 I/O schema** 即可。
- 工人若是 **`codex exec --json`**，主脑解析其 **输出 schema**，同样不必照搬 HTTP Messages。

## 11.11 单机最小工人协议草案（Codex 主脑调度 CLI 工人）

> 目的：把 §10/§11.10 收敛成 **可在个人电脑实现** 的最小协议。此处仍是研究草案，不代表已写实现。

### 11.11.1 目录与台账

建议先用 **目录分文件**，比一开始上 SQLite 更便于人工检查：

```text
runs/
  <run_id>/
    run.json                 # 编排平面：目标、DAG、节点状态
    nodes/
      <node_id>/
        job.json             # 主脑发给工人的完整输入
        result.json          # 工人交付的机器可读结果
        last-message.md      # 可选：人类可读最终消息
        stdout.jsonl         # 可选：codex exec --json 原始流
        artifacts/           # 工人产物
```

**原则**：`run.json` 是 **编排平面**；`nodes/<node_id>/job.json/result.json/artifacts` 是 **节点工件 / Session 平面**。Codex 主脑可以读写它们，但不把全局 DAG 藏进工人进程内存。

### 11.11.2 `job.json` 最小字段

```json
{
  "run_id": "run-20260512-001",
  "node_id": "research-sdk-contract",
  "kind": "research | edit | verify | summarize",
  "cwd": "C:/Users/wuhen/工具优化",
  "prompt": "本节点要完成的自然语言任务",
  "inputs": [
    { "name": "methodology", "path": "02-智能体-agents/Agent编排与无状态执行-方法论.md" }
  ],
  "constraints": {
    "no_network": false,
    "write_scope": ["runs/run-20260512-001/nodes/research-sdk-contract/artifacts"],
    "timeout_sec": 600
  },
  "acceptance": [
    "产出 result.json",
    "列出证据来源",
    "不得修改方法论正文"
  ],
  "session": {
    "messages": []
  }
}
```

**关键点**：若工人无状态，`job.json` 必须带足上下文；若需要恢复/重跑，主脑用同一 `job.json` 或更新后的 `session.messages` 再起一个新进程。

### 11.11.3 `result.json` 最小字段

```json
{
  "run_id": "run-20260512-001",
  "node_id": "research-sdk-contract",
  "status": "succeeded | failed | needs_input",
  "summary": "一句话结果",
  "artifacts": [
    { "name": "notes", "path": "artifacts/notes.md", "type": "text/markdown" }
  ],
  "evidence": [
    { "kind": "file", "ref": "research-cache/anthropic-sdk-typescript/tests/lib/fixtures/basic_response.txt" }
  ],
  "session_delta": {
    "messages": []
  },
  "error": null
}
```

**主脑只信 `result.json` 的状态与 artifact 列表**；stdout 仅作运输层或排障流，避免工人输出散落。

### 11.11.4 两类工人命令模板

**A. `codex exec` 重工人**（适合该节点本身需要读写/推理/多步工具）：

```powershell
codex exec --ephemeral --json `
  --cd "C:\Users\wuhen\工具优化" `
  --sandbox workspace-write `
  --ask-for-approval never `
  --output-schema "schemas\worker-result.schema.json" `
  --output-last-message "runs\<run_id>\nodes\<node_id>\last-message.md" `
  "读取 runs/<run_id>/nodes/<node_id>/job.json，完成任务，并只按 schema 输出最终 JSON。"
```

**B. 自研薄 CLI 工人**（默认推荐，便宜、可控、无多余会话）：

```powershell
worker.exe --job "runs\<run_id>\nodes\<node_id>\job.json" --out "runs\<run_id>\nodes\<node_id>\result.json"
```

取舍：**A** 更像“再开一个小 agent”；**B** 更像真正的蜂群工人。单机实现先做 **B**，只有复杂节点再考虑 **A**。

### 11.11.5 节点状态机（主脑台账）

最小状态：

```text
pending -> runnable -> running -> succeeded
                         |-> failed
                         |-> needs_input
                         |-> canceled
```

- `pending`：依赖未满足。
- `runnable`：依赖已满足，可派给 CLI。
- `running`：子进程已启动；记录 `pid`、命令、开始时间。
- `succeeded`：`result.json.status == "succeeded"`，且 artifact 存在。
- `failed`：进程失败、超时、schema 校验失败。
- `needs_input`：工人明确要求主脑/人补信息。

### 11.11.6 本草案对《方法论》的意义

- 对 §1：把“工人双手”具体化为 **可杀可重启的 CLI 进程**。
- 对 §4：`run.json` 是 **编排器状态**，不混入工人私有会话。
- 对 §6：`job.json/session` 与 `result.json/session_delta` 是 **Session 外置** 的最小入口。
- 对 §9：每个 DAG 节点自然拥有独立 `job/result/artifacts`，Join 只读上游 artifact。

## 11.12 继承旧派单模式：从「人工复制提示词」到 `job.json/result.json`

> 研究结论：`02-智能体-agents` 下已经存在成熟的 **人工派单治理**，尤其是 `多模型分工与使用方式.md` 与 `模型研判与派活-2026-04-30.md`。当前路线不是发明新模式，而是把旧模式的 **派单字段、闸门、收单口径** 机器化。

### 11.12.1 旧模式的真源与定位

- **现行真源**：`多模型分工与使用方式.md`。目录索引已标明它是“多执行位分工、任务路由与回填习惯”的真源。
- **历史短口径**：`模型研判与派活-2026-04-30.md`。包含 P4/P5 模型优先级、派活规则、三层闸门、主控轻派轻收、收单四问、执行方自找上下文规则。
- **人工流程现状**：主控手写/复制提示词给不同执行位；执行方按提示词找上下文、执行、验证、回写；主控红黄绿收单。

### 11.12.2 旧派单字段 → `job.json`

| 旧人工提示词要素 | `job.json` 字段 | 说明 |
|------------------|-----------------|------|
| 任务目标 | `prompt` | 仍保留自然语言，但放进结构化 envelope。 |
| 模型/工具选择 | `worker` / `model` / `channel` | 可先做可选字段；Codex 主脑仍可人工裁决。 |
| 可改边界 / 不可碰红线 | `constraints.write_scope` / `constraints.red_lines` | 替代“复制提示词里反复提醒”。 |
| 验收方式 | `acceptance` | 旧文要求“验收口径清楚”，这里直接数组化。 |
| 相关入口文件 | `inputs[]` | 来自“执行方自找上下文规则”。 |
| 执行方自报工具/模型/时间 | `report_required[]` | 由 result 侧回填。 |
| 本机北京时间检查点 | `report_required.started_at_local` / `ended_at_local` | 若继续需要，可保留为固定字段。 |
| 停手条件 | `constraints.stop_conditions` | 避免工人越界扩大任务。 |

### 11.12.3 旧回写格式 → `result.json`

旧文要求执行方回写：

```text
探查对象：
相关入口文件：
关键依赖关系：
可改边界：
不可碰红线：
建议后续任务：
验证方式：
仍不确定的问题：
```

可机械映射为：

```json
{
  "status": "succeeded | failed | needs_input",
  "summary": "",
  "context_map": {
    "target": "",
    "entry_files": [],
    "dependencies": [],
    "editable_scope": [],
    "red_lines": [],
    "next_tasks": [],
    "verification": [],
    "unknowns": []
  },
  "execution_report": {
    "tool": "",
    "model": "",
    "started_at_local": "",
    "ended_at_local": "",
    "duration": "",
    "tokens_or_credits": "",
    "limitations": []
  },
  "evidence": [],
  "artifacts": []
}
```

### 11.12.4 三层闸门 → 派单前校验

旧闸门：

1. **赚钱闭环**
2. **环境可验**
3. **模型适配**

在单机协议中对应：

- `job.json.priority_reason`：说明为什么值得派。
- `job.json.acceptance`：必须有可验证证据；没有则先派探查单。
- `job.json.worker_hint`：只是建议，不让工人自作主张换任务形态。

### 11.12.5 主控轻派轻收 → Codex 主脑收单逻辑

旧口径是红黄绿：

| 旧状态 | 机器化状态 | 下一步 |
|--------|------------|--------|
| 绿 | `succeeded` | 关闭节点，触发下游。 |
| 黄 | `needs_input` 或 `succeeded_with_warnings` | 要求补字段或补证据，不重派全单。 |
| 红 | `failed` | 记录失败原因，换工人或重派。 |

**关键保留**：Codex 主脑仍做 **裁决**，不是让脚本自动相信工人。脚本只把“人工复制提示词 + 回写表格”变成稳定文件协议。

### 11.12.6 迁移策略（不推倒重来）

1. **第一步**：保留人工派单，只把每次手写提示词保存成 `job.json`，每次回单保存成 `result.json`。
2. **第二步**：Codex 主脑读取 `result.json` 做红黄绿收单，仍由人确认。
3. **第三步**：对低风险探查单，用 `codex exec` 或薄 CLI 自动读取 `job.json` 并生成 `result.json`。
4. **第四步**：只对“探查单 / 文档整理 / 低风险验证”自动派下一单；涉及实现、生产、账务、鉴权仍由 Codex 主脑与人类裁决。

### 11.12.7 对当前研究路线的修正

- 不再把 `job.json/result.json` 视为新架构，而是 **旧人工派单模式的文件化版本**。
- 旧文中的 **三层闸门、收单四问、执行方自找上下文规则、红黄绿收单** 是协议字段设计的主来源。
- `codex exec` 只是可选执行器；真正要保留的是 **派单治理语义**。

## 11.13 接入已有会话抽取模块：`intermediate.json` 作为 Session 工件来源

> 用户补充：`03-记忆系统-memory/02-存储工程` 下已有 **会话抽取 / 存储工程**。研究结论：当前编排方案不应重新定义一套记忆抽取，而应复用其 **原始会话 → intermediate.json → 热/冷记忆** 管线。

### 11.13.1 现有记忆系统的边界

来自 `03-记忆系统-memory/02-存储工程/总纲.md`：

| 层 | 现有定位 | 对本编排方案的意义 |
|----|----------|--------------------|
| **提取工程** | 从原始会话（Cursor/Qoder 等）抽取成 `intermediate.json` | 对应 §6 的 **Session 外置输入管线** |
| **存储工程** | 决定中间层怎么存、怎么组织 | 对应本方案 `runs/` 与长期 `memory/` 的边界 |
| **热记忆层** | 3 天内会话快照 | Codex 主脑开新窗口时可读的近期上下文 |
| **冷记忆层** | 3 天前归档 | 不进入默认派单上下文，只在需要追溯时读取 |
| **中间层** | `memory/intermediate/YYYY-MM-DD-HHmm-intermediate.json` | 工人/主脑 Session 工件的候选标准格式 |

### 11.13.2 会话路径索引对本方案的价值

来自 `AI助手-会话路径索引.md`：

- 该索引用于 **会话抽取 / 记忆管理**，不是工具可用性探测。
- 现有脚本入口包括：
  - `scripts/02-memory/extract-intermediate.py`
  - `scripts/11-orchestrator/extract-session-digest.ps1`
  - `scripts/02-memory/extract-daily-diary.ps1`
  - `scripts/12-export/extract-claude-code-sessions.js`
- 现有 npm 入口包括：`pnpm run memory:extract` / `memory:extract:cursor` / `memory:extract:qoder`。

**接入判断**：`job.json/result.json` 不必替代 `intermediate.json`。更合理的关系是：

```text
工人原始会话 / CLI stdout / Codex transcript
  -> 会话抽取模块
  -> intermediate.json
  -> result.json 中引用关键摘要 / artifact
  -> memory 热/冷层长期保留
```

### 11.13.3 内容类型全景给出的关键约束

来自 `AI助手-会话内容类型全景.md`：

- 会话内容不只是 user/assistant 文本，还包括：
  - 工具调用：读文件、编辑、命令、搜索、子 Agent 派发、Todo
  - 工具结果：文件内容、命令输出、搜索结果、子 Agent 报告、编辑确认
  - 元数据：模型、token、compaction 摘要
  - 多 Agent 通信
- 当前脚本缺口：把所有消息扁平化为 `role + content`，导致 **动作语义丢失**。

**对本编排方案的硬要求**：

`result.json` 不能只收自然语言摘要；至少要保留：

```json
{
  "actions": [
    { "type": "read_file", "target": "path", "evidence_ref": "..." },
    { "type": "edit_file", "target": "path", "evidence_ref": "..." },
    { "type": "shell", "command": "...", "exit_code": 0, "evidence_ref": "..." }
  ]
}
```

否则主脑无法判断工人到底是 **看了什么、改了什么、验证了什么**。

### 11.13.4 对 `job.json/result.json` 的修正

在 §11.11 草案基础上补两个字段：

**`job.json` 增加：**

```json
{
  "session_extract": {
    "source": "codex | cursor | qwen | claude | codebuddy | raw-cli",
    "expected_intermediate": "runs/<run_id>/nodes/<node_id>/intermediate.json",
    "preserve_action_semantics": true
  }
}
```

**`result.json` 增加：**

```json
{
  "session_artifacts": {
    "intermediate": "intermediate.json",
    "raw_transcript": "stdout.jsonl",
    "digest": "digest.md"
  },
  "actions": [],
  "decisions": [],
  "open_questions": []
}
```

### 11.13.5 与 Codex 主脑的关系

- **Codex 主脑** 不需要把所有历史会话吃进上下文；只读取 **热记忆 / digest / result.json**。
- **工人原始过程** 可先落在 `stdout.jsonl`、Codex transcript 或工具自带会话目录中，再由会话抽取模块转成 `intermediate.json`。
- **长期记忆** 仍走 `03-记忆系统-memory` 的冷热分层，不塞进 `runs/` 目录长期膨胀。

### 11.13.7 对当前路线的修正

- `runs/<run_id>/nodes/<node_id>/result.json` 是 **收单结果**，不是完整记忆库。
- `memory/intermediate/*.json` 是 **会话过程证据**，用于追溯、复盘和后续压缩。
- 因此本机最小实现应拆成两条管线：
  1. **调度管线**：`job.json -> CLI 工人 -> result.json`
  2. **记忆管线**：`raw transcript -> intermediate.json -> 热/冷记忆`

## 11.14 任务上下文到底放哪里：`job.json`、Session、任务单回写的分工

> 结论：**任务上下文不是单一对象**。在当前单机拓扑里，应拆成 **派单上下文、工人运行会话、收单结果、过程证据** 四层。默认不要把完整工人 Session 当成下一步任务上下文。

### 11.14.1 四层上下文

| 层 | 载体 | 谁写 | 谁读 | 用途 |
|----|------|------|------|------|
| **派单上下文** | `job.json` | Codex 主脑 | 工人 CLI | 告诉工人本次要做什么、边界、输入、验收。 |
| **工人运行会话** | 工人进程内上下文；可落 `stdout.jsonl` / transcript | 工人 CLI / Codex exec | 会话抽取模块、排障者 | 记录工人怎么做；不是默认给下游直接吃的上下文。 |
| **收单结果** | `result.json` / 任务单回写 | 工人 CLI | Codex 主脑 | 主脑判断红黄绿、触发下游、汇合结果。 |
| **过程证据** | `intermediate.json` / 热冷记忆 | 会话抽取模块 | 主脑按需追溯 | 复盘、审计、找证据；默认不塞进下一单全文。 |

### 11.14.2 默认规则

1. **进工人 Session 的**：只放完成当前节点所需的上下文，包括 `job.json.prompt`、必要输入文件、上游产物摘要、验收口径。
2. **工人必须回写任务单 / `result.json` 的**：结论、证据、产物路径、验证结果、仍不确定的问题、建议后续任务。
3. **进入下一节点上下文的**：默认只取 **上游 `result.json` + 被主脑选中的 artifact 摘要**，不直接继承上游完整 Session。
4. **进入记忆系统的**：完整过程、原始 transcript、工具调用轨迹，交给 `intermediate.json` 管线保留。

### 11.14.3 为什么不直接把工人 Session 传下去

- 工人 Session 里有大量过程噪声：探索、失败命令、临时猜测、重复输出。
- 直接传完整 Session 会破坏 §9 的 **节点上下文隔离**，让下游继承上游的偏差。
- Codex 主脑才负责 **汇合与裁剪**：它读 `result.json`，必要时追溯 `intermediate.json`，再决定下游 `job.json` 里该带什么。

### 11.14.4 “工人已经回写任务单”是什么意思

这里应理解为：

```text
工人完成节点
  -> 写 result.json
  -> 可选写 artifacts/
  -> 原始会话被抽取为 intermediate.json
  -> Codex 主脑读取 result.json 做收单
  -> 主脑更新 run.json / 任务单状态
```

也就是说，**工人回写的是收单材料，不是全局上下文真源**。全局任务状态仍由 **Codex 主脑 / run 台账** 维护。

### 11.14.5 什么时候才把 Session 内容带入下一单

仅在这些情况：

- 下游必须复用上游的 **工具调用闭环** 或多轮对话状态。
- 上游 `result.json` 证据不足，主脑需要从 `intermediate.json` 抽一段关键原文。
- 任务设计明确要求“接着上一个工人的会话继续推理”。

即便如此，也应由主脑 **摘取片段** 写入下游 `job.json.inputs` 或 `session.messages`，而不是整段 Session 透传。

### 11.14.6 一句话口径

**任务上下文由 Codex 主脑裁剪后写进 `job.json`；工人 Session 是运行过程证据；工人回写 `result.json` 供主脑收单；长期记忆走 `intermediate.json`。**

## 11.15 主脑续跑：别人怎么实现，单机怎么落地

> 问题：工人回写后，**Codex 主脑会停下来问人类“还继续吗”**。如果没有续跑机制，编排会退回人工复制提示词。研究结论：成熟系统都把“暂停/恢复”做成 **显式状态 + 外部事件/信号 + 持久化游标**。

### 11.15.1 外部实现模式

| 模式 | 代表 | 核心机制 | 对本机方案的启发 |
|------|------|----------|------------------|
| **Interrupt / Resume** | LangGraph `interrupt()` / `Command(resume=...)` | 节点中断时保存 graph state；外部带 `thread_id` 恢复；resume 值作为中断点返回值 | `run_id` 就是 thread；`continue_signal.json` 类似 resume payload |
| **Signal / wait_condition** | Temporal Human-in-the-loop | workflow 设置等待状态；外部 signal 注入 approve/reject；历史持久化，崩溃可恢复 | watcher 不做判断，只投递“继续/停止/补参”信号 |
| **Wait for Event** | Inngest `step.waitForEvent()` | 等待匹配事件，带 timeout；事件到达后继续 | `result.json` 文件落地可视为事件；`match` = `run_id/node_id` |
| **CLI Resume** | `codex exec resume` / `codex resume` | 根据 session id 或 `--last` 恢复旧 Codex 会话并追加 prompt | 本机可直接把“收单并继续”作为 resume prompt 喂回 Codex |

### 11.15.2 本机已核实 Codex 续跑能力

本机 `codex-cli 0.130.0`：

- `codex exec resume [SESSION_ID] [PROMPT]`
- `codex exec resume --last [PROMPT]`
- `codex resume [SESSION_ID] [PROMPT]`
- `codex resume --last [PROMPT]`
- `codex resume --include-non-interactive` 可把 non-interactive 会话也纳入选择

差异：

| 命令 | 适合场景 |
|------|----------|
| `codex resume` | 人类回到交互式主脑会话继续裁决 |
| `codex exec resume` | watcher / scheduler 自动把收单结果喂给主脑，让主脑输出下一步 |
| 新开 `codex exec` | 不依赖 Codex 自有 session，只读取 `run.json/result.json` 做一次无头收单 |

### 11.15.3 本机 continuation gate 最小对象

```json
{
  "run_id": "run-20260512-001",
  "gate_id": "gate-after-node-a",
  "trigger": {
    "type": "file_created",
    "path": "runs/run-20260512-001/nodes/node-a/result.json"
  },
  "decision": {
    "mode": "auto_low_risk | ask_human | stop",
    "next_action": "dispatch_next | ask_user | close_run | inspect_intermediate",
    "reason": ""
  },
  "resume": {
    "codex_session_id": null,
    "command": "codex exec resume --last \"读取 run.json 和 node-a/result.json，收单并决定下一步\""
  }
}
```

### 11.15.4 推荐本机落地：watcher 只是闹钟，不当主脑

```text
worker 写 result.json
  -> watcher 检测文件变化
  -> 校验 result.json schema
  -> 更新 run.json 节点状态
  -> 若存在 runnable 下游：
       低风险：调用 codex exec / codex exec resume 做“收单 + 生成下一 job”
       高风险：写 continue_signal.json，状态置为 waiting_human
  -> Codex 主脑读取 run.json/result.json/intermediate 摘要
  -> 主脑裁剪上下文，生成下一个 job.json
```

**关键边界**：watcher 只做机械事：

- 发现文件
- 校验 schema
- 更新状态
- 拉起 Codex

watcher **不**自己决定任务语义、风险裁决、上下文裁剪。那些仍属于 **Codex 主脑**。

### 11.15.5 两种续跑策略

**策略 A：恢复同一个 Codex 主脑会话**

```powershell
codex exec resume --last --json `
  "读取 runs/<run_id>/run.json 与最新 result.json；按任务单规则红黄绿收单；如可继续，生成下一节点 job.json。"
```

优点：主脑能继承之前对话上下文。  
缺点：容易再次把大量历史带入上下文；与“外置台账为真源”有双记风险。

**策略 B：每次新开 Codex 收单**

```powershell
codex exec --ephemeral --json `
  --output-schema schemas/continuation-decision.schema.json `
  "只读取 runs/<run_id>/run.json、最新 result.json 和必要 intermediate 摘要，决定下一步。"
```

优点：更接近无状态；`run.json` 是唯一真源。  
缺点：需要 `run.json` 写得足够完整。

**当前推荐**：先用 **策略 B**。只有需要保留主脑长程判断时，才用 **策略 A**。

### 11.15.6 状态机补充

在 §11.11 节点状态上增加图级等待状态：

```text
running
  -> succeeded
  -> failed
  -> needs_input

succeeded
  -> awaiting_continuation
  -> dispatching_next
  -> closed

awaiting_continuation
  -> auto_continue
  -> waiting_human
  -> stopped
```

### 11.15.7 一句话口径

**Session 外置解决“记得住”；任务单回写解决“交了什么”；continuation gate 解决“主脑什么时候、带什么上下文、以什么方式继续”。**

## 12. 【附录·已冻结】编排延展研究：耐久执行 · 人机闸门 · 交接与委托（外查归纳）

> 接续 §11 外查笔记；见上文「附录 A」说明。

### 12.1 耐久执行（Durable Execution）与「主脑 / Worker」边界

**在说什么**：长历程、多远程调用、易失败的 agent 流程需要 **跨崩溃可恢复的编排状态**。[Temporal 等](https://www.temporal.io/blog/durable-execution-meets-ai-why-temporal-is-the-perfect-foundation-for-ai) 将 **工作流代码（workflow）** 与 **活动（activity）** 分离：**工作流侧要求确定性**（便于事件溯源重放），**活动侧可做非确定工作**（LLM、HTTP、工具）。这在概念上 **贴近** 你们的划分：  
- **编排器 / 图推进**（何时派哪一节、等待哪个 Join）→ 宜 **耐久、可重放**；  
- **Worker 双手**（调模型、跑沙箱）→ **非确定、可重试**，结果写入 Session 或节点输出工件。

**对方法论的对照（研究笔记，非需求变更）**：

- 若未来 run 时长跨小时/天、要强韧恢复，需在 **§4 编排器** 实现策略里额外考虑：是 **自研轻量状态机** 还是 **挂耐久执行引擎**；二者都改变运维面，但不改变「Coordinator 不代做工件细节」的原则。
- **风险**：把 LLM 计划 **写进** 必须确定重放的 workflow 代码路径时，要克制（随机性、模型升级都会导致重放不一致）；实践中常把 **「计划」当 activity 产物持久化**，workflow 只读 **已持久化的计划版本**。

**入口**：[Temporal：Durable Execution meets AI](https://www.temporal.io/blog/durable-execution-meets-ai-why-temporal-is-the-perfect-foundation-for-ai)、[Durable AI Agent 教程](https://learn.temporal.io/tutorials/ai/durable-ai-agent/)。

### 12.2 人机闸门（Human-in-the-Loop, HITL）

**在说什么**：在关键动作前 **挂起 Run**，外存 **待审批上下文**，人 **批准/拒绝/改参** 后再 **恢复**（[Cloudflare Workflows `waitForApproval` 模式叙述](https://developers.cloudflare.com/agents/guides/human-in-the-loop/)；多篇产品文讨论 **审批疲劳** 与 **按风险分级**）。

**与 DAG 节点流转的合成方式（概念）**：

- 某节点状态不仅是 `running`，还可进入 **`waiting_human`**（或等价），退出条件为 **外部信号 + 超时策略**；Join 之前仍可并行跑 **不依赖** 该闸门输出的分支。
- 持久化需含：**工作流快照 / 触发操作说明 / 可修改参数界面模型**（参见 [Athenic 等对审批队列字段的总结](https://getathenic.com/blog/human-in-the-loop-approval-workflows)）——对应你们 **编排平面** 与 **Session 平面** 中的哪几列，需在 PRD 里定，不在此代定。

**入口**：[Cloudflare Agents HITL](https://developers.cloudflare.com/agents/guides/human-in-the-loop/)、[Brightlume：checkpoint 设计](https://brightlume.ai/blog/building-human-in-the-loop-checkpoints-agentic-systems)、[Particula：何时要审批](https://particula.tech/blog/human-in-the-loop-ai-agent-approval)。

### 12.3 Handoff（交接）vs Agent-as-Tool（专员作工具）

**在说什么**：[OpenAI Agents 编排指南](https://developers.openai.com/api/docs/guides/agents/orchestration) 区分：**交接** = 专员 **接管后续对话**；**作工具** = 经理 **保留合成权**，专员只做有界子任务。[Microsoft Agent Framework：Handoff 模式](https://devblogs.microsoft.com/agent-framework/a-tour-of-handoff-orchestration-pattern/) 强调 **显式图拓扑**、**所有权迁移**、**上下文共享策略**。

**与方法论名词的映射（建议性的研究对齐）**：

| 外部 | 与你们可能的读法 |
|------|------------------|
| Manager + specialist **as tool** | **编排器派活**，Worker 交 **结构化结果** 回 Coordinator；Session 可按 **每 Worker** 隔离（与 §9 一致） |
| **Handoff**（会话所有权转移） | 接近「**下游节点继承上游整段会话**」的产品选项——与 §9 **默认不共享长线程** 可能 **冲突**，若采用须在《方法论》里 **显式升格为允许的模式**（目前未代你写） |
| **Sub-agent 委托**（[模式文](https://www.inngest.com/blog/three-patterns-you-need-for-agentic-systems)） | 常分 **同步等待 / 异步报告 / 定时** 三种——对应 DAG 边上 **同步边（Join 前必须完成）** vs **fire-and-forget（即发即弃，会削弱汇合语义，少见）** |

**设计提示（研究级）**：Handoff 容易把 **Session 边界搅浑**；若主线坚持 **DAG + 分列持久化**，更自然的是 **专员作工具 + Coordinator 汇合**，Handoff 仅作 **明确产品线**。

### 12.4 研究结论摘录（仍只进任务单）

- **耐久执行** 给了一种 **「编排状态机如何实现」** 的成熟路径，与 **蜂群 Worker** 不矛盾：死的可以是 **计算实例**，活的应是 **外置 run 状态 + Session 工件**。
- **HITL** 应建模为 **显式节点/显式悬挂状态**，避免「人机交互」散落在 chat UI 里却不进图状态。
- **Handoff vs 工具化专员** 是产品级分岔；与 §9 **消息层隔离** 的默认值关系最大，实施前应用 **主脑**（你们人）拍板，而非研究助手默认。

### 13.1 Saga 与补偿（长流程一致性）

**在说什么**：长历程、多副作用步骤下，**分布式事务**往往不可行；**Saga** 用 **前向步骤 + 失败时的补偿步骤** 维持 **最终一致**。常见两式：**编排（orchestration）**——中心化协调器驱动并可显式回滚；**编舞（choreography）**——各参与方靠事件自决（扩展与调试更难）。AWS 将 **Saga 编排** 列为 agentic AI 参考模式之一（[Saga orchestration patterns（AWS Prescriptive Guidance）](https://docs.aws.amazon.com/prescriptive-guidance/latest/agentic-ai-patterns/saga-orchestration-patterns.html)）。

**与你们的 Coordinator / DAG 的对照（研究笔记）**：

- **Join 成功后** 若下游失败，需定义：**哪些上游副作用可逆**、**不可逆时是否仅记录审计+人工介入**（补偿常 **不是** 数学逆操作，例如已发邮件不能撤回，只能发更正）。
- **幂等** 与 **稳定业务键**：补偿与重试安全依赖 **§10** 里「读码不足、需线上验证」的一类，但 **设计阶段** 就要在编排平面预留 **saga_id / step_id**。
- 社区文常把 **Temporal activities + 显式 compensate** 与 Claude/LLM 工作流拼在一起（示例：[Claude Lab：Temporal × Saga 叙述](https://claudelab.net/en/articles/api-sdk/claude-agent-sdk-temporal-durable-ai-workflows-production-guide)；[Restate：rollback 模式](https://docs.restate.dev/ai/patterns/rollback)；[Cordum：rollback/compensation 综述](https://cordum.io/blog/ai-agent-rollback-compensation)）。

**研究结论（不进《方法论》正文）**：若产品触及 **写库 / 对公发信 / 扣款**，应在 PRD 层要求 **每类副作用登记补偿策略**；纯「读+生成草稿」可弱化 Saga，但 **仍建议** 统一 run 级 **失败语义**（重跑、跳过、dead-letter）。

### 13.2 可观测性、追踪与「血缘」

**在说什么**：**OpenTelemetry** 正成为 **LLM/agent 调用链** 的通用发射层；后端可用 Langfuse 等接 **OTLP**（见 [Langfuse：OpenTelemetry](https://langfuse.com/docs/opentelemetry)）。Agent 观测强调：**多跳 LLM 调用、工具调用、路由决策、护栏** 等 **结构化 span**（[Langfuse：agent observability 概述](https://langfuse.com/blog/2024-07-ai-agent-observability-with-langfuse)）。

**与「血缘」的关系（术语建议）**：

- **Trace（追踪）**：一次用户请求或一次 **Run** 内的跨度树；应用把 **`run_id` / `dag_node_id` / `worker_session_id`** 写入 span attribute，才能从 Jaeger 类 UI **穿回** 你们的图执行。
- **Lineage（血缘）**：在 ML/数据域常指 **数据从哪来**；在编排里可 **扩用** 为 **「产物由哪些节点版本 + 哪些上游 artifact 推导」**——实现上常是 **Run 元数据 + 节点输出哈希 + 模型版本** 的表，不一定等同于 OTEL，但应 **可关联** 到同一 `trace_id`。

**实操要点（研究级）**：**trace 上下文要在 Coordinator 派生活动时传播**；否则会看到「许多断 span」无法拼 DAG。

### 13.3 Agent2Agent（A2A）协议与「黑箱工人」

**在说什么**：**A2A**（协议站点：[A2A Specification](https://a2aproject.github.io/A2A/v0.2.5/specification/)，仓库线索：[a2aproject/A2A](https://github.com/a2aproject/A2A)）定义 **异构 agent 系统** 之间的互操作：**Agent Card**（能力元数据）、**Task** 生命周期、**Message / Artifact**，强调 **不透明执行**（协作只看对外声明，不看对方内存/工具实现）。

**与方法论的对照**：

- **Worker** 可类比为 **对内** 接模型与沙箱；**对外** 若要以 **标准协议** 被其他组织调用，A2A 提供 **「黑箱 + 能力声明 + 任务约定」** 的词汇——接近 §2 **可替换执行单元** 的 **跨团队** 推广版。
- **Coordinator** 若将来要 **联邦调度**（多租户、多厂商 agent），A2A 的 **Task/Message** 与你们的 **DAG 节点 I/O** 之间 **仍需一层映射**（研究未代定字段级映射）。

### 13.4 本节研究结论摘录

- **Saga**：把 **失败语义** 从产品 bug 提升为 **一等设计**；与 **DAG** 天然兼容（每边一步可挂 compensate）。
- **OTEL**：实现 **可调试编排** 的「最低成本公交通道」；**血缘** 建议 **Run/节点/artifact 表 + trace 关联**，而非只靠日志。
- **A2A**：当 **Worker 需对外开放** 时的 **互操作参考面**；不实现协议也不妨碍 **内部** Coordinator–Worker。

## 14. 【附录·已冻结】编排延展研究（续）：配额/成本 · 多租户隔离 · 护栏与策略

> 接续 §13；角色与证据强度约定相同。检索：2026-05-12。

### 14.1 配额、成本与「长尾」消耗

**在说什么**：多租户共用模型能力时，**按请求限流** 往往失真——**TPM（tokens per minute）** 与 **input/output 分拆** 更贴近稀缺资源；Agent **多步** 会放大 **重尾**（同一条业务路径可从数千 token 到数十万 token）。实践上常见：**令牌桶（按租户）**、**分层预算**、**软硬双阈值**、**优先级队列**（见 [DEV：Multi-Tenant Token Budgets](https://dev.to/gabrielanhaia/multi-tenant-token-budgets-quota-patterns-that-dont-starve-your-best-customers-2op6)）；网关注入 **Org / Team / User / virtual key** 多级预算（多篇架构文，如 [tianpan：multi-tenant LLM gateway](https://tianpan.co/blog/2026-04-09-multi-tenant-llm-api-gateway-production)、[noisy neighbor / tenancy](https://tianpan.co/blog/2026-04-16-noisy-neighbor-llm-shared-infrastructure-tenancy)）。

**与 Coordinator / DAG 的对照（研究笔记）**：

- **预算应挂在谁的「边」上**：既可 **per-tenant 全局**，也可 **per-run、per-node**（贵模型节点单独 bucket）——后者与 §9 **异构模型** 一致。
- **Join 汇合前**：并行扇出可能造成 **瞬时 TPM 叠加**；调度器需要 **预见性**（headroom）而不只是事后计费。
- **编排平面** 应持久化 **已计量消耗**（或引用计费系统 id），否则 **§13 血缘** 与 **财务对账** 对不上。

### 14.2 多租户隔离（身份 · 数据 · 工具 · 缓存）

**在说什么**：租户隔离要 **每层** 显式化（DB、队列、缓存、向量库、webhook）；**RLS** 等机制依赖 **请求全程携带 tenant 上下文**，缺失应 **硬失败**（[Zylos：multi-tenant security patterns](https://zylos.ai/research/2026-02-23-multi-tenant-security-patterns-saas-platforms)；[Callsphere：isolating customers](https://callsphere.ai/blog/multi-tenant-ai-agent-platform-isolation-billing.md)；[Agent Patterns：multi-tenant](https://www.agentpatterns.tech/en/architecture/multi-tenant)）。

**Agent 特有风险**（综述性）：**RAG/向量检索** 交错、**语义缓存** 串租户、**工具 OAuth** 未按租户切 credential、**prompt 注入** 扩大越权面（[Blaxel：isolation guide](https://blaxel.ai/blog/multi-tenant-isolation-ai-agents)；[Scalekit：access control](https://www.scalekit.com/blog/access-control-multi-tenant-ai-agents)）。

**与方法论的对照**：

- **Session 外置（§6）**：存储与索引必须 **带 tenant_id** 且与 **编排 run** 同源注入，避免「恢复了别人的会话」。
- **Worker 参数化（§8）**：**凭据、工作目录、向量集合名** 等应是 **租户范围闭包** 的一部分，而不是全局常量。
- **蜂群模式（§3）**：实例可丢，但 **租户上下文** 必须从 **不可伪造** 的控制面注入（JWT / mTLS / signed task envelope），不能靠 Worker 自建猜测。

### 14.3 护栏（Guardrails）与「策略即配置」

**在说什么**：生产系统常把 **输入/输出过滤、敏感数据、越权工具调用** 等变成 **可版本化策略**：如 **NeMo Guardrails**（配置/Colang 驱动，见 [官方介绍](https://docs.nvidia.com/nemo/guardrails/introduction.html)）、**Lakera Guard**（policy 集中管理项目与灵敏度，见 [Policies 文档](https://docs.lakera.ai/docs/policies)）。**要点**是：护栏产生 **阻断 / 降敏 / 升级人工（可与 §12.2 汇合）** 等 **显式 outcome**，便于审计。

**与编排的对照（研究级）**：

- **放置层**：可在 **网关**（进模型前）、**Coordinator 派活前**（按节点风险 profile）、**Worker 出工具前**（最小权限）多层重复；需定义 **哪一层为权威**，避免重复矛盾。
- **与 DAG**：高危节点（对外写、对公通信）可 **默认绑定更严 policy**；与 §14.2 **租户策略** 组合时，注意 **policy 版本** 也应进 **血缘（§13.2）** 以便解释「为何当时被拒」。

### 14.4 本节研究结论摘录

- **配额**：以 **token 与重尾** 为第一性原理；**DAG 并行** 会叠峰值，调度与预算要联动。
- **多租户**：**租户上下文** 是 **贯穿编排、Session、工具、检索** 的横切 concern，不是数据层补丁。
- **护栏**：宜 **策略化、可版本、可观测**；与 **HITL**、**Saga** 一起构成「风险三角」的三条独立轴。

## 15. 【附录·已冻结】编排延展研究（续）：隐私/PII · 模型路由与降级 · 评测与回归

> 接续 §14；角色与证据强度约定相同。检索：2026-05-12。

### 15.1 隐私、PII 与数据最小化

**在说什么**：Agent **多步** 会把敏感字段 **摊进** 提示、工具入参、日志与缓存；合规上常需 **按步骤声明「本步所需字段」**，在 **步骤边界** 剥离多余 PII（能力/最小权限思想），并打 **审计事件**（见 [Trezalabs：step-boundary redaction](https://www.trezalabs.com/blog/redacting-pii-in-agentic-systems-least-privilege-data-access)）。检测常见 **双层**：规则（结构化 PII）+ 语义模型（上下文型 PII）（[Callsphere：agent pipelines](https://callsphere.ai/blog/pii-detection-redaction-ai-agent-pipelines.md)）。**Mask & Rehydrate**：进模型前 **占位符化**，推理在脱敏载荷上完成，**出安全区调真 API 前** 再 **还原**（[IBM Community：mask & rehydrate](https://community.ibm.com/community/user/blogs/mohammed-ali-shaik/2026/03/25/protecting-pii-in-agentic-frameworks-the-mask-rehy)）。企业文档亦归纳 **入库 / skill / 指令** 三层干预（[Workato：anonymization patterns](https://docs.workato.com/en/agentic/agent-studio/genie-governance/pii-anonymization-patterns.html)）。

**与方法论的对照（研究笔记）**：

- **DAG 节点边界** 天然可作为 **PII 最小化的切口**（与 §9「节点任务边界清晰」一致）：每节点 **入参 schema** 声明可携带字段；**Join 汇合** 时只提升 **下游需要的子集**。
- **Session 外置（§6）**：持久化里若存原文，需 **与租户加密域、保留期** 策略一致；**血缘（§13.2）** 里对「曾出现 PII 的 artifact」打标，避免日志查询二次扩散。
- **蜂群 Worker（§3）**：进程 disposable **不意味** 内存里短期持有 PII 无风险——**崩溃 dump、核心转储、排障包** 仍是泄露面。

### 15.2 模型路由、级联（Cascade）与故障降级

**在说什么**：**Routing** = 按任务类型/复杂度选模型；**Cascading** = 先小后大、达标即停；**Cascade routing** 试图统一二者（学术预印本：[arXiv:2410.10347](https://arxiv.org/abs/2410.10347)）。工程上常见 **置信度/质量校验**（结构化输出、启发式、**法官模型**）触发 **升级到更强模型**；**429/503** 等可触发 **链式 fallback**（产品叙述示例：[E.D.D.I model cascade](https://docs.labs.ai/agent-configuration/model-cascade)、[cascadeflow 文档](https://docs.cascadeflow.ai/get-started/how-it-works)、[Microsoft Foundry model router](https://learn.microsoft.com/en-us/azure/foundry/openai/concepts/model-router-how-it-works)）。

**与 §9 异构模型的关系**：

- 方法论已允许 **按节点选规格**；本节补充 **运行时策略**：同一节点可配置 **主模型 + 有序备选**（latency/cost/可用性维度），须在 **编排平面** 记录 **实际用哪一档**（否则成本与复现对不上）。
- **并行扇出（Join 前）** 若各分支 **独立 escalation**，可能破坏 **可比较性**（同题不同分支走了不同模型）；是否需要 **统一路由器在 Coordinator** 决策，属产品权衡。

### 15.3 离线评测、黄金集与 CI 门禁

**在说什么**：多步 agent 评测常同时看管：**任务完成率**、**工具选择与参数**、**输出质量**（LLM-as-judge / 人评）、**故障恢复**（综述性归纳见 [Inductivee](https://inductivee.com/blog/ai-agent-evaluation-testing-framework)、[AI Vineet 框架文](https://aivineet.com/agent-evaluation-framework-test-llm-agents/)、[Braintrust 实践框架](http://www.braintrust.dev/articles/ai-agent-evaluation-framework)）。生产回归依赖 **黄金数据集**（规模级几十到数百真实分布用例）+ **统计阈值门禁**（降幅超阈即拦发布）（[Callsphere：agentic CI/CD](https://callsphere.ai/blog/agentic-ai-cicd-github-actions-automated-agent-testing.md)）。流水线常含：**静态检查（schema/prompt）** → **对话回归** → **质量打分** → **金丝雀**。

**与编排的对照**：

- **应对 DAG 分级测**：**单元** 测单节点 I/O；**集成** 测 **Join/扇出** 与 **Coordinator 状态迁移**（否则只在 chat 层测会漏 **并发竞态**）。
- **非决定论**：门禁看 **分布**（多次采样/N 次运行置信区间），不是单次 `assert`；**模型版本** 必进 **实验元数据**（与 §13 血缘衔接）。

### 15.4 本节研究结论摘录

- **PII**：以 **步骤边界 + 最小权限字段** 为主轴；Mask/Rehydrate 解决「模型要看结构、工具要真值」的张力。
- **路由**：把 **fallback 决策** 和 **实际选中模型** 纳入 **编排可观测与成本**；小心 Join 下 **分支独立升级** 带来的 **语义漂移**。
- **评测**：**DAG 级** 集成测与 **统计门禁** 是方法论落地后的 **质量保险丝**，与 §2「补丁贬值」互补——**靠数据集锁行为**，而非堆 harness。

## 16. 【附录·已冻结】编排延展研究（续）：长上下文与记忆分层 · 工具身份/OAuth · 多区域与灾备

> 接续 §15；角色与证据强度约定相同。检索：2026-05-12。

### 16.1 长程上下文：压缩、分页式记忆与索引回取

**在说什么**：固定 **上下文窗口** 与 **长历程轨迹**（多轮工具、观测）矛盾。常见工程模式：**滑动截断**、**周期摘要**、**分层记忆**（热/温/冷；原始轮次 / 段摘要 / 主题摘要）。研究/产品向思路包括：类 OS **虚拟内存分页** 的上下文管理（[Virtual Context 论文页](https://virtual-context.com/paper/)）、**Acon**（针对长程 agent 的压缩指引优化，[arXiv:2510.00615](https://arxiv.org/abs/2510.00615)）、**自适应压缩**（[arXiv:2603.29193](https://arxiv.org/abs/2603.29193)）、**索引化经验库 Memex**（工作集精简 + 外部库按需 **dereference** 全保真证据，[arXiv:2603.04257](https://arxiv.org/abs/2603.04257)）。工程笔记类综述：[working memory compression / distillation](https://notes.muthu.co/2026/03/working-memory-compression-and-context-distillation-in-long-horizon-agents/)。

**与方法论的对照（研究笔记）**：

- **§4「编排抗上下文衰减」**：压缩策略应明确 **谁负责摘要**——**Coordinator**（跨节点全局简报）vs **各 Worker Session**（局部轨迹）；混用时要防止 **双头摘要互相矛盾**。
- **§6 外置 Session**：「热上下文在内存、冷证据在库」与 **蜂群 Worker** 一致：**进程可死**，但 **可回取索引** 必须落在 **持久化层**；否则摘要不可审计。
- **§9 DAG**：Join 前各分支若都做了 **本地截断**, 汇合时可能 **丢失可比对细节**——需要 **显式「提升」字段**（哪些进 Join payload）。

### 16.2 工具身份：OAuth/委托、刷新令牌与持续评估

**在说什么**：Agent **自主调工具** 打破「浏览器里有活人」假设，常见 **用户委托 → 范围化 agent 令牌**；风险在 **长期 refresh token**、**跨租户串用**、**超范围调用**（[Scalekit：tool calling authentication](https://www.scalekit.com/blog/tool-calling-authentication-ai-agents)）。建议族：**短 TTL access**、**按工具按需取凭证**、**scopes 只缩不放**、**全链审计 id**（[SuperTokens：auth for agents](https://supertokens.com/blog/auth-for-ai-agents)）；刷新令牌宜 **等同 Tier-0 秘钥** 管理（[Learn IAM：refresh tokens](https://learn-iam.com/blog/refresh-tokens-for-ai-agents)）。进阶：**SSF/CAEP** 一类 **持续会话评估** 做事件型吊销（[Learn IAM：OAuth abuse / CAEP](https://learn-iam.com/blog/securing-agentic-ai-oauth-token-abuse-caep)）；OAuth 2.1 向 **常驻公共客户端** 约束演进的产品讨论见 [LoginRadius](https://www.loginradius.com/blog/engineering/auth-for-ai-agents)。

**与方法论的对照**：

- **§8 Worker 参数化**：**凭据** 应是 **短时 lease + 可吊销**，挂在 **tenant × tool × run** 上，不写进 **长寿命 Session 快照** 明文区（可存 **引用句柄**）。
- **§12 HITL**：高危工具可在 **令牌 issuance** 或 **每次调用** 时再要 **人机确认**。
- **§14.2**：OAuth **回调与 token 仓储** 是 **租户隔离** 高危区，与 RAG 并列排查。

### 16.3 多区域、灾备与编排状态复制

**在说什么**：长历程 **工作流/编排** 常见 **异地冗余**：**active-active**（多活，强调吞吐与分区） vs **active-passive**（热备，强调冲突与版本）（Camunda/Zeebe 叙述：[geo-redundancy with Zeebe](https://camunda.com/blog/2022/06/how-to-achieve-geo-redundancy-with-zeebe/)）。Cadence/Temporal 系文档讨论 **跨 DC 复制、故障切换语义、至少一次 activity、事件序版本**（[Cadence cross-DC replication](https://cadenceworkflow.io/docs/concepts/cross-dc-replication)）。Temporal 商业/平台文档中的 **多区域 namespace / 复制类型** 见 [Temporal：multi-cloud / replication 能力页](https://docs.temporal.io/evaluate/development-production-features/multi-cloud-namespace)。Akka 等 **双写/主区域** 模型可作为对照（[Akka multi-region](https://doc.akka.io/snapshots/akka-documentation/operations/regions/setup.html)）。

**与方法论的对照（研究级）**：

- **编排平面（§4）** 与 **Session 平面（§6）** 的复制策略 **不必相同**：或可接受 **编排 RPO 更严、Session blob 异步复制**，但 ** failover 后** 必须定义：**正在跑的 Worker** 与 **半成品 Join** 如何裁决（避免双提交流）。
- **§13 Saga**：跨区域时 **补偿** 的 **观察可见性** 与 **重复投递** 概率上升；幂等与 **全局 id** 更要硬。
- 若不自研而 **托管编排**，应把 **RPO/RTO、脑裂行为、activity 至少一次** 写进 **供应商尽职调查**，而非当实现细节。

### 16.4 本节研究结论摘录

- **长上下文**：**外置 + 分层 + 可回取索引** 与 **无状态 Worker** 同向；**摘要与 Join 提升字段** 要在产品层规定，否则丢证据不可复盘。
- **OAuth**：把 **refresh** 当高危面；**短活 access + 强审计 + 持续吊销**；凭证与 **Session 快照** 分离。
- **多区域**：**双平面状态** 各自定义 **RPO/RTO**；**Join/Saga** 在切换下易出错，需 **确定性 Run id** 与 **幂等** 打底。

## 17. 【附录·已冻结】编排延展研究（续）：Prompt/策略版本化与发布 · 混沌与演练 · 保留与删除

> 接续 §16；角色与证据强度约定相同。检索：2026-05-12。

### 17.1 Prompt / 策略注册表、不可变版本与别名发布

**在说什么**：生产上倾向 **集中注册表** + **不可变版本**（版本一经发布不原地改）+ **别名/标签**（`production`、`staging`）解耦「代码部署」与「策略内容部署」（[MLflow：prompt lifecycle / aliases](https://mlflow.org/docs/latest/genai/prompt-registry/manage-prompt-lifecycles-with-aliases/)、[PromptLayer：registry](https://docs.promptlayer.com/features/prompt-registry)）。实践文讨论 **变更与评测挂钩**、回滚路径（[LangWatch：prompt management](https://langwatch.ai/blog/what-is-prompt-management-and-how-to-version-control-deploy-prompts-in-productions)、综述 [MyEngineeringPath](https://myengineeringpath.dev/genai-engineer/prompt-management/)）。**特性开关/金丝雀**：检索未给独立权威 spec，工程中常 **借用** 通用 feature flag 与 **按租户/百分比** 路由到不同 **prompt_version**；与 §15 **评测门禁** 联动（新 prompt 先离线过闸再上别名）。

**与方法论的对照（研究笔记）**：

- **Agent 模板（§5）** 应能解析为 **versioned manifest**（模型 id、system、工具 schema、护栏策略 id），与 **Coordinator** 记录的 **run 元数据** 一起进 **血缘（§13.2）**。
- **多 Worker（§8）** 可 **同图不同节点** 指向不同 **prompt 别名**（例如「廉价草拟」vs「昂贵定稿」），回滚时 **按节点别名回切** 而非全系统一键。
- **编排器（§4）** 若不记录 **当时策略版本**，事后无法解释 **为何调度结果突变**——与审计/合规（§17.3）冲突。

### 17.2 混沌工程、Game Day 与灾备演练

**在说什么**：**Game Day / Chaos Day** = 受控故障演练，测 **人+流程+系统**（[AWS：build your own game day](https://aws.amazon.com/blogs/architecture/build-your-own-game-day-to-support-operational-resilience/)；[Harness：gamedays](http://developer.harness.io/docs/chaos-engineering/guides/gamedays)；规划文 [Gremlin：planning chaos day](https://www.gremlin.com/community/tutorials/planning-your-own-chaos-day)）。**混沌工作流**（如 Chaos Mesh）可把 **串并行故障场景** 编排成实验（[Chaos Mesh intro](https://mintlify.com/chaos-mesh/chaos-mesh/introduction)）。DR drill 常见：声明 **RTO/RPO 置信度**、跑 **模拟切换**（[GitLab Runbooks：gameday](https://runbooks.gitlab.com/disaster-recovery/gameday/)）。

**与方法论 / §12–§16 的合成视角**：

- 对 **Coordinator**：注入 **API 429/503、队列滞后、Join 超时**；观察 **重试风暴、积压、成本**.
- 对 **Worker / Session**：实例 **SIGKILL**；验证 **外置恢复** 是否丢 **tool_use_id** 对齐（§6）。
- 对 **多区域（§16.3）**：分区网络、主备切换；验证 **半成品 Join** 与 **Saga** 是否 **双提交或悬挂**.
- **HITL（§12）**：演练 **审批 SLA 超时** 默认策略（拒绝？升级？）。

### 17.3 保留期限、删除权与「记忆黑箱」

**在说什么**：企业产品常见 **可配置保留期**（如对话 30 天–数年档）、到期批量删除或 **按需删除 API**（示例文档：[Agent Red：retention](https://agentredcx.com/docs/admin-guide/data-retention)、[Ada：data retention](https://docs.ada.cx/docs/other/data-retention)）。GDPR **删除/反对处理** 的工程叙述见 [Oracle Fusion：chat transcript deletion](https://docs.oracle.com/en/cloud/saas/fusion-service/fairs/enable-deletion-of-chat-transcripts-to-comply-with-general-data.html)、综述 [Kronvex：GDPR & AI agents](https://kronvex.io/blog-gdpr-ai-agents)。**硬问题**：agent **自主写入** 的非结构化记忆若 **未打点数据主体 id**，会导致 **删不干净**（[Astraea：memory tool privacy](https://astraea.law/insights/ai-agent-memory-tool-privacy-compliance)）。

**与方法论的对照**：

- **Session 工件 + 向量库 + 工具侧写文件** 必须共用 **subject_id / tenant_id 索引**；删除请求走 **级联清单**（编排 run、span、artifact、blob）。
- **§15.1 PII** 与 **§16.1 摘要**：压缩后的 **摘要** 仍可能载有可识别信息；**删除** 须 **穿透** 各层（热上下文、冷存储、嵌入缓存）。
- **血缘与审计**：法规常要求 **保留一定期限的处置证明**——**物理删** 与 **审计日志保留** 可能 **同时** 需要产品设计。

### 17.4 本节研究结论摘录

- **版本化**：把 **模板/提示/护栏** 当 **发布物**；**别名切换** = 低风险回滚；与 **评测 + 金丝雀** 同源。
- **演练**：Agent 编排的薄弱点常在 **Join、HITL 超时、跨区切换、幂等**——Game Day 应用 **剧本化故障** 打穿这些点。
- **删除**：**可删性前置**——任何 **自主记忆写路径** 若不能 **列举**，合规上即高风险（「不知道存了啥就删不掉」）。

# Paperclip Canonical Operating Model

## Status

FOUNDATIONAL DOCUMENT

This document defines the intended architecture and operating philosophy for all Paperclip deployments.

Before modifying Paperclip, agents and developers MUST read and understand this document.

---

# Core Understanding

Paperclip is not a blank framework requiring a custom UI to be built.

Paperclip already provides the operational interface and organizational primitives.

The existing Paperclip platform includes:

- companies
- agents
- projects
- goals
- routines
- issues
- inbox workflows
- comments
- artifacts
- skills
- model connections
- agent configuration
- operational dashboards

The correct approach is:

## Understand first.
## Configure second.
## Extend only when required.

Do NOT rebuild existing Paperclip capabilities.

---

# Paperclip Mental Model

Paperclip is the operating system for AI organizations.

Paperclip provides:

- workforce management
- agent lifecycle
- organizational structure
- workflow execution
- communication
- task management


Paperclip does NOT replace governance.

---

# System Layers

The complete architecture consists of three layers.

---

## Layer 1 — Paperclip

Operational execution layer.

Responsible for:

- Companies

- Skills

- confidence

- audit trails
- permissions

Knowledge relationship layer.

# Company Isolation Principle

A Paperclip installation may contain multiple companies.

However, companies are operational boundaries.

Example:Paperclip Instance

|
+-- QuantumShield Labs
|
+-- TheBinMap
|
+-- Client Organization

Each company has:

- independent agents
- approved skills
- scoped memory
- controlled tools
- defined permissions

Never mix experimental, client, or unrelated company data.

---

# Clean Instance Rule

Legacy experiments are not production foundations.

Old:

- companies
- agents
- issues
- projects
- conversations

must be treated as historical artifacts.

Production instances should begin from a clean operational state.

---

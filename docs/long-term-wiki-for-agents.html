Long-Term Wiki for Agents

1. Problem

Pain Point

Agents are amnesiac

Every heartbeat run starts from scratch. Conventions learned in run N are forgotten by run N+1. The agent re-reads the same files, re-discovers the same patterns, re-asks the same questions.

Pain Point

Context window is the only memory

Session resume only carries the adapter's internal transcript. There is no structured, queryable, agent-authored place for durable facts like "the repo uses pnpm workspaces" or "DB migrations live in packages/db".

Pain Point

No cross-session knowledge transfer

When an agent is resumed hours or days later, operationally learned state (ongoing PRs, broken fixtures, who-owns-what) is gone. Humans end up re-priming agents manually in comments.

Pain Point

Token budget waste

Re-exploring the same codebase every run burns tokens that should be spent on the actual task. The marginal cost of each run stays flat instead of trending down as the agent "learns" the repo.

2. Goals & Non-Goals

Goals

  • Per-agent persistent knowledge store that survives across heartbeats.
  • Zero DB migrations — file-based, living alongside the agent's home directory.
  • Automatic injection into every run's system context.
  • Agent-authored updates — the agent decides what is worth remembering.
  • Opt-out via runtimeConfig.memory.enabled: false; enabled by default.
  • Non-fatal failures — wiki errors must never break a run.
  • Browseable from the UI (new Wiki tab on agent detail).

Non-Goals

  • Cross-agent shared knowledge graph (that's a future RAG layer).
  • Full-text search / vector index (markdown + path-based for now).
  • Versioning / diff history of wiki pages (filesystem is the source of truth).
  • Fine-grained permissions on individual pages.
  • Replacing CLAUDE.md / project docs (those are human-authored, project-wide).

3. Solution at a Glance

Every agent gets a directory: {agentHome}/wiki/. Markdown pages live inside; an auto-rebuilt index.json is the manifest. On every heartbeat, the server loads the wiki and injects a formatted bundle into the adapter prompt. After the run completes, a second adapter invocation resumes the same session with MCP write/delete tools exposed — letting the agent synthesize what it just learned into durable pages.

4.2 Storage layout on disk

{agentHome}/
├── wiki/
│   ├── index.json          # manifest: [{ path, title, updatedAt, bytes }]
│   ├── conventions.md      # agent-authored markdown pages
│   ├── ongoing-work.md
│   └── subdir/
│       └── db-schema.md    # nested paths are allowed
└── ...other adapter state
  • index.json is auto-rebuilt by the service whenever pages change — never trust it, always re-scan when in doubt.
  • Paths are normalized and validated — .. traversal, absolute paths, and symlink escapes are rejected.
  • A single agent wiki is capped for sanity (page count + per-page byte limit enforced in services/agent-wiki.ts).

5. V1 — Naive Approach reverted

The first commit (2c76f301) tried to capture wiki updates by parsing a wikiUpdates field from the adapter's structured resultJson.

V1 — resultJson capture (did not work)

Why it failed

Structured-output adapters (Claude, Codex) don't let the agent inject arbitrary top-level fields into resultJson. The shape is dictated by the adapter's own tool contract, so wikiUpdates was always undefined. Updates never landed.

6. V2 — Adapter Synthesis shipped

Commit a62eab16 replaces the broken capture with a second adapter invocation that resumes the same session. The agent still has the full run context, but this time is prompted to synthesize wiki updates using MCP tools:

  • paperclipWikiListPages — read-only, available during the main run
  • paperclipWikiReadPage — read-only, available during the main run
  • paperclipWikiWritePage — write, exposed only during synthesis
  • paperclipWikiDeletePage — delete, exposed only during synthesis
V2 — post-run synthesis via session resume
main run synthesis pass

Why this works

The agent is the author of its own memory. During synthesis it has full context of what just happened and decides which pages to write, update, or delete using first-class tool calls. Failures in synthesis are swallowed as warnings — the run itself is already complete.

8.2 Shared types

// packages/shared/src/types/wiki.ts
export interface WikiPageInfo {
  path: string;        // relative, normalized, posix-style
  title: string;       // from first markdown heading or path
  updatedAt: string;   // ISO-8601
  bytes: number;
}

export interface WikiContextBundle {
  pages: WikiPageInfo[];          // index
  inlinedPages: {                 // small pages inlined into prompt
    path: string;
    content: string;
  }[];
}

export interface WikiUpdate {
  kind: "write" | "delete";
  path: string;
  content?: string; // required for write
}

9. Side Fixes Along the Way

fix

opencode · read-only symlinked config

On NixOS / Home Manager setups, ~/.config/opencode is often a symlink into the read-only Nix store. The runtime config copy was failing with EACCES because fs.cp preserved the symlink and its read-only perms.

Fix

  • dereference: true so file contents are copied, not the symlink.
  • Remove the copied file before writing the merged config (in case it inherits read-only bits).

packages/adapters/opencode-local/src/server/runtime-config.ts

fix

hiring skill · hardcoded codex_local

The paperclip-create-agent skill example had "adapterType": "codex_local" and model: "o4-mini" hardcoded. CEOs/CTOs following the example were hiring software engineers with codex even when the codex CLI wasn't installed, producing: 

Command not found in PATH: "codex" (adapter_failed)

Fix

  • Replaced hardcoded values with <your-adapter-type> / <model> placeholders.
  • Added workflow guidance: "default to your own adapter type unless the role specifically requires a different one".

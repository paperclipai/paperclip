# Phase 61 Plan 01: Release Channel And Signed Updater Evidence Gate

<objective>
Add a deterministic RealTycoon2 release channel and signed updater evidence gate that validates internal/beta/stable channel metadata, per-platform artifact URLs/checksums/signatures, rollout and rollback policy, installed/update state visibility, and Phase 60 signing prerequisite evidence without adding native dependency or lockfile churn.
</objective>

<requirements>
- `DIST-04`: Operators can manage internal, beta, and stable release channels with version, artifact URL, checksum, signature, rollout policy, and rollback candidate metadata.
- `DIST-05`: The app/update pipeline can validate signed updater metadata and expose download, install, relaunch, and failure state to operators.
</requirements>

<must_haves>
## Requirements
- DIST-04: `internal`, `beta`, and `stable` channels are represented in the manifest and blocker report, including version, artifact URL, checksum, signature, rollout policy, and rollback candidate metadata.
- DIST-05: The updater/channel gate validates metadata before it can be considered available, records installed/latest build identity, and reports download/install/relaunch/failure state.

## Decisions Covered
- D-01 through D-04: distinct native channel identities, required metadata, per-platform artifacts, and mandatory rollback candidate.
- D-05 through D-08: deterministic repo-local updater/channel gate, fail-closed validation, updater key separation, and Tauri-compatible dependency boundary.
- D-09 through D-11: installed/latest build identity, explicit updater state vocabulary, and stable blocker codes.
- D-12 through D-15: Phase 60 signing prerequisite consumption, package scripts, operator docs, and guarded release workflow boundary.
- D-16 through D-17: v2.9 regression protection and no default Playwright E2E.
</must_haves>

<truths>
- Phase 61 is a release evidence gate and updater feed contract. It does not need to add `apps/desktop`, Tauri dependencies, native build outputs, or `pnpm-lock.yaml` changes.
- Tauri expects updater signature values to be the content of generated `.sig` files, not a path or URL.
- The repo-local gate should verify checksum values against local artifact files when fixture/artifact paths are provided; otherwise it must at least validate checksum shape and evidence fields.
- The gate must return non-zero when blockers exist and write durable evidence under a timestamped planning evidence directory.
- Existing dirty worktree changes must be preserved. Stage/commit only Phase 61 files.
</truths>

<threat_model>
| ID | Threat | Mitigation |
|----|--------|------------|
| T-61-01 | A channel feed is treated as publishable while required channel, platform, checksum, signature, rollout, or rollback metadata is missing. | Required manifest fields become blockers with stable codes and non-zero CLI exit. |
| T-61-02 | Updater metadata is accepted for an artifact that lacks macOS/Windows signing trust evidence. | Each platform entry references a Phase 60 signing summary and blocks when missing, blocked, or platform-mismatched. |
| T-61-03 | Updater private keys, passwords, certificate material, or tokens are accidentally committed in fixtures/docs/manifests. | Secret-pattern rejection and sensitive-key secret-reference checks are covered by focused tests. |
| T-61-04 | Operators cannot see installed build identity, update lifecycle state, or failure reason. | Summary/report output includes installed state, update state, blockers grouped by channel/platform/check, and next actions. |
| T-61-05 | Phase 61 expands into tray/shortcut/push or reopens v2.9 capture behavior. | Files modified exclude capture source files; docs and planning artifacts defer Phase 62/63/64 scopes explicitly. |
</threat_model>

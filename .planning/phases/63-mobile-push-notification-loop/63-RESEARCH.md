# Phase 63: Mobile Push Notification Loop - Research

**Researched:** 2026-05-01
**Status:** Complete

## Research Question

What does planning need to know to implement the smallest safe mobile/Web Push/APNs notification readiness loop for RealTycoon2 without adding provider credential churn, native mobile scaffolding, or v2.9 capture/review regressions?

## Summary

Phase 63 should continue the v3.0 native distribution pattern: deterministic evidence gate, focused test, package scripts, and operator docs. The repo still has no native mobile scaffold or Tauri dependency, and APNs/Web Push credentialed sends require real provider secrets that should not be introduced into local tests. The right first implementation is a push notification readiness gate that validates subscription/token lifecycle evidence, minimal payload semantics, delivery/retry/failure state, and click/deep-link return to board review targets.

Web Push and service worker click handling are viable future implementation points because `ui/public/sw.js` already exists. MDN documents Push API as broadly available and service-worker based, and `notificationclick` is specifically a service worker event that can focus or open a window. Web Push protocol references also reinforce the need to treat provider delivery as intermediated and encrypted, with VAPID identifying the application server. For Phase 63, the plan should model those constraints without implementing real encrypted provider sends.

APNs registration is a native/mobile responsibility. Apple documentation says remote notifications are generated by the company's server and delivered by APNs, and APNs needs a device token that identifies the app/device destination. In local/CI tests, represent APNs as provider evidence with device token hash, topic/bundle identity, environment, permission state, and registration failure/revocation state rather than real sends.

## Findings

### Web Push and service worker route

- The repo already has `ui/public/sw.js` with install/activate/fetch handlers and a network-first offline fallback. It has no `push`, `notificationclick`, or `pushsubscriptionchange` handlers today.
- MDN's Push API page marks Push API as widely available and available in workers.
- MDN's `notificationclick` page says the event fires in `ServiceWorkerGlobalScope` when a notification created by `ServiceWorkerRegistration.showNotification()` is clicked. Examples use `clients.matchAll`, `client.focus`, and `clients.openWindow`.
- Phase 63 can validate click-through semantics by requiring evidence that a notification click opens/focuses the target board/review route. It does not need browser E2E by default if the evidence contract is deterministic.

### Web Push protocol and credentials

- RFC 8030 describes Web Push as generic event delivery through a push service.
- RFC 8291 covers Web Push message encryption and notes push messages are intermediated through push services, so payloads should be minimized and encrypted in real implementations.
- RFC 8292 defines VAPID as application-server identification for Web Push. It uses public/private key material; private keys must remain secret references.
- The first implementation should not add `web-push` or similar dependencies unless the plan can avoid lockfile churn. Since the lockfile policy forbids committing `pnpm-lock.yaml`, a provider-agnostic evidence gate is safer.

### APNs registration and delivery route

- Apple User Notifications documentation describes user-facing local and remote notifications; for remote notifications, the company server generates pushes and APNs handles delivery to devices.
- Apple's APNs registration documentation centers on a unique device token that identifies the app/device destination.
- Phase 63 evidence should track provider `apns`, environment (`sandbox`/`production` if modeled), device token hash/reference, bundle/topic identity, permission state, registration state, and failure reason without storing raw tokens or `.p8` key material.

### Existing RT2 capture/review boundary

- `server/src/services/rt2-work-board.ts` already creates persistent capture drafts, stores source evidence, blocks invalid/missing signed sources, detects duplicates, creates revisions, and requires explicit review before promotion.
- `packages/shared/src/types/rt2-task.ts` includes `mobile` and `native` capture sources, draft statuses, source evidence, and current working-tree additions for capture queue filters and reliability reports.
- `ui/src/components/Rt2DailyBoard.tsx` already has approval waiting filters, a capture review inbox, source evidence display, and in-flight reliability report UI.
- Push should route to these surfaces. It should not create a separate notification inbox or direct apply path.

### Evidence gate implementation pattern

- `scripts/rt2-native-signing-gate.mjs`, `scripts/rt2-release-channel-gate.mjs`, and `scripts/rt2-resident-surface-gate.mjs` are the implementation analogs. They parse a manifest, collect blockers/passed checks, reject raw secrets, write `summary.json`/`report.md`, and exit non-zero when blockers exist.
- Focused tests use Node `assert/strict`, temporary fixture roots, pure evaluator assertions, run function assertions, report assertions, and CLI assertions.
- `doc/NATIVE-DISTRIBUTION-FOUNDATION.md` is the canonical native distribution contract. `doc/RELEASE-HOST-VERIFICATION.md` is the correct operator runbook surface.

## Validation Architecture

### Automated validation

Add `scripts/rt2-push-notification-gate.test.mjs` to cover:

- A complete manifest with Web Push and APNs registrations, minimal RT2 work signals, delivery attempts, click-through evidence, and capture reliability references produces `status: passed`.
- Missing company/user/device/provider scope blocks.
- Non-active registrations without reason/evidence block.
- Raw device tokens, VAPID private keys, APNs auth keys, passwords, or provider tokens block.
- Payloads containing sensitive content or unsupported signal types block.
- Missing deep-link/review target blocks.
- Delivery failures without error code, retry decision, or invalid-token revocation evidence block.
- Permission denied without operator evidence blocks.
- Missing click-through evidence blocks.
- CLI execution writes `summary.json` and `report.md` and exits `0` for pass, non-zero for blocker.

### Commands

- Focused: `pnpm run test:push-notification-gate`
- Operator gate: `pnpm run rt2:push-notification-gate -- --manifest <path>`
- Type safety: `pnpm typecheck`
- Do not run `pnpm test:e2e` by default.

### Manual validation

- Inspect generated `report.md` for registration, signal, delivery, click, and reliability grouping.
- Inspect docs to ensure Phase 63 does not claim real credentialed APNs/Web Push delivery in CI or full native mobile packaging.
- Confirm `pnpm-lock.yaml` remains unchanged.
- Preserve existing dirty working-tree edits in shared/server/ui capture reliability files.

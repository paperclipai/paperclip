# Phase 63 Plan 01: Mobile Push Notification Evidence Gate

<objective>
Add a deterministic RealTycoon2 push notification evidence gate that validates Web Push/PWA and APNs/native-mobile subscription state, minimal work-signal payloads, delivery/retry/failure state, and click/deep-link return to board review targets without adding provider credentials, native mobile scaffolding, or lockfile churn.
</objective>

<requirements>
- `PUSH-01`: Operators can register, revoke, and rotate web/mobile/PWA/native push subscriptions or APNs/Web Push tokens by company, user, and device scope.
- `PUSH-02`: RT2 work signals such as approval waiting, failed sync, and review requested are delivered as minimal payload push notifications that click/deep-link back to board review targets.
- `PUSH-03`: Operators can see push permission denied, token invalid, delivery failure, retry, and click-through metrics in capture reliability or release gate evidence.
</requirements>

<must_haves>
## Requirements
- PUSH-01: The manifest and report represent company/user/device/provider-scoped registrations, registration state, revocation, rotation, provider identity, platform, and secret-safe token references.
- PUSH-02: The manifest and report validate minimal RT2 work-signal payloads and review-target routes for approval waiting, failed sync, and review requested notifications.
- PUSH-03: The manifest and report validate delivery attempts, retry decisions, invalid-token handling, permission-denied evidence, click-through evidence, and capture reliability references.

## Decisions Covered
- D-01 through D-03: evidence-first push gate, no provider/native dependency churn, timestamped `.planning/native-push-runs/` output.
- D-04 through D-08: company/user/device/provider registration lifecycle, provider/platform vocabulary, fail-closed states, token secrecy, rotation/revocation evidence.
- D-09 through D-12: minimal payload, required RT2 signals, board/review deep-link targets, no sensitive work content.
- D-13 through D-16: delivery/retry/click evidence, stable failure codes, invalid token and click-through proof.
- D-17 through D-19: existing capture reliability integration, RT2 work-state reuse, additive company-scoped backend/schema boundary if needed.
- D-20 through D-22: existing service worker hook, APNs evidence contract, no unnecessary Tauri/native dependency.
- D-23 through D-28: focused package scripts, blocker taxonomy, docs, and Phase 64-consumable summary.
- D-29 through D-30: focused verification and explicit no-auto-promote/apply boundary.
</must_haves>

<truths>
- The current repo has no `apps/desktop` package, no Tauri dependency, and no provider push dependency. Do not add them for this evidence gate.
- `ui/public/sw.js` is the existing PWA service worker surface, but adding push handlers is optional unless the plan can keep it credential-free and focused.
- Existing capture/review APIs and UI surfaces are the review target. Do not create a second notification inbox.
- Push evidence must return non-zero when blockers exist and write durable evidence under a timestamped planning evidence directory.
- Existing dirty worktree changes in shared/server/ui capture reliability files must be preserved. Stage/commit only Phase 63 files.
</truths>

<threat_model>
| ID | Threat | Mitigation |
|----|--------|------------|
| T-63-01 | Push readiness claims pass while company/user/device/provider registration scope is missing. | Registration validator requires scope and fails closed with stable blocker codes. |
| T-63-02 | Raw APNs tokens, VAPID private keys, provider credentials, or passwords are committed in manifests or reports. | Secret-pattern rejection and sensitive-key secret-reference checks are covered by focused tests. |
| T-63-03 | Push payload exposes task/draft content, secrets, or private context outside authenticated app navigation. | Payload validator allows only minimal routing fields and blocks sensitive content fields. |
| T-63-04 | Failed delivery, permission denial, or invalid token is hidden from operators. | Delivery validator requires failure codes, retry decisions, invalid-token revocation evidence, and permission-denied evidence. |
| T-63-05 | Notification click opens the wrong target or bypasses board review. | Click validator requires route/reached-state evidence for board/review targets and no auto-promote/apply path. |
</threat_model>

# Phase 54 Plan 01: Persistent Capture Draft Revision Lifecycle

<objective>
Make capture drafts reopenable, editable, and auditable before board promotion. The existing capture queue remains the parent lifecycle owner, while operator edits are persisted as draft revisions and promotion uses the latest revision snapshot.
</objective>

<context>
Read before implementation:
- `.planning/phases/54-persistent-capture-draft-revision/54-CONTEXT.md`
- `.planning/REQUIREMENTS.md`
- `.planning/ROADMAP.md`
- `.planning/phases/51-one-liner-to-board-capture-flow/51-CONTEXT.md`
</context>

<tasks>

## 1. Add revision persistence

- Add `rt2_capture_draft_revisions` to `packages/db/src/schema/rt2_work_board.ts`.
- Columns: `id`, `draftId`, `companyId`, `revisionNumber`, `snapshot`, `changeSummary`, `createdByUserId`, `createdAt`.
- Add indexes for `(draftId, revisionNumber)` and company/draft lookups.
- Add migration `0104_rt2_capture_draft_revisions.sql`.
- Export the new table from `packages/db/src/schema/index.ts`.

## 2. Extend shared contracts

- Add capture draft statuses: `revised`, `on_hold`, `revision_requested`, `rejected`.
- Add `Rt2CaptureDraftRevisionSummary`, latest revision fields, and detail type.
- Add validators/types for:
  - creating a revision from editable draft fields
  - review state transitions: `hold`, `reject`, `request_revision`, `mark_review_required`
- Keep existing promote/fail validators backward compatible.

## 3. Update backend service and routes

- Initial inbound draft creation must create revision 1 from parsed input.
- `listCaptureQueue` should include latest revision summary but not full history.
- Add detail route returning draft plus full revision history.
- Add revision route that appends a new revision, updates latest denormalized `parsedDraft`, marks status `revised` when applicable, and appends audit evidence.
- Add transition route for hold/reject/request-revision/reopen-to-review.
- Update `promoteCaptureDraft` to use latest revision snapshot instead of reparsing `rawText`.
- Promotion/failure/transition activity logs should include draft/revision ids and Korean UI can still use concise state labels.

## 4. Update daily board capture inbox

- Add reopen/edit controls in `Rt2DailyBoard` using the draft summary already present in queue rows.
- Show latest revision number and compact history summary when available.
- Add editable fields for title, To-Do, deliverable, base price, quality hint, OKR candidate, and operator note.

## 5. Verification

- Shared tests cover new validators and status labels.

- Run focused Vitest command and `pnpm typecheck`.

</tasks>

<threat_model>
- **Audit loss:** In-place draft edits could erase source evidence. Mitigation: immutable raw/source fields and append-only revision rows.
- **Unauthorized cross-company draft access:** Detail/revision/transition routes must always check company access and draft company id.
- **Promotion mismatch:** Promotion could accidentally use stale `rawText`. Mitigation: focused route test proves latest revision snapshot creates promoted task/deliverable values.
- **State confusion:** Hold/reject/request-revision could disappear from inbox. Mitigation: explicit statuses and UI/server tests for active review states.
</threat_model>

## 2. Add public signed messaging inbound route and service helpers

- Add a dedicated route for external messaging capture, for example:
  - `POST /api/companies/:companyId/rt2/capture-sources/:source/inbound`
- Do not call `assertCompanyAccess` or require a board actor on this public route.
- Limit `:source` to `slack`, `teams`, and `webhook`. Reject other source values with structured 400.
- Resolve the source record by company and source. If no safe source record exists, return structured 404/403 without creating a draft.
- Normalize payloads into the existing `CreateOneLinerInboundDraft` shape and call `rt2WorkBoardService.createInboundDraft` for valid text payloads.
- Accept signature from `x-rt2-signature` header and/or payload `signature`, while preserving existing `signature` behavior.
- Add service helper(s) if needed so recognized malformed payloads can persist a `failed` capture draft with:
  - `failureCode: "parse_error"`
  - `failureMessage` describing malformed payload
  - source evidence with reason code `malformed_payload`
  - redacted provider metadata
- Update source health for success, missing/invalid signature, blocked/stale source, duplicate, and malformed attempts.
- Log activity entries for public inbound attempts with source, status, duplicate id, permission/signing status, reason code, event id, and source installation id.
- Extend `server/src/__tests__/rt2-task-routes.test.ts` for:
  - source setup/secret rotation does not expose stored secret
  - signed Slack inbound success
  - missing/invalid signature creates `permission_blocked`
  - blocked/stale source produces blocked evidence
  - duplicate payload remains distinguishable
  - malformed recognized payload creates failed draft/evidence
  - unknown source/company fails without draft

<threat_model>
- **T-56-01 Public route abuse:** External inbound route can be called without board auth. Mitigation: restrict route to messaging sources, resolve company-scoped source records, require/verify source signing when configured, and return structured errors for unknown sources.
- **T-56-02 Secret leakage:** Signing secret could appear in UI, logs, or audit evidence. Mitigation: hash secrets, clear UI input after save, never return stored secret, and redact auth/signature material from evidence.
- **T-56-03 Payload tampering:** Invalid or missing signatures could create trusted tasks. Mitigation: invalid/missing signature creates `permission_blocked` evidence and never auto-promotes.
- **T-56-04 Evidence overcollection:** Provider payloads can contain private tokens or excessive raw data. Mitigation: store bounded redacted metadata only.
- **T-56-05 Failure ambiguity:** Duplicate, unauthorized, and malformed messaging failures could look identical to operators. Mitigation: explicit reason codes, source health updates, audit details, and distinct Korean board labels.
</threat_model>

<success_criteria>

- MSG-03: Duplicate, unauthorized/signature, and malformed payload failures are distinguishable in board review and audit evidence.
- Public messaging route does not weaken authenticated first-party capture routes.

  <action>
**Verify API-02: Semantic versioning with breaking change policy.**

1. **Verify versioning in route structure**:
   - All RT2 routes under `/api/v1/rt2/` prefix
   - Version in URL path: `/api/v1/rt2/tasks`, `/api/v1/rt2/executions`

2. **Verify Accept header handling**:
   - Routes check `Accept: application/vnd.rt2.v1+json` header
   - Returns 406 Not Acceptable for unsupported versions

3. **Verify deprecation header on sunset fields**:
   - When removing a field, response includes `Deprecation: true` header
   - `Sunset` header with date (2 minor versions ahead)
   - `Link: <url>; rel="deprecation"` with migration docs URL

4. **Verify breaking change detection**:
   - Breaking changes: removed fields, changed types, removed endpoints
   - Non-breaking: new optional fields, new endpoints, expanded enums

5. **Create versioning policy documentation**:
   - Document in API_TYPES.md: version lifecycle
   - Define what constitutes breaking vs non-breaking change

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-85-01 | Tampering | API request body | mitigate | Zod schema validation on all request bodies; reject unknown fields |
| T-85-02 | Denial | Version flooding | mitigate | Rate limit per version endpoint; oldest version deprecated first |
| T-85-03 | Information Disclosure | API error messages | mitigate | Error responses sanitized; no stack traces in production |

</threat_model>

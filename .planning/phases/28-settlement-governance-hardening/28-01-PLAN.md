# Phase 28 Plan 01: Settlement Governance Hardening

## Goal

Deliver `SETTLE-01` through `SETTLE-04`: database-level duplicate settlement prevention, anti-gaming signal visibility with company thresholds, linked ledger evidence on approved settlements, and a compact threshold editor in the existing settlement governance UI.

## Scope

- Add settlement uniqueness and threshold persistence in DB schema/migration.
- Extend `rt2PersonalPnLService` to use conflict-safe settlement creation, threshold-backed signal detection, and ledger evidence enrichment.
- Add settlement threshold read/update API behavior under the existing RT2 P&L route group.
- Extend `ui/src/api/rt2-economy.ts` and `ui/src/pages/rt2/PnlPage.tsx`.
- Add focused embedded Postgres route/service coverage.

## Tasks

1. DB
   - Add `uniqueIndex("rt2_settlement_company_work_product_uq").on(companyId, workProductId)`.
   - Add `rt2_settlement_thresholds` table with `company_id` unique plus numeric threshold columns.
   - Add migration `0080_rt2_settlement_governance_hardening.sql`.
   - Export new schema.

2. Server
   - Add default threshold normalization and get/update helpers.
   - Make settlement creation conflict-safe against `(companyId, workProductId)`.
   - Use thresholds in `deriveSettlementGate()` and `detectSettlementSignals()`.
   - Enrich settlement responses with linked ledger evidence.
   - Add route handlers for reading/updating thresholds or include settings in overview.

3. UI
   - Update settlement API types for thresholds and ledger evidence.
   - Add a compact threshold editor in `PnlPage`.
   - Show signal threshold basis and approved ledger evidence inline on cards.

4. Verification
   - Extend `server/src/__tests__/rt2-phase7-economy-marketplace.test.ts` for duplicate guard, threshold config, and ledger evidence.
   - Run focused tests if feasible, then run broader `pnpm typecheck && pnpm test` unless blocked by environment/runtime issues.

## Risks

- Existing worktree is heavily dirty and many RT2 files are untracked; avoid unrelated cleanup.
- `gsd-sdk query` is unavailable, so state commits and roadmap mutations must be handled manually or skipped with a note.

## Success Criteria

- Duplicate settlement materialization is impossible at DB level and handled gracefully by the service.
- Settlement approval cards show ledger entry and `balanceAfter`.
- Anti-gaming thresholds are company-configurable and affect signal generation.
- Focused tests cover the new behavior.

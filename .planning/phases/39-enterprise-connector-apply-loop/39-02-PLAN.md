<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Operator browser -> UI API client | Operator-selected SCIM candidates and deactivate acknowledgement are untrusted until server verifies them. |
| UI API client -> rollout API | Shared contract fields must not be assumed valid client-side; server remains source of truth. |
| Rollout overview -> operator decision | UI readiness/audit evidence can cause operational action, so stale or hidden failures are a safety risk. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-39-07 | Spoofing | `ui/src/api/rt2-enterprise.ts` | transfer | Server route owns authz; UI passes companyId through existing authenticated route contract and does not bypass `assertCompanyAccess`. |
| T-39-08 | Tampering | `EnterpriseRolloutPage` SCIM selection | mitigate | UI sends preview ID/fingerprint and selected candidate IDs; server revalidates stale preview and deactivate acknowledgement per D-04/D-05. |
| T-39-09 | Repudiation | Enterprise rollout audit surface | mitigate | UI renders evidence IDs, timestamps, status, counts, and recent activity details so operator actions can be traced per D-07/D-11. |
| T-39-10 | Information Disclosure | Rollout evidence UI | mitigate | UI only consumes company-scoped rollout API responses; no cross-company evidence lookup or client-side company switching is added per D-09. |
| T-39-11 | Denial of Service | UI apply controls | accept | UI disables invalid apply actions, but server-side validation remains the real mitigation; Phase 39 does not add external live provider calls. |
| T-39-12 | Elevation of Privilege | Deactivate apply UX | mitigate | Deactivate candidates are visibly marked and require explicit acknowledgement before apply can be submitted per D-05. |
</threat_model>

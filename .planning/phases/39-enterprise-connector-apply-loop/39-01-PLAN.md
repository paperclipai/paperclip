<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Operator client -> enterprise rollout API | Untrusted SSO metadata, callback-state, and SCIM source/apply requests enter server routes. |
| Enterprise rollout API -> company-scoped DB | Connector evidence, previews, apply results, and rollback candidates are persisted under `companyId`. |
| Enterprise rollout API -> activity log | Audit details must be accurate and cannot leak cross-company evidence. |
| SCIM preview -> SCIM apply | A stale or tampered preview can become an unauthorized identity mutation if fingerprinting and acknowledgement checks fail. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-39-01 | Spoofing | `server/src/routes/rt2-enterprise.ts` | mitigate | Call `assertCompanyAccess(req, companyId)` before every SSO/SCIM evidence or apply service call per D-09. |
| T-39-02 | Tampering | `applyScimPreview` | mitigate | Require preview ID plus preview fingerprint generated from normalized source/candidates per D-04, and reject stale fingerprint requests. |
| T-39-03 | Repudiation | `logActivity` actions | mitigate | Write `rt2.rollout.sso_handshake_validated` and `rt2.rollout.scim_applied` with evidence IDs, actor, status, counts, failure reasons, and rollback counts per D-07. |
| T-39-04 | Information Disclosure | connector evidence table | mitigate | Include `companyId` on every evidence row and query latest evidence by company scope only per D-02/D-09. |
| T-39-05 | Denial of Service | SCIM apply payloads | mitigate | Use deterministic in-process validation and bounded candidate processing from request payloads; do not add mandatory live IdP/SCIM network calls per D-03. |
| T-39-06 | Elevation of Privilege | SCIM deactivate apply | mitigate | Require explicit deactivate acknowledgement and selected candidates before high-risk deactivate actions per D-05. |
</threat_model>

# Paperclip Truth Extract

Convert any transcript into a complete, evidence-backed **truth ledger**.

A **truth atom** is the smallest useful claim, fact, need, constraint, risk,
capability, workflow, decision, open question, or next step that would make
planning, proposal strategy, system design, or follow-up worse if forgotten.

The output is **not** a summary. It is a structured, auditable ledger where
every atom carries a verbatim quote, speaker, timestamp, and source utterance ID.

## The governing question

> **What truth would hurt us if forgotten?**

Ask this on every line. If forgetting it would not harm future planning or
execution, it is not a truth atom — it is context or noise.

## When to use

- A transcript (call, meeting, interview, voice note) needs to feed proposal,
  planning, or system-design work.
- You have timestamped utterances (`sentence`, `startTime`, `endTime`,
  `speaker_name`, `speaker_id`) or a raw `[HH:MM] Name: text` transcript.
- The downstream consumer needs **speaker-attributed, timestamped evidence** — not
  a prose summary.

## When NOT to use

- User asked for a "meeting summary" or "action items" — produce those directly.
- Source is not a transcript (a PDF, a code diff, a spreadsheet).
- Transcript is < 20 utterances and the user wants a one-paragraph takeaway.

## Non-negotiable rules

1. **Evidence-first.** Every atom includes a verbatim quote from the transcript,
   the speaker name, and the timestamp span. No outside knowledge. No assumed
   context. No invented relationships.
2. **Atomize.** One durable idea per atom. Never merge unrelated ideas.
3. **Preserve speaker and time.** Every atom traces back to exact utterance IDs.
4. **Separate direct from inferred.** Mark `evidence_mode: inferred` only when
   the inference is local, obvious, and supported by adjacent utterances.
5. **Keep noise — but label it.** `truth_atoms` / `context_atoms` / `noise_atoms`
   are separate lists. Banter may establish tone or risk; do not discard it, but
   do not promote it to durable business truth unless it carries real weight.
6. **Account for every utterance.** A `coverage_ledger` entry exists for every
   utterance with a status of `covered`, `context`, `noise`, `duplicate`,
   `fragment`, or `unclear`.
7. **Violating the letter of these rules is violating the spirit.** A clever
   aggregation that buries evidence is not permitted.

## The deterministic pipeline

```
raw transcript
  → normalize utterances (assign stable U-IDs)
  → segment into evidence spans
  → extract truth atoms (chunked if > ~120 utterances)
  → classify by atom_type
  → score durability (0–5) + confidence (0.0–1.0)
  → produce coverage ledger
  → omission audit (what was missed?)
  → hallucination audit (unsupported claims?)
  → merge chunk outputs; reassign final T-IDs in transcript order
  → export JSON (canonical), YAML, Markdown
```

Each stage is specified in `references/protocol.md`. Do not skip stages.

## Durability → routing

| Score | Meaning | Route to |
|-------|---------|----------|
| 5 | Mission-critical | `truth_atoms` |
| 4 | Strong durable truth | `truth_atoms` |
| 3 | Useful truth | `truth_atoms` |
| 2 | Contextual | `context_atoms` |
| 1 | Low-durability banter | `context_atoms` |
| 0 | Filler, greeting, fragment | `noise_atoms` |


Chunking for long transcripts: **120 utterances per chunk, 10 overlap**. Output
each atom only once, assigned to the earliest chunk where its full evidence
appears. Merge chunk outputs with the merge prompt in `references/prompts.md`.
**Do not ask one model call to extract everything from a long transcript.**

## Special handling

- **Fragmented utterances:** merge meaning across split sentences but keep all
  source utterance IDs.
- **Demo content:** if a speaker narrates a mock or product demo, tag
  `atom_type: demo_content` — do not treat it as a real commitment unless the
  speaker explicitly frames it as current capability or a concrete plan.
- **Jokes and banter:** `banter_context` or `noise_atoms`. Promote only if they
  carry relationship, trust, IP, or risk signal.
- **Repeated ideas:** mark later utterances as `duplicate` in the coverage
  ledger unless they add new detail (then create a new atom).
- **Speaker uncertainty with "we":** do not over-identify the organization —
  write `the speaker said "we" …` unless local context makes the referent clear.

## Red flags — stop and restart

- An atom has no `evidence_quote` → remove it.
- An atom cites a speaker or timestamp that does not match the utterance → fix.
- You find yourself summarizing multiple ideas into one atom → split.
- You invented a fact not present in the transcript → remove.
- Demo/mock content promoted to a real-world commitment → downgrade to
  `demo_content`.
- Coverage ledger has unassigned utterances → every utterance gets a status.

## Deterministic validation (run before accepting output)

- Every `truth_atom` has: `atom_text`, `atom_type`, `durability_score`,
  `confidence_score`, `evidence_mode`, `speaker_name`, `start_time`, `end_time`,
  at least one `source_utterance_id`, a non-empty `evidence_quote`, and
  `planning_relevance`.
- Every `evidence_quote` appears verbatim in the cited utterance(s).
- Every utterance appears in `coverage_ledger` exactly once.
- Omission audit ran. Hallucination audit ran. Final T-IDs assigned in
  transcript order (sorted by first source utterance ID → start_time →
  atom_type → atom_text).

## Final reusable command

> Extract all transcript truths with zero unsupported claims: every durable
> claim must be atomic, classified, scored, evidence-backed, speaker-attributed,
> timestamped, and accounted for in a coverage ledger.

That sentence is the contract. If the output does not satisfy it, iterate.

## Use inside Paperclip

Agents with this skill attached run **Layer A only**: transcript →
`ledger.json` (evidence-bound atoms, coverage ledger, audits). Paperclip then
hands that ledger off to `proposal-forge` for **Layer B** (claims → brief →
frozen export with gates). Paperclip does **not** re-author Layer B.

The binding is:

1. Agent with this skill writes `ledger.json` back to the issue as a document.
2. `plugin-truth-extract-example` shells out to `proposal-forge`'s `truth:*`
   artisan commands via `PAPERCLIP_TRUTH_FORGE_PATH` (subprocess — no HTTP
   shim, no TS/Node re-implementation).
3. Each Layer B step is a discrete, named action. Human gates
   (`truth:review-claims`, atom review) are surfaced as explicit status, not
   auto-run.
4. The cumulative `proposal-forge.receipt.json` is written back to the issue:
   `run_id`, `synthesis_run_id`, `brief_id`, `brief_payload_hash`,
   `export_ids[]`, `export_sha256s[]`, `artifact_paths[]`, gate verdict, and
   raw stdout per step.

Buyer-facing completion is **unsupported in this binding**. The
`truth:review-brief` command does not exist in `proposal-forge`. Until it is
authored there, Paperclip offers the archival chain only.

Full spec: `references/thin-binding.md`.

The skill is the **what** and the **how** of Layer A. The plugin is the
**where** and the **when** — and the thin caller of Layer B.

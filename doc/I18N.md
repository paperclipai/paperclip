# UI internationalization (i18n)

The board UI (`ui/`) uses [i18next](https://www.i18next.com/) with [react-i18next](https://react.i18next.com/). User-visible strings are loaded from JSON **resource files** per locale and **namespace**, so new features can ship by **adding or editing mappings** without changing i18n plumbing.

## Defaults

- **Fallback language:** `en` (English) — used when a key is missing in the active locale.
- **Initial language:** resolved in this order:
  1. `import.meta.env.VITE_UI_LOCALE` at build time (`en` or `zh`), if set.
  2. `localStorage` key `paperclip.ui.locale` (`en` or `zh`), if set.
  3. **`zh` (Chinese)** if neither is set.

See `ui/src/i18n/locale.ts` (`resolveInitialUILocale`).

## Directory layout

```
ui/src/i18n/
  bootstrap.ts          # i18n.init + resource registration (add namespaces here)
  locale.ts             # supported locales, persistence, optional runtime switch
  index.ts              # re-exports i18n, locale helpers, react-i18next hooks
  locales/
    en/                 # English (source-of-truth keys; keep in sync with zh)
      common.json
      app.json
      sidebar.json
      instanceSidebar.json
      mobileNav.json
      layout.json
      notFound.json
      onboarding.json
    zh/                 # Chinese
      … (mirror en filenames and key paths)
```

**Rule:** For every user-visible string, add the **same key path** under both `en/` and `zh/` (and any future locale folders). Missing keys fall back to English at runtime.

## Using translations in React

```tsx
import { useTranslation } from "react-i18next";

function Example() {
  const { t } = useTranslation("sidebar");
  return <span>{t("dashboard")}</span>;
}
```

- First argument to `useTranslation` is the **namespace** (matches the JSON filename without `.json`).
- Keys inside the file are nested JSON paths, e.g. `t("sections.work")` for `{ "sections": { "work": "…" } }`.
- Interpolation: `t("onboardingRoute.titleAddAgent", { companyName: "Acme" })` with `{{companyName}}` in the JSON string.

Runtime language change (e.g. future settings toggle):

```ts
import { setUILocale } from "@/i18n";
import { i18n } from "@/i18n";

await setUILocale("en", i18n);
```

## Adding or updating copy (feature work)

1. Choose a namespace (or add a new JSON pair under `en/` and `zh/` — see below).
2. Add the key to **`ui/src/i18n/locales/en/<namespace>.json`** (English).
3. Add the same key to **`ui/src/i18n/locales/zh/<namespace>.json`** (Chinese).
4. Replace the hard-coded string in the component with `useTranslation` + `t(...)`.
5. Run `pnpm --filter @paperclipai/ui typecheck`.

Do **not** embed long product copy in TypeScript; keep it in JSON so translators and future locales only touch the locale folders.

## Adding a new namespace

1. Create `ui/src/i18n/locales/en/<name>.json` and `ui/src/i18n/locales/zh/<name>.json`.
2. Import both in `ui/src/i18n/bootstrap.ts` and add them to the `resources.en` / `resources.zh` objects and the `ns` array.

## Adding a new language (e.g. `ja`)

1. Copy `ui/src/i18n/locales/en/` to `ui/src/i18n/locales/ja/` and translate values.
2. In `ui/src/i18n/locale.ts`, extend `SUPPORTED_UI_LOCALES` and `isUILocale`.
3. In `ui/src/i18n/bootstrap.ts`, add `ja: { ... }` under `resources` and include `ja` in `resources` typing (same shape as `en`/`zh`).
4. Document `VITE_UI_LOCALE=ja` in `ui/.env.example` when wired.

## Build-time locale

For production UI builds, set `VITE_UI_LOCALE` in `ui/.env` (see `ui/.env.example`). The value is baked in at build time; users can still override via `localStorage` if your app calls `setUILocale` later.

## Scope notes

- This setup covers **React UI** strings in `ui/src`. API error messages, CLI output, and server logs remain English unless separately internationalized.
- Adapter environment checks may still return English messages from backend probes; the UI labels around them are translated.

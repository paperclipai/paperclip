# Token 优化计划（中文译本）

## 目标

在**不削弱**智能体能力、控制面可观测性或任务完成质量的前提下，**实质性地**降低 token 消耗。

本计划基于：

- 当前 V1 控制面设计  
- 当前适配器与心跳实现  
- 上述链接中的用户讨论  
- 2026-03-13 当天、默认 Paperclip 实例上的本地运行数据

## 执行摘要

讨论在两个方向上是**对路的**：

1. 应更积极地保留**会话**与 **prompt 缓存局部性**。
2. 应把**稳定的启动说明**与**每次心跳的动态上下文**拆开。

但仅靠这两点**不够**。

在审阅代码与本地运行数据之后，token 问题似乎有**四个相互独立**的根源：

1. **会话型适配器上的计量膨胀。** 部分 token 计数器——尤其是 `codex_local`——记下来的更像是**会话累计总和**，而不是**单次心跳的增量**。
2. **可避免的会话重置。** 任务会话在 **timer 唤醒**和**手动唤醒**时会被有意重置，破坏了常见心跳路径上的缓存复用。
3. **重复的上下文拉取。** `paperclip` 技能指示智能体在**几乎每一次**心跳都重新抓取：分配、事务详情、祖先链、**完整**评论线程。API 目前没有高效的**增量**取向的替代方案。
4. **巨大的静态指令面。** 智能体说明文件与全局注入的技能，在多数内容与当前任务无关且未变的情况下，仍会在启动时重新进入上下文。

正确路径应是：

1. 先修遥测，让数字可信
2. 在安全的前提下保留复用
3. 让上下文获取**增量化**
4. 增加会话压缩/轮转，避免长会话越来越贵

## 维护者视角（读者注：不是「不重视」，而是「不敢单靠体感」）

英文计划作者**并非**认为省 token 不重要；恰恰相反：当用户侧**投诉量级很大、情绪很集中**时，若**没有**可信遥测，维护者**很难**把「体感很贵」直接当成**单一真相**（报表可能灌水、口径可能混了会话累计与单次 run、复现环境各异）。

本文档的价值在于：**用代码审阅 + 本地实例数据**，把现象**拆成四条可分别施工的线**——与下文「执行摘要」里的四点一一对应，方便对齐沟通、避免笼统骂「你们不懂事」：


| #   | 线                               | 要点                                                                               |
| --- | ------------------------------- | -------------------------------------------------------------------------------- |
| 1   | `**codex_local` 等会话型适配器的计量/上报** | 易出现「单次 run 记的是**会话总轮次/总量**」类口径；要在 **Paperclip 归一化**与**适配器侧**把**增量**算清楚，否则优化无处落锤。 |
| 2   | **心跳拉起却**不**恢复任务会话**            | `timer`/手动路径上**有意重置**等，导致缓存与连续性断掉（下文 §2 数据）。                                     |
| 3   | **巨量上下文**                       | 尤其每次全量读评论、全量 enriched issue 等**高变动**载荷；其中一块就是**破坏 prompt 缓存命中**（前缀抖动、重复全量动态段）。   |
| 4   | **技能面**                         | 注入过多、体积大、`codex_local` symlink/工作区错位等——增大**启动面**与**分析噪声**。                       |


以上四条是**同一轮排查里拆出来的结构**，不是「四个独立闲聊主题」；落地顺序仍以正文 **Rollout Order** 为准。

## 已核实结论

### 1. 当前 token 遥测至少部分是夸大的

在本地默认实例上观测到：

- `heartbeat_runs`：2026-02-18 至 2026-03-13 之间共 11,360 次 run  
- `usage_json.inputTokens` 求和：`2,272,142,368,952`  
- `usage_json.cachedInputTokens` 求和：`2,217,501,559,420`

在观测到的 prompt 规模下，这些总量**不可能**是真实的「每次心跳」用量。

佐证：

- `adapter.invoke.payload.prompt` 平均值很小：  
  - `codex_local`：平均约 193 字符，最大 6,067  
  - `claude_local`：平均约 160 字符，最大 1,160
- 尽管如此，许多 `codex_local` run 仍上报数百万级 input token  
- 本地数据中一条复用的 Codex 会话跨 **3,607** 次 run，`inputTokens` 增长到 `1,155,283,166`

解读：

- 对会话型适配器（尤其 Codex），我们很可能把运行时回报的用量存成了**会话总量**，而非**单次 run 的增量**  
- 这会让趋势报表、优化工作和客户信任**全部变差**

这**不**等于没有真实的 token 问题；而是说在判断优化效果之前，我们需要**可信的基线**。

### 2. Timer 唤醒当前会丢弃可复用的任务会话

在 `server/src/services/heartbeat.ts` 中，`shouldResetTaskSessionForWake(...)` 在以下情况返回 `true`：

- `wakeReason === "issue_assigned"`  
- `wakeSource === "timer"`  
- 手动的 on-demand 唤醒

因此许多正常心跳**即使用工作区稳定**，也会跳过已保存的任务会话恢复。

本地数据印证影响：

- `timer/system` run：共 6,587  
- 其中仅 976 曾有 previous session  
- 其中仅 963 最终以同一 session 结束

因此 **timer 唤醒是最大心跳路径**，且多数情况下**没有**恢复先前的任务状态。

### 3. 我们反复让智能体重载同一段任务上下文

`paperclip` 技能当前在**几乎每次**心跳都要求智能体：

- 抓取分配  
- 抓取事务详情  
- 抓取祖先链  
- 抓取**完整**事务评论

当前 API 形态强化了该模式：

- `GET /api/issues/:id/comments` 返回**整条**线程  
- 没有面向心跳消费的 `since`、游标、摘要类端点  
- `GET /api/issues/:id` 返回完整 enriched 上下文，而不是最小增量包

这样安全但**贵**；迫使模型反复消费未变化的信息。

### 4. 静态指令载荷与动态心跳 prompt 没有干净分离

用户讨论里提到的 bootstrap prompt 方向**是对的**。

当前状态：

- UI 暴露了 `bootstrapPromptTemplate`  
- 适配器执行路径**尚未**使用它  
- 多个适配器把 `instructionsFilePath` 内容直接前置进每次 run 的 prompt 或 system prompt

结果：

- 稳定指令与动态心跳内容走在**同一条路**上被重发/重套  
- 没有刻意为各服务商的 **prompt 缓存**做优化

### 5. 我们注入的技能面大于多数智能体所需

本地适配器把仓库技能注入运行时技能目录。

`codex_local` 的重要细节：

- Codex **不会**直接从活动工作区读技能。  
- Paperclip 从当前 checkout 发现仓库技能，再 symlink 到 `$CODEX_HOME/skills` 或 `~/.codex/skills`。  
- 若已存在的 Paperclip 技能 symlink 指向**另一个**活跃 checkout，当前实现会**跳过**而非重新指向。  
- 这可能导致 Codex 在 Paperclip 侧技能变更之后仍在使用**别的工作区**的旧技能内容。  
- 既是**正确性**风险，也是 **token 分析**风险：运行行为可能与正在测试的 checkout 中的说明不一致。

当前仓库技能体积示例：

- `skills/paperclip/SKILL.md`：17,441 字节  
- `.agents/skills/create-agent-adapter/SKILL.md`：31,832 字节  
- `skills/paperclip-create-agent/SKILL.md`：4,718 字节  
- `skills/para-memory-files/SKILL.md`：3,978 字节

在公司专属指令之前，仅技能 Markdown 就近 **58 KB**。  

未必每一 run 都全部进模型上下文，但会增大启动面，应视为 **token 预算**问题。

## 原则

在以下规则下优化 token：

1. **不丢能力。** 智能体仍须能安全恢复工作、理解任务为何存在，并在治理规则内行动。
2. **稳定上下文优于重复上下文。** 未改变的指令不应走最贵路径反复发送。
3. **增量优于全量重载。** 心跳应只消费自上次有效 run 以来**变化**的部分。
4. **度量归一化后的增量，而非原始适配器声称值。** 尤其对会话型 CLI。
5. **保留逃生口。** 看板/手动 run 仍可强制全新会话。

**计费视角（补一句）：** 在支持 **prompt 缓存**且前缀稳定时，大块静态指令多走缓存/折扣档；**反复未命中缓存、按全价计费的巨量动态上下文**（典型如每次全量重拉评论）才是账单主痛点。

## 计划

## 阶段 1：让 token 遥测可信

应**最先**做。

### 改动

- 同时保存：  
  - 适配器回报的**原始**用量  
  - Paperclip **归一化**的单次 run 用量
- 对会话型适配器，相对**同一会话**的上一次用量计算归一化增量。  
- 增加显式字段，例如：  
  - `sessionReused`  
  - `taskSessionReused`  
  - `promptChars`  
  - `instructionsChars`  
  - `hasInstructionsFile`  
  - `skillSetHash` 或技能数量  
  - `contextFetchMode`（`full`、`delta`、`summary`）
- 为各适配器增加解析测试，区分**会话累计**计数器与**单次 run**计数器。

### 原因

没有这些，就无法判断下降来自真实优化还是报表 artifacts。

### 成功标准

- 单次 run 的 token 累计不会在长会话上失控爆炸  
- 恢复会话时，用量曲线在**会话层面**可信且单调，但在 **run 层面**不重算双份  
- 成本页面在迁移期间可同时展示原始值与归一化值

## 阶段 2：默认保留安全的会话复用

这是**杠杆最高**的行为变更。

### 改动

- **停止**在**普通 timer 唤醒**上重置任务会话。  
- **继续**在以下情况重置：  
  - 显式手动的「全新 run」调用  
  - 指派变更  
  - 工作区不匹配  
  - 模型不匹配 / 无效恢复错误
- 在看板需要重置时增加显式唤醒标志，例如 `forceFreshSession: true`。  
- 在 run 元数据中记录会话被复用或重置的**原因**。

### 原因

Timer 唤醒是**主路径**；重置会破坏会话连续性与 prompt 缓存复用。

### 成功标准

- 在**工作区稳定**的大多数情形下，timer 唤醒会恢复先前任务会话  
- 陈旧会话失败**未**增加  
- 每次 timer 心跳的归一化 input token **下降**

## 阶段 3：将静态 bootstrap 上下文与每次心跳上下文分离

这是讨论里 bootstrap 设想的**正确落地版**。

### 改动

- 在适配器执行路径中实现 `bootstrapPromptTemplate`。  
- **仅**在开启全新会话时使用，**不在**恢复的会话上使用。  
- 刻意保持 `promptTemplate` **小而稳定**：  
  - 我是谁  
  - 本次因何被唤醒  
  - 要优先处理哪条任务/评论/审批
- 尽量把长期设置类正文移出**循环性**的每次 run prompt。  
- 当 `promptTemplate` 含高变动或大块内联内容时，增加 UI 提示与警告。

### 原因

静态指令与动态唤醒上下文缓存行为不同，应分开建模。

对 `codex_local`，还需要**按工作区隔离** Codex 技能 home，或在源 checkout 变化时让 Paperclip **重定向**自有技能 symlink；否则活动工作区内的 prompt/技能改进可能到不了运行中的智能体。

### 成功标准

- 全新会话的 prompt 可以仍较丰富，而不抬升**每一次**恢复后心跳  
- 恢复后的 prompt 变短且结构稳定  
- 对保留会话的适配器，缓存命中率提升

## 阶段 4：让事务/任务上下文增量化

这是**最大的产品面改动**，在会话复用之后很可能是**最大的真实省 token**来源。

### 改动

增加面向心跳的端点与技能行为，例如：

- `GET /api/agents/me/inbox-lite`  
  - 最小分配列表  
  - issue id、identifier、status、priority、updatedAt、lastExternalCommentAt
- `GET /api/issues/:id/heartbeat-context`  
  - 紧凑事务状态  
  - 父链摘要  
  - 最新执行摘要  
  - 变更标记
- `GET /api/issues/:id/comments?after=<cursor>` 或 `?since=<timestamp>`  
  - 仅返回新评论
- 可选 `GET /api/issues/:id/context-digest`  
  - 服务端生成的紧凑摘要供心跳使用

更新 `paperclip` 技能，使默认模式变为：

1. 拉紧凑 inbox
2. 拉紧凑任务上下文
3. 除非首次读取、mention 触发唤醒或缓存未命中，否则**只**拉新评论
4. 仅在需要时拉**完整**线程

### 原因

今天我们在用**完整保真**的看板 API 当**心跳 API**——方便，但 token 效率差。

### 成功标准

- 在首次拿到任务之后，多数心跳只消费**增量**  
- 反复阻塞任务或长线程工作不再重放整条评论历史  
- Mention 触发唤醒仍有足够上下文正确回复

## 阶段 5：会话压缩与受控轮转

防止长会话无限膨胀。

### 改动

- 按适配器/会话增加轮转阈值，例如：  
  - 轮次  
  - 归一化 input token  
  - 存活时间  
  - 缓存命中劣化
- 轮转前生成结构化**交接摘要**，例如：  
  - 当前目标  
  - 已完成工作  
  - 待定决策  
  - 阻塞项  
  - 触及的文件/工件  
  - 建议的下一步
- 将该摘要持久化到任务会话或运行时状态。  
- 下一会话以 **bootstrap prompt + 紧凑交接摘要 + 当前唤醒触发** 启动。

### 原因

即便复用可取，有些会话也会变得太贵，不宜无限延续。

### 成功标准

- 极长会话不再无界增长  
- 轮转**不**导致任务连续性丢失  
- 任务成功率保持持平或提升

## 阶段 6：减少不必要的技能面

### 改动

- 从「注入**全部**仓库技能」改为按智能体或按适配器的**允许列表**。  
- 默认本地运行时技能集可能应为：  
  - `paperclip`
- 为专门智能体提供可选技能：  
  - `paperclip-create-agent`  
  - `para-memory-files`  
  - `create-agent-adapter`
- 在智能体配置与 run 元数据中暴露**当前激活**技能集。  
- 对 `codex_local`，要么：  
  - 使用**工作区专属**的 `CODEX_HOME` 运行，要么  
  - 把 Paperclip 拥有的 Codex 技能 symlink 视为：指向不同 checkout 时可**修复**

### 原因

多数智能体不需要每次 run 都带适配器创作或记忆系统类技能。

### 成功标准

- 启动指令面更小  
- 显式需要额外技能的专家智能体**不**损失能力

## 上线顺序

建议顺序：

1. 遥测归一化
2. Timer 唤醒的会话复用
3. Bootstrap prompt 实现
4. 心跳增量 API + `paperclip` 技能改写
5. 会话压缩/轮转
6. 技能允许列表

## 验收指标

仅当**效率**与**任务结果**都改善时，才视为本计划成功。

主指标：

- 每次成功心跳的归一化 input token  
- 每个完成事务的归一化 input token  
- 会话型适配器的 cache-hit 比例  
- 按唤醒来源划分的会话复用率  
- 拉取**完整**评论线程的心跳占比

护栏指标：

- 任务完成率  
- 阻塞任务率  
- 陈旧会话失败率  
- 人工介入率  
- 智能体完成后事务重开率

初期目标：

- 每次**成功恢复的**心跳，归一化 input token 降 **30%～50%**  
- 稳定 timer 唤醒上 **80%+** 会话复用  
- 首次读过任务后，全量评论重载降 **80%+**  
- 完成率或失败率**无**统计意义上的回退

## 具体工程任务

1. 增加归一化用量字段及 run 分析迁移支持。
2. 修补会话型适配器计费：相对先前会话总量算增量。
3. 修改 `shouldResetTaskSessionForWake(...)`，使 timer 唤醒**默认不**重置。
4. 端到端在适配器执行中实现 `bootstrapPromptTemplate`。
5. 增加紧凑心跳上下文与增量评论 API。
6. 围绕 delta-fetch 行为改写 `skills/paperclip/SKILL.md`。
7. 增加带交接摘要的会话轮转。
8. 用显式允许列表取代全局技能注入。
9. 修复 `codex_local` 技能解析，使工作区本地技能变更**可靠**进入运行时。

## 建议

视为**双轨**推进：

- **轨道 A：正确性与无悔快赢**  
  - 遥测归一化  
  - Timer 唤醒会话复用  
  - Bootstrap prompt 实现
- **轨道 B：结构性减 token**  
  - 增量 API  
  - 技能改写  
  - 会话压缩  
  - 技能允许列表

若**只做轨道 A**，会变好，但智能体仍会读太多未变的任务上下文。

若**不做 A 先做 B**，将无法干净地证明收益。

# Paperclip 技能精简计划

## 状态

延期跟进。除记录本计划外，不要将其纳入当前的 token 优化 PR。

## 延期原因

`paperclip` 技能属于关键控制平面安全层。精简该技能可能减少新会话的 token 消耗，但也存在提示词回归风险。我们目前尚无评估手段来安全地验证任务分配处理、检出规则、评论礼仪、审批流程和升级路径的行为一致性。

当前 PR 应优先交付风险较低的基础设施改进：

- 遥测标准化
- 安全会话复用
- 增量式 issue/评论上下文
- 启动提示词与心跳提示词分离
- Codex 工作树隔离

## 当前问题

即使在上下文路径修复之后，全新运行仍然消耗大量输入 token。剩余的较大启动开销似乎来自运行开始时将完整的 `paperclip` 技能及相关指令内容加载到上下文中。

该技能目前在一个文件中混合了三类内容：

- 几乎每次运行都会使用的热路径心跳流程
- 关键策略和安全不变量
- 大多数运行不需要的低频工作流/参考资料

这种结构是安全的，但代价昂贵。

## 目标

- 在不削弱代理安全性的前提下减少首次运行的指令 token
- 保留所有现有的 Paperclip 控制平面能力
- 保持常见心跳行为明确且易于代理遵循
- 将低频工作流和参考资料移出热路径
- 创建一个后续可以系统性评估的结构

## 非目标

- 更改 Paperclip API 语义
- 移除必要的治理规则
- 删除低频工作流
- 在当前 PR 中更改代理默认行为

## 建议方向

### 1. 将热路径与查阅资料分离

将技能重构为：

- 一个始终加载的核心部分，用于常见心跳循环
- 按需加载的资料，用于低频工作流和深度参考

核心部分应仅覆盖几乎每次唤醒都需要的内容：

- 认证和必需的请求头
- 收件箱优先的任务获取
- 强制检出行为
- `heartbeat-context` 优先
- 增量评论获取规则
- 提及/自我分配例外
- 阻塞任务去重
- 退出前的状态/评论/发布预期

### 2. 围绕一个规范流程标准化技能

相同的规则目前在多处重复表达：

- 心跳步骤
- 关键规则
- 端点参考
- 工作流示例

重构使每个操作事实只有一个主要归属位置：

- 流程
- 不变量列表
- 附录/参考

这可以降低提示词权重，并减少内部指令漂移的可能性。

### 3. 将描述性文本压缩为高信号指令形式

使用紧凑的操作形式重写热路径：

- 简短的有序检查清单
- 扁平的不变量列表
- 仅在可能产生歧义风险时提供最少的示例

减少以下内容：

- 叙述性说明
- 已在其他地方涵盖的重复警告
- 常见操作的大型示例载荷
- 正文中的长端点矩阵

### 4. 将低频工作流移至显式触发器后

以下工作流应保持可用，但不应在全新运行时占据上下文：

- OpenClaw 邀请流程
- 项目设置流程
- 规划 `<plan/>` 回写流程
- 指令路径更新流程
- 详细的链接格式化示例

建议方法：

- 在主技能中保留简短的指引
- 将详细流程移入兄弟技能或引用文档中，代理仅在需要时读取

### 5. 将策略与参考分离

技能应区分：

- 强制性操作规则
- 端点查阅/参考
- 业务流程手册

这种分离使后续评估提示词变更更加容易，并允许适配器或编排层选择哪些内容必须始终加载。

## 建议的目标结构

1. 目的和认证
2. 紧凑的心跳流程
3. 硬性不变量
4. 必需的评论/更新风格
5. 触发式工作流索引
6. 附录/参考

## 发布计划

### 阶段 1. 盘点与度量

- 按章节标注当前技能并估算 token 权重
- 识别哪些章节真正属于热路径，哪些属于低频
- 采集代表性运行数据以比较前后的提示词大小和行为

### 阶段 2. 不改变语义的结构性重构

- 按目标结构重写主技能
- 保留所有现有规则和能力
- 将低频工作流细节移入引用的配套资料
- 保持措辞变更保守

### 阶段 3. 针对真实场景验证

运行以下场景检查：

- 正常的已分配心跳
- 评论触发的唤醒
- 阻塞任务去重行为
- 审批解决后的唤醒
- 委派/子任务创建
- 看板移交回用户
- 规划请求处理

### 阶段 4. 决定默认加载策略

验证完成后，决定：

- 整个主技能是否仍默认加载，还是
- 仅默认加载紧凑核心，低频部分按需获取

在未经验证前不要更改此加载策略。

## 风险

- 控制平面安全规则的提示词退化
- 代理遗忘低频但重要的工作流
- 意外删除了实际承载有用行为的重复措辞
- 在核心技能与配套资料之间引入模糊的指令优先级

## 实施前的先决条件

- 定义控制平面正确性的验收场景
- 至少为关键 Paperclip 流程添加轻量级评估或脚本化场景覆盖
- 确认适配器/启动层应如何加载技能内容与参考资料

## 成功标准

- Paperclip 协调的代理首次运行输入 token 显著降低
- 检出纪律、issue 更新、阻塞处理或委派无回归
- 格式错误的 API 使用或所有权错误不增加
- 当被明确要求时，代理仍能正确完成低频工作流

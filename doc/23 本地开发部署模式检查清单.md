# 本地开发部署模式 — 检查清单（统一口径）

> **给 AI / 助手的硬性约定**：在根据文档拼接 **localhost URL**、打开运维链接、或让人类执行与环境相关的命令之前，**先问人类一句**：「你要用哪种本地模式？」并让对方选 **A / B / C**（见下表）。不要默认假定端口永远是 `3100`。

## 三种模式（一句话）

| 代号 | 模式名称 | 谁在跑 HTTP API | 数据库在哪 | 典型 `DATABASE_URL` |
|------|----------|-----------------|------------|---------------------|
| **A** | 本地服务 + 本地嵌入数据库 | 宿主机 `pnpm dev` / `pnpm dev:once` | 宿主机嵌入式 Postgres（实例目录下的数据簇） | **不设置**（由配置解析到嵌入库） |
| **B** | 本地服务 + 容器数据库 | 宿主机 `pnpm dev` / `pnpm dev:once` | Docker 里的 Postgres（Compose `db` 服务） | `postgres://paperclip:paperclip@127.0.0.1:5432/paperclip` |
| **C** | 容器服务 + 容器数据库 | Docker Compose `server` 服务（需 profile） | 同一 Compose 里的 `db` | 在 **容器内** 为 `...@db:5432/...`（由 Compose 注入） |

补充说明：

- **模式 A** 下，数据目录由实例配置决定，常见为 `~/.paperclip/instances/<instanceId>/db`（含 `PG_VERSION`）。不设 `DATABASE_URL` 时，进程会起/连**嵌入式 Postgres**，默认端口常见为 **54329**（以启动日志为准）。
- **模式 B** 适合你希望 **Node 在宿主机调试**，但 **Postgres 在容器**、数据进 Docker 卷。可从模式 A 的数据迁到容器：仓库根执行  
  `node cli/node_modules/tsx/dist/cli.mjs packages/db/scripts/migrate-embedded-home-to-docker-url.mts`  
  （可选环境变量 `DATABASE_TARGET_URL`）。
- **模式 C** 对应 `docker/docker-compose.yml` 中 **`server` 服务**；默认 **仅起 `db`**，要起 Paperclip 容器需加 **`--profile container-server`**（见 `docker-compose.yml` 内注释）。

更细的启动与停止见：`doc/05 开发指南 DEVELOPING.md`、容器侧见 `doc/08 容器部署 DOCKER.md`。

---

## A — 本地服务 + 本地嵌入数据库

| 检查项 | 做法 |
|--------|------|
| 不指向外部库 | 启动前 **不要**设置 `DATABASE_URL`（除非你明确要切到模式 B） |
| 依赖与命令 | 仓库根：`pnpm install`，再 `pnpm dev` 或 `pnpm dev:once` |
| 端口 | UI/API 默认请求 **3100**；若被占用，服务会像往常一样**顺延下一个空闲端口**，以启动横幅为准 |
| 健康检查 | `curl http://127.0.0.1:<横幅端口>/api/health` |
| 干净停止 | 前台 **Ctrl+C** 后，再 `pnpm dev:list` / `pnpm dev:stop`；嵌入库端口/锁文件问题见 `doc/05 开发指南 DEVELOPING.md` 与 `docs/项目计划/最佳实践/001-运维-回形针本地.md` |
| 密钥 | 使用 `local_encrypted` 等时，`master.key` 与库需匹配；见 `doc/06 数据库 DATABASE.md` |

## B — 本地服务 + 容器数据库

| 检查项 | 做法 |
|--------|------|
| Postgres 已起 | `docker compose -f docker/docker-compose.yml up -d db`（或你实际使用的等同 compose） |
| DB 健康 | `docker compose -f docker/docker-compose.yml ps`，`db` 为 healthy |
| 连接串 | 宿主机上：`DATABASE_URL=postgres://paperclip:paperclip@127.0.0.1:5432/paperclip`（与 compose 中用户/库名一致） |
| 启动 API | 仓库根：先 `set`/`export` `DATABASE_URL`，再 `pnpm dev` 或 `pnpm dev:once`；横幅应出现 **external-postgres** |
| 端口 | 仍以启动横幅为准（可能不是 3100） |
| 可选迁数据 | 见上文迁移脚本（嵌入簇目录需存在且有效） |

---

## C — 容器服务 + 容器数据库

| 检查项 | 做法 |
|--------|------|
| 编排文件 | `docker/docker-compose.yml` |
| 启动方式 | 需要 Paperclip **和** DB 都在容器时：`docker compose -f docker/docker-compose.yml --profile container-server up --build`（或加 `-d`） |
| 浏览器入口 | 默认 `http://localhost:3100`（映射自容器 `3100`） |
| 数据持久 | Postgres：`pgdata` 卷；Paperclip 家目录：`paperclip-data` 卷（见 compose） |

## 容器里「有没有业务数据」— 快速自查

在 **模式 B / C** 下，对正在使用的库执行（用户名/库名与 compose 一致时）：

```bash
docker exec docker-db-1 psql -U paperclip -d paperclip -c 'SELECT COUNT(*) AS companies FROM companies;'
docker exec docker-db-1 psql -U paperclip -d paperclip -c 'SELECT COUNT(*) AS users FROM "user";'
docker exec docker-db-1 psql -U paperclip -d paperclip -c 'SELECT COUNT(*) AS issues FROM issues;'
```

> **说明**：仅有迁移表行、而 `companies`/`issues` 为 0，通常表示**几乎是空库或只跑过初始化**；不以「有条数」代替业务验收。

## 3100 被占用 — 追到「到底是谁起的」

统一流程（与 `doc/05 开发指南 DEVELOPING.md` 一致，此处收拢便于检索）：

1. **先看是不是自家 dev runner**  
   `pnpm dev:list` → 有则 `pnpm dev:stop`。

2. **Windows：谁占 3100**  
   ```powershell
   Get-NetTCPConnection -LocalPort 3100 -ErrorAction SilentlyContinue |
     Select-Object LocalAddress, LocalPort, State, OwningProcess
   ```
   记下 `OwningProcess`（PID）。

3. **追到命令行（最终源头）**  
   ```powershell
   (Get-CimInstance Win32_Process -Filter "ProcessId=<PID>").CommandLine
   ```
   若 `CommandLine` 里是 `node`/`tsx` 且路径偏向本仓库的 `server/src/index.ts`，一般是 Paperclip；也可能是另一个项目的 dev 服。

4. **Linux / macOS 思路**  
   `ss -lntp` 或 `lsof -i :3100`，再对 PID `ps -p <pid> -o args=`。

5. **与模式的关系**  
   - **A / B**：宿主机进程占 3100 最常见。  
   - **C**：若映射 `3100:3100`，同样是**宿主机**的 3100 被容器映射占住；此时停止需 `docker compose down` 或停对应容器，而不是只杀本地 node。
